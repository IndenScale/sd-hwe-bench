# SD-HWE-Bench 评估方法论

> 本文档定义 SD-HWE-Bench 的任务组织、评分体系和对比基线。按领域分层设计，而非一刀切。

---

## 1. 组织模型：领域 → 任务类型 → 实例

```text
SD-HWE-Bench
├── 电信 (telecom)
│   ├── 实例声明 (10-15 tasks)
│   ├── 布局设计 (10-15 tasks)
│   ├── 连接设计 (10-15 tasks)
│   ├── 配合设计 (10-15 tasks)
│   ├── 综合设计 (10-15 tasks)
│   └── 增量修改 (5-10 tasks)
├── 数据中心 (datacenter) [planned]
│   └── ...
├── 机械/键盘 (keyboard) [planned]
│   └── ...
├── 暖通/流体 (hvac) [planned]
│   └── ...
└── 建筑 (building) [planned]
    └── ...
```text

每个叶子节点是一个**自包含任务实例**，包含需求文本、scaffold、参考方案和期望交付物。

## 2. 任务分解与能力映射

每个领域的任务都按六种原子能力拆解，覆盖从简单到复杂的工程推理层级：

| 类型 | 考察能力 | 难度 | 输入特征 | 输出特征 |
|------|---------|------|---------|---------|
| **实例声明** | 型号选择、参数赋值、属性推导 | 简单 | 需求直接给出型号名/参数 | Instance YAML |
| **布局设计** | 离散/连续空间分配、约束满足 | 简单 | 需求给出目标位置或约束条件 | Layout YAML |
| **连接设计** | 接口兼容匹配、拓扑推理 | 中等 | 需求给出连接意图或约束 | Connection YAML |
| **配合设计** | 机械约束匹配、耦合类型选择 | 中等 | 需求给出装配关系 | Mate YAML |
| **综合设计** | 全链路协调、多约束并行满足 | 困难 | 仅给出高层设计目标 | 完整的 piki 项目 |
| **增量修改** | 变更影响分析、约束保持 | 中等 | 已有设计 + 修改需求 | 部分更新的 YAML |

每种类型内部又按难度梯度分三档：

- **L1（直接）**：需求文本直接给出目标名、坐标、端口名——Agent 只需要填入正确的 YAML 结构
- **L2（推理）**：需求给出一组约束条件——Agent 需要推理出最优选择（如"选择功耗最低的 1U 服务器"）
- **L3（规划）**：需求只给出高层目标——Agent 需要自行分解为子任务并协调多个文件

## 3. 按领域的评分设计

### 3.1 通用层：L0-L2（所有领域共享）

这些检查是跨领域通用的，由 piki 核心引擎提供：

| 层级 | 检查内容 | 评分方式 | 反馈延迟 |
|------|---------|---------|---------|
| L0 | YAML 是否合法 | 通过/不通过（门槛） | < 1ms |
| L1 | Schema 校验（字段类型、值域） | 通过/不通过 + 错误计数 | < 10ms |
| L2 | 引用完整性（rack_id 存在？interface_type 已注册？） | 通过/不通过 + 错误计数 | < 10ms |

### 3.2 领域层：L3-L4（按领域定制）

不同领域的 L3（业务规则）和 L4（几何检查）权重和内容不同：

#### 电信 (telecom)

| 层级 | 权重 | 典型规则 |
|------|------|---------|
| L3 | 40% | 功率预算（sum(tdp) ≤ capacity × 0.8）、U 位冲突（区间不重叠）、接口兼容（SFP28 ↔ SFP28）、端口唯一性 |
| L4 | 20% | AABB 碰撞检测、深度越界（设备深度 ≤ 机柜深度） |
| 交付物 | 15% | BOM.csv（型号完整性）、port-map（端口对应） |

#### 数据中心 (datacenter) [planned]

| 层级 | 权重 | 典型规则 |
|------|------|---------|
| L3 | 35% | 功率预算（含 UPS 容量）、冷热通道分配、PDU 相位平衡、网络拓扑连通性 |
| L4 | 25% | 机柜间距、通道宽度、三维空间碰撞（设备+桥架+管道） |
| 交付物 | 15% | rack-layout.svg、BOM.csv、cable-run.csv |

#### 机械/键盘 (keyboard) [planned]

| 层级 | 权重 | 典型规则 |
|------|------|---------|
| L3 | 35% | PCB 布线约束、键轴间距（19.05mm 标准）、外壳装配间隙、按键行程冲突 |
| L4 | 30% | 三维装配碰撞、螺丝孔对齐、壳体干涉检查 |
| 交付物 | 10% | 装配图、BOM、PCB gerber 完整性 |

#### 暖通/流体 (hvac) [planned]

| 层级 | 权重 | 典型规则 |
|------|------|---------|
| L3 | 30% | 管径约束（流量→管径映射）、坡度约束（重力流≥0.5%）、保温层厚度、阀门可及性 |
| L4 | 30% | 三维管道碰撞（含变径/弯头/分支）、避让结构梁柱、空间净高检查 |
| 仿真 (L5) | 15% | CFD 压降校验（可选，按任务启用） |
| 交付物 | 10% | 管道平面图、管道轴测图、材料表 |

#### 建筑 (building) [planned]

| 层级 | 权重 | 典型规则 |
|------|------|---------|
| L3 | 30% | 防火分区、疏散距离、房间邻接拓扑、面积合规 |
| L4 | 25% | 三维空间碰撞（含结构梁柱）、层高约束、日照遮挡 |
| 交付物 | 20% | 平面图、剖面图、面积表、门窗表 |

### 3.3 权重设计原则

- **L3 始终是最高的权重**（30-40%）：规则化是 SD-HWE 的核心差异化能力
- **L4 权重随空间复杂度递增**：电信（一维离散）→ 数据中心（三维离散）→ 键盘（三维装配）→ 暖通（三维连续）
- **交付物权重反映了领域对产出格式的依赖度**：建筑交付物更复杂，权重更高
- **仿真（L5）仅在需要数值求解的领域启用**：暖通、结构，不作为所有领域的默认层

## 4. 对比基线

SD-HWE-Bench 为每个任务类型设置四级基线，从弱到强：

### 4.1 基线等级

| 级别 | 名称 | 描述 | 用于 |
|------|------|------|------|
| **B0** | Random / Naive | 随机生成合法 YAML，不做任何工程推理 | 证明任务不是随便猜就能过的——底限有效性检查 |
| **B1** | LLM Zero-shot | 不给任何 piki 示例，仅告诉 LLM"生成 piki YAML" | 测试 LLM 对 piki DSL 的零样本理解能力 |
| **B2** | LLM Few-shot | 给 3-5 个同类型任务的示例 | 测试 LLM 从示例中归纳 piki 语法的能力 |
| **B3** | Human (Junior) | 经过 2 小时 piki 培训的初级工程师 | 人类可学习性基线——piki DSL 本身对人类的门槛 |
| **B4** | Human (Senior) | 资深电信/机械/暖通工程师 | 领域专家上界——当前最佳工程判断的水平 |

### 4.2 评估指标与基线对照

每条基线和被评测模型，对每个任务类型报告：

| 指标 | 含义 | 基线的用途 |
|------|------|-----------|
| **Pass@1** | 首次尝试通过所有必检层的比例 | B0 应接近 0%；B1-B4 应递增 |
| **L0-L4 分项通过率** | 各层独立通过率 | 诊断能力瓶颈——卡在 L2（不会引用）还是 L3（不懂业务规则） |
| **平均迭代次数** | 从第一次尝试到通过（或放弃）的轮数 | 测试 Agent 的自我修正能力 |
| **任务完成时间** | 总耗时（Agent：token+tool 耗时；人类：实际工时） | 效率对比 |
| **交付物完整率** | 四类交付物各是否产出 | 评估"能不能交付"而不只是"YAML 对不对" |
| **交付物结构化评估** | 按 rubrics 逐维度打分（1-5 分，自然语言描述评分标准） | 评估每个交付物的具体维度的正确性和完整性 |
| **交付物总体评价** | LLM-as-judge：一段自然语言总体评价，不拆维度 | 粗略判断方案整体工程合理性 |

### 4.3 结构化评估 rubrics 示例

以电信机柜部署的 BOM 完整性为例：

| 分数 | 标准 |
|------|------|
| 5 | 所有主物料项齐全，型号可采购，仅缺辅助耗材（标签纸、扎带） |
| 4 | 主物料项齐全，型号可采购，缺光纤跳线或理线器等辅助物料 |
| 3 | 主物料缺失 ≤2 项，或型号不完整（如只写了"光模块"未指定 SFP28） |
| 2 | 主物料缺失 3-5 项，或关键型号不可采购 |
| 1 | 缺失 ≥6 项，或 BOM 无法用于采购 |

每个交付物类型（BOM、port-map、面板图、交底工单）定义独立的 rubrics。结构化评估由 LLM-as-judge 按 rubrics 执行，人工抽查 10% 样本验证一致性。总体评价由 LLM-as-judge 给出单段自然语言判断，不作维度拆解。

### 4.4 基线数据收集

- **B0**：程序化生成——对每个任务随机选型号、随机分配 U 位、随机连接端口。不需要人类参与
- **B1-B2**：通过 API 调用模型，使用标准化的 system prompt + task prompt 模板
- **B3**：招募 3-5 名初级工程师，统一培训后独立完成任务，取中位数
- **B4**：招募 3-5 名资深工程师，独立完成任务，取中位数

## 5. 任务实例格式

每个任务实例是一个自包含目录，结构如下：

```yaml
# tasks/telecom/rack-deploy-001/task.yaml
id: telecom-rack-deploy-001
domain: telecom
type: comprehensive          # instance | layout | connection | mating | comprehensive | incremental
difficulty: L2               # L1 (direct) | L2 (reasoning) | L3 (planning)
description: "从零设计一个 42U 机柜的完整部署方案"
requirement: |
  某数据中心需要在一个 42U 标准机柜（RACK-A01，深度 1000mm）中部署以下设备：
  - 8 台 2U 服务器（型号自选，要求总功耗不超过机柜 PDU 容量的 80%）
  - 2 台 1U 交换机（型号自选，每台至少 24 个 25G SFP28 端口）
  - 所有服务器双上联到两台交换机
  - 提供完整的设备布置表、线缆连接表、面板图、端口图、采购清单

expected_deliverables:
  - type: rack-layout
    format: csv
    description: "设备布置表：设备名、型号、U位"
  - type: cable-connection
    format: csv
    description: "线缆连接表：源设备、源端口、目标设备、目标端口"
  - type: rack-panel
    format: svg
    description: "机柜面板图"
  - type: port-map
    format: csv
    description: "端口分配表"
  - type: bom
    format: csv
    description: "采购清单：名称、型号、数量"

scoring:
  layer_weights:
    L0: 0     # gate
    L1: 10
    L2: 15
    L3: 40
    L4: 20
    deliverables: 15
  human_baseline:
    target_time_minutes: 45
    expected_l3_pass: true
    expected_l4_pass: true

scaffold:
  models:
    - source: "piki-telecom/models"     # 型号库路径
  existing_instances: []                 # 无已有实例（综合设计从零开始）
```text

## 6. 领域扩展协议

添加新领域需满足以下条件：

1. **语言扩展**（如在 piki 中新增空间类型、约束语义）——需更新 piki 核心
2. **规则实现**：为该领域实现 L3 规则集（每个任务类型 ≥ 3 条）
3. **型号库**：提供该领域常见的 Family/Model 定义（≥ 10 个 Model）
4. **任务集**：至少 10 个任务，覆盖 ≥ 3 种任务类型
5. **基线数据**：至少 B0 + B1 数据
6. **人类基线**：至少 1 名该领域工程师完成 ≥ 5 个任务
