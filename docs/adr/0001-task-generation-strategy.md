# ADR 0001: SD-HWE-Bench 任务生成策略

## 状态

已提出 → **已通过**，2026-06-24。

## 背景

SD-HWE-Bench 需要从当前 5 个手工编写的 telecom 任务，扩展为一个规模和质量都足以投向 AI 会议 Dataset/Benchmark Track 的数据集/基准（目标：NeurIPS 2027 Evaluations & Datasets Track）。

我们评估了三种候选的任务生成策略：

1. **继续手工编写孤立任务** —— 当前做法。扩展性差，且难以在不同任务之间维持一致的领域约定。
2. **从真实工程仓库挖掘** —— 类似 SWE-bench。被否决，因为高质量、文本化、声明式的硬件工程开源仓库根本不存在；真实的 commit 历史通常很脏，很少对应干净、可自动评分的单个设计步骤。
3. **构建 canonical ADL 工程并从顺序 commit 中提取任务** —— 自行编写 2–3 个完整、高质量的 ADL 工程，并将它们的演进历史切分为“设计增量”任务。

## 决策

采用策略 **#3**。

具体做法：

- 编写 **2–3 个 canonical 工程**（例如：电信机柜扩容、数据中心一排机架、机械结构装配），全部用 ADL 描述。
- 每个工程代表从概念设计到可加工/可施工完整状态的全生命周期。
- 每个工程作为独立的 Git 仓库维护，commit 历史经过精心设计。
- 任务提取为 commit 之间的状态转移：给定 commit `k` 的仓库状态和描述下一步设计需求的 requirement，Agent 需要生成 commit `k+1` 的状态。
- 每个任务的参考解就是 commit `k+1`；测试套件为 `piki check` 加上交付物生成检查。

## 结果

### 正面

- **可扩展性**：一个 30 commit 的工程可产生 15–30 个任务；3 个工程即可轻松覆盖 Dataset Track 所需的 30–50 个任务规模。
- **质量可控**：因为工程由我们自行编写，可以保证每个 commit 都合法、有意义、可评分。
- **自然难度梯度**：早期 commit 是简单实例声明，中期加入布局/配合/连接设计，后期涉及调试、修改和优化。
- **叙事统一**：任务不是孤立的谜题，而是一个连贯工程故事中的步骤，这对 benchmark 论文非常有利。
- **结构熟悉**：issue/requirement → scaffold → patch → test 的流程与 SWE-bench 类似，AI 社区容易理解。

### 负面 / 风险

- **创作成本高**：一个 30 commit 的高质量工程可能需要 1–2 周专注工作。
- **需要领域知识**：每个 canonical 工程都需要足够的工程真实感。
- **过拟合风险**：如果不同工程的风格过于相似，Agent 可能记住模式而不是学到通用设计能力。必须刻意在不同工程之间引入领域约束和设计模式的差异。
- **工具链缺口**：需要脚本自动从 commit 历史中提取任务并生成 task 元数据。
