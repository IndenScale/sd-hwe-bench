## 1. 投稿与审稿周期

### 1.1 关键日期

| 事项                  | 日期              | 备注                                                      |
| --------------------- | ----------------- | --------------------------------------------------------- |
| 全文投稿截止          | **2026-10-02**    | 无 abstract 提前截止，一次性提交完整 PDF                  |
| 初审结果通知          | **2027-03-31**    | 结果分为 Accept / Reject / Major Revision                 |
| Major Revision 修改期 | 约 6–8 周         | 2027 年具体时长未公布；FSE 2025 为 6 周，FSE 2026 为 8 周 |
| 终稿通知              | Major Revision 后 | 同一批评审人复评修改稿与回复信                            |
| 会议召开              | 2027-07-12 起     | 中国深圳                                                  |

### 1.2 审稿流程

- **双盲审（heavy double-anonymous）**：作者与审稿人互不可见，且审稿与讨论全程保持匿名。
- **至少 3 位 PC 审稿人**；若评审分歧较大或需要特定领域专家，PC chairs 可能追加审稿。
- **三态初审结果**：Accept、Reject、Major Revision。
- **Major Revision 机制**：
  - 作者需在修改期内回应所有审稿意见（包括补充实验、重写算法与解释、澄清动机、调整范围等）。
  - 必须提交 **response letter**，逐条说明如何回应每条意见。
  - 原审稿人会复评修改稿是否充分满足其要求。
- **PC 全体会议**做出最终录用决定。
- **历史录用率**：1995–2014 年平均约 **20.9%**（近年竞争更激烈，实际可能更低）。

### 1.3 投稿系统

- 使用 **HotCRP**（具体链接截稿前公布）。
- 不接受邮件投稿，必须通过系统提交 PDF。

## 2. 关注领域

FSE Research Track 覆盖软件工程全领域，尤其欢迎理论、实证、概念与实验性研究的原创工作。官方列出的主题包括但不限于：

- Artificial intelligence and machine learning for software engineering
- Autonomic computing
- Debugging and fault localization
- Dependability, safety, and reliability
- Distributed and collaborative software engineering
- Embedded software, safety-critical systems, and cyber-physical systems
- Empirical software engineering
- Human and social aspects of software engineering
- Human-computer interaction
- Mining software repositories
- Mobile development
- Model checking
- Model-driven engineering
- Parallel, distributed, and concurrent systems
- Performance engineering
- Program analysis
- Program comprehension
- Program repair
- Program synthesis
- Programming languages
- Recommendation systems
- Requirements engineering
- Search-based software engineering
- Services, components, and cloud
- Software architectures
- Software engineering education
- Software engineering for machine learning and artificial intelligence
- Software evolution
- Software processes
- Software security
- Software testing
- Software traceability
- Symbolic execution
- Tools and environments

### 2.1 与本研究的潜在契合点

从领域列表看，以下方向与 Engineering-as-Code / piki 工作高度相关：

- **Model-driven engineering**：ADL 与 ESA 可视为面向硬件/物理系统的模型驱动方法。
- **Program analysis / Static analysis**：ESA 的 L0-L4 规则检查可映射到静态分析。
- **Tools and environments**：piki 作为语言工具链属于该范畴。
- **Software engineering for AI/ML**：若强调为 AI 生成工程设计提供可验证表示，可落入 AI for SE / SE for AI 交叉区。
- **Software architectures / Components and services**：Part、Mate、Connection 等抽象与架构描述语言（ADL）研究直接相关。
- **Embedded software, safety-critical systems, cyber-physical systems**：硬件/机电系统设计的正确性验证与该领域接壤。

## 3. 典型篇幅、格式与章节结构

### 3.1 篇幅与格式要求

| 项目                    | 要求                                                       |
| ----------------------- | ---------------------------------------------------------- |
| 模板                    | ACM `acmart`，使用 `acmsmall` 选项                         |
| 推荐 LaTeX 头           | `\documentclass[acmsmall,screen,review,anonymous]{acmart}` |
| 引用格式                | numeric 或 author-year 均可                                |
| 版式                    | 单栏                                                       |
| 初稿正文                | **≤ 18 页**（含所有正文与图表）                            |
| 初稿参考文献            | **≤ 4 页**                                                 |
| Major Revision 正文     | **≤ 20 页**（含正文与图表）                                |
| Major Revision 参考文献 | **≤ 4 页**                                                 |
| Data Availability       | 放在 Conclusion 之后，**不计入页数**                       |
| Appendix                | 不限页数，可放形式化定义、语法、额外示例                   |

不符合格式要求的稿件将被 **desk-rejected**，无需进入审稿流程。

### 3.2 本会议的审稿取向与偏好（会议自身特点）

FSE Research Track 与同一等级的 ICSE Research Track 相比，具有以下鲜明特征：

1. **强调原创性与重要性**：
   - 接受开创性方法，但更偏好“解决了真实研究问题”的论文，而非纯愿景或立场陈述。
   - “groundbreaking approaches to emerging problems”被明确鼓励，但必须有扎实的技术或实证支撑。

2. **重视 soundness 与 evaluation**：
   - 审稿标准明确列出 soundness、evaluation（如相关）、presentation、related work。
   - 概念性论文仍需形式化定义、实现验证或案例研究；仅有想法而无论证很难通过。

3. **实验可复现是硬性倾向**：
   - FSE 近年推行 Open Science Policy，要求提交 replication package 或说明无法提供的原因。
   - 论文必须有 Data Availability 声明，且 accepted 论文强烈鼓励提交 artifact evaluation。

4. **双盲审非常严格**：
   - “heavy” double-anonymous：不仅初审匿名，Major Revision 阶段及 response letter 也必须维持匿名。
   - 不得出现作者姓名、单位、能暴露身份的 URL、致谢等；引用自己过往工作需用第三人称。

5. **Major Revision 是常见录用路径**：
   - 初审直接 Accept 比例较低，大量论文通过 Major Revision 后录用。
   - 审稿人可能要求补充实验、新增分析、重写算法解释、重新定位研究范围。

6. **相关工作的系统对比是标配**：
   - 审稿标准明确包含 “appropriate comparison to related work”。
   - 高水平论文通常包含对比矩阵或系统讨论，而非简单罗列文献。

7. **对跨领域研究的开放态度**：
   - FSE 关注 software engineering 全谱系，对借鉴 PL、形式化方法、AI、系统工程思想的工作持开放态度，但要求作者明确说明对 SE 社群的价值。

### 3.3 典型论文章节结构

根据 FSE 近年录用论文与审稿标准，Research Track 长文常见结构如下（18 页正文版）：

| #   | 章节                     | 建议页数 | 核心任务                                   |
| --- | ------------------------ | -------- | ------------------------------------------ |
| 1   | Abstract                 | 0.5      | 问题、方法、贡献、关键验证结果             |
| 2   | Introduction             | 1.5–2    | 动机、研究问题（RQ）、贡献列表、论文组织   |
| 3   | Background & Motivation  | 2–3      | 领域背景、现有方法局限、研究缺口           |
| 4   | Approach / Method        | 4–6      | 核心方法、形式化定义、设计 rationale       |
| 5   | Implementation           | 2–3      | 工具架构、关键组件、实现细节               |
| 6   | Evaluation               | 3–5      | 实验设计、RQ 答案、定量/定性结果、威胁分析 |
| 7   | Related Work             | 1.5–2    | 系统对比，突出差异                         |
| 8   | Conclusion & Future Work | 0.5–1    | 总结、局限、未来方向                       |
| —   | Data Availability        | 0.25–0.5 | artifact / 数据可用性声明（不计页数）      |
| —   | References               | ≤4       | 完整引用                                   |
| —   | Appendices               | 不限     | 形式化证明、语法、额外数据                 |

不同论文类型可调整：

- **理论/形式化论文**：方法章节更长，Evaluation 可为证明、复杂度分析或小规模验证。
- **实证/数据驱动论文**：Background 与 Evaluation 占比更高，方法相对简洁。
- **工具论文**：Implementation 与 Evaluation 并重，需展示工具能力与性能。

## 4. 写作与研究偏好

### 4.1 研究贡献偏好

- 明确列出 **Research Questions (RQs)** 并在 Evaluation 中逐一回答。
- 贡献需具体、可验证，避免空泛宣言（如 "we propose a novel framework" 而不说明解决了什么问题）。
- 偏好“问题驱动”而非“技术驱动”：先讲清为什么现有方法不行，再引出解决方案。
- 概念性工作需说明边界：哪些已实现、哪些是 future work、当前限制是什么。

### 4.2 写作风格偏好

- **清晰优先于华丽**：FSE 审稿人来自多元背景，避免过度领域术语堆砌。
- **motivating example 早出现**：在 Introduction 或 Background 中用具体例子说明问题。
- **形式化与实例并重**：给出定义后，紧跟 running example 帮助理解。
- **诚实说明局限**：明确 threats to validity、未实现部分、失败案例，反而增加可信度。
- **第三人称自引**：双盲审要求引用自己工作时用 “the prior work of XYZ” 而非 “our prior work”。

### 4.3 与相关工作的关系

- 相关工作经验位置灵活：可在 Background 前、方法后或 Evaluation 后，但需系统。
- 必须包含 **对比矩阵（comparison matrix）** 或等效系统讨论，维度通常包括：
  - 目标问题 / 领域
  - 表示能力
  - 验证机制
  - 自动化程度
  - 可复现性 / 开源情况
- 避免贬低式表述，客观说明差异即可。

## 5. 支持性证据要求

### 5.1 审稿标准中的 evidence 维度

FSE Research Track 明确按以下维度评审，其中 soundness 与 evaluation 直接取决于证据质量：

- **Originality**：与已有工作的区分度。
- **Importance**：问题对 SE 社群的重要程度。
- **Soundness**：方法、证明、实现的技术正确性。
- **Evaluation**：实验设计、数据规模、指标、统计显著性（如适用）。
- **Presentation**：可读性、组织结构、图表质量。
- **Appropriate comparison to related work**：系统对比。

### 5.2 实证证据的常见形式

| 证据类型                       | 适用场景      | 说明                                  |
| ------------------------------ | ------------- | ------------------------------------- |
| 实现与工具链                   | 方法论文      | 证明方法可落地，提供 CLI/GUI/库       |
| 案例研究                       | 新概念/语言   | 用 1–3 个真实或代表性示例展示表达能力 |
| 规则/测试覆盖                  | 静态分析/验证 | 统计规则数、测试数、覆盖率            |
| 性能指标                       | 工具论文      | 运行时间、内存、可扩展性              |
| 正确性/缺陷发现                | 分析工具      | 发现的真实 bug 数、误报/漏报率        |
| 用户研究                       | HCI/SE 交叉   | 可用性、任务完成时间、用户满意度      |
| 复现实验                       | 复制/扩展研究 | 与 baseline 的对比、统计检验          |
| Artifact / Replication Package | 所有实证论文  | 代码、数据、脚本、文档，匿名化处理    |

### 5.3 开放科学与 Artifact 要求

- **Data Availability 声明为必需**：
  - 在 Conclusion 之后单独成节，标题固定为 **Data Availability**。
  - 说明 replication package 是否提供、如何获取（投稿时使用匿名链接或上传附件）。
  - 若不提供，需解释原因。
  - 该节**不计入页数**，但仍需维持匿名。

- **Replication Package 强烈推荐**：
  - 可上传为 supplemental material，或提供匿名/私有仓库链接。
  - 内容通常包括：源码、数据集、实验脚本、构建说明、README。

- **Artifact Evaluation（AE）**：
  - 录用论文可额外提交至 FSE 2027 artifact evaluation committee。
  - AE 通过可获得 artifact badges，提升论文影响力与可信度。

- **生成式 AI 披露**：
  - 遵循 ACM Authorship Policy，使用 ChatGPT 等生成式 AI 辅助创作需披露（可在 Acknowledgments 中说明）。
  - 投稿时通常避免致谢，可在 camera-ready 中补充。

## 6. 双盲审与投稿合规

### 6.1 双盲审要求

- 正文不得出现作者姓名、单位、邮箱。
- 不得包含可暴露作者身份的 URL（如个人主页、GitHub 实名仓库）。
- 引用自己过往工作时用第三人称。
- 机构名称需泛化（如 “a large cloud provider” 代替具体公司名）。
- 投稿时不写 Acknowledgments。
- 允许 arXiv 预印本，但**不得说明**该稿已投 FSE 2027。
- Major Revision 的 response letter 也必须匿名。

### 6.2 其他合规要求

- 不得一稿多投；审稿期间不得同时投其他审稿制会议/期刊。
- 需遵守 ACM Authorship Policy（2025-09-16 版）。
- 所有作者需准备 ORCID ID，用于录用后的出版流程。
- 录用后强烈建议参会报告，但非强制。

## 7. 与本研究的适配要点（简要）

基于上述会议特点，若向 FSE 2027 Research Track 投稿 Engineering-as-Code / piki 工作，需特别注意：

1. **研究问题要清晰**：从 “提出范式” 转向 “提出并验证一种方法”，用 RQs 组织全文。
2. **实证必须强化**：提供 piki 的实现、样本验证、规则/测试统计、运行性能等量化证据。
3. **诚实边界**：明确 L4b/L5/L6 未实现、部分 sample 失败等当前限制。
4. **与 SE 社群关联**：强调对 program analysis、static analysis、model-driven engineering、tools 等 SE 子领域的贡献。
5. **准备匿名 artifact**：piki 仓库需匿名化处理，作为 replication package 提交。
6. **双盲细节**：删除作者信息，自引第三人称，匿名化组织名。
