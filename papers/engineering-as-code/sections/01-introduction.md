# 1. 引言

软件工程作为一门学科，构建的是关于正确性的推理基础设施。源代码之所以存在，是为了让程序被阅读、检查、差分、分支、合并和版本化。静态分析器在执行前捕获缺陷。测试套件提供快速、确定性的通过/失败信号。CI 管道对每次提交设卡。这些不是 AI 的创新，而是软件工程的创新——AI 后来利用了这些创新，因为它们提供了可验证奖励强化学习（RLVR）所需的结构化、可验证反馈 [@lightman2024letsverify; @deepseek2025r1]。当代码有测试套件时，Agent 可以提出补丁并收到毫秒级的正确性反馈，策略随之改进 [@sweagent2025swerl]。当芯片网表有设计规则检查（DRC）时，布局 Agent 可以在 DRC 监督下学习 [@mirhoseini2020rlchip]。这一模式在代码 [@anthropic2025claude]、数学 [@shao2024deepseekmath] 和硬件领域是一致的：结构化、可检查的表示使 AI 成为可能。

现在考虑物理工程。电信机柜设计师指定设备、连接器、功率预算和空间约束。结构工程师定义梁、节点和荷载路径。HVAC 工程师规划风管并选型风机。这些工程师产出的工件——三维 CAD 模型、BIM 数据库、IFC 交换文件——以几何为中心且与工具深度耦合 [@buildingsmart2023ifc]。它们捕获了设计"长什么样"以及"如何拼在一起"，但并未将设计**意图**与几何**实例化**分离。

这种耦合的后果不仅限于反馈延迟。当前主流的检查方法——自动合规检查（ACC）——直接作用于几何模型。而几何模型没有关于**预期**关系的表示：一根管道穿过墙体被报为碰撞，仅仅因为墙模型上没开槽，而非设计真的错了。一台服务器插入 PDU 对检查器不可见，除非两者遵循了规则引擎被预先配置解析的特定项目命名约定。结果是 ACC 在语义合理的几何交点中产生大量假阳性，同时在规则执行前还要求每项目进行昂贵的翻模——重命名、重组、重建。这些不是算法缺陷，而是**表示**的缺陷：几何无法编码"这根管道是否**应该**穿过这堵墙"的答案。

对 AI 的后果是严重的。一个生成建筑模型或机架布局的 Agent 无法收到关于其输出是否正确的快速、确定性反馈。闭环是断开的。工程设计领域没有 `cargo check`。

本文的核心论点是：瓶颈在于**表示基础设施**，而非算法能力。物理工程缺乏一种等价于源代码的东西——一种结构化、文本原生、可被机器检查的设计意图表示。而软件工程社群，而非 AI 社群，最有能力设计这样一种表示。具体来说，发明了可编程 lint、pre-commit hook、静态单赋值和包级依赖解析的社群，也拥有定义以下内容的词汇和设计模式：

- 一个**工程设计单元**长什么样（*Part*，类比于函数或模块）
- 如何声明**兼容性约束**（*配合规范*，类比于接口或类型签名）
- 如何将**空间布局**与部件身份分离（*布局语言*，类比于表现与逻辑分离）
- 如何按成本和精度对**正确性规则**进行分层（*L0–L4 规则层次*，类比于语法→类型→语义→性能检查）
- 如何对整个工件进行**版本化、分支和评审**（*Git 工作流*，在代码领域已历经实战检验）

我们将产生的框架称为 **Engineering as Code（EaC）**。EaC 不是"AI for engineering"，它是**为工程领域设计和实现源代码表示和工具链的软件工程方法论**。

## 1.1 源代码鸿沟

表 1 从可验证反馈所需维度刻画了软件工程与物理工程之间的差距。

| 维度 | 软件工程 | 物理工程（现状） |
|------|---------|---------------|
| 主工件 | 文本（`.rs`、`.py`、`.ts`） | 几何（`.stp`、`.rvt`、`.ifc`） |
| 原子单元 | 函数 / 模块 / 类 | CAD 体 / 装配节点 |
| 类型系统 | 静态类型、接口、泛型 | 隐式（连接器族） |
| 正确性检查 | 编译器 / lint / 测试套件 | 下游审查 / 人工审查 |
| 检查延迟 | 毫秒至分钟 | 小时至周 |
| 版本控制 | Git（行级 diff、合并） | PDM / PLM（检出/锁定、二进制 diff） |
| CI 管道 | `git push → lint → test → deploy` | 人工审查门禁 |
| 包管理器 | npm / cargo / pip | 零件库（厂商特定、无版本化） |
| 关系编码 | 显式（import、类型签名、依赖注入） | 隐式于几何或命名约定 |
| 检查信噪比 | 规则以语义类别为对象 | 规则依赖命名规范；纯几何碰撞产生大量假阳性 |

物理工程不缺少规则——建筑规范、电信标准和接口规格是丰富且精确的。缺少的是一种能够在**设计时**使这些规则**可执行**的表示——正如编译器在编译时使类型安全规则可执行。

## 1.2 本文贡献

本文对软件工程作出三项贡献：

1. **ADL：装配体定义语言**（§4）。一种三层的文本原生设计语言。ADL 将部件身份（PDL）、部件配合（PML）和部件布局（PLL）正交分离。每层均可独立解析、检查和差分。ADL 被设计为与 Git、CI 和 LSP 风格诊断集成——它是 SE 工具链的**设计表面**，而非 CAD 替代品。

2. **ESA：工程静态分析**（§5）。一种确定性分层规则引擎（L0–L4），将合规检查从设计后审查前移到提交前门禁。ESA 是编译器分析管道的工程类比：语法→引用完整性→业务规则→几何约束。与 ACC 不同——后者作用于几何模型，受困于假阳性噪音和命名依赖开销——ESA 在语义类别（Part Family、Mate 类型、接口签名）上检查设计**声明**。本文定义了 ESA 的可操作性纲领（豁免、底线、信噪比、AI 辅助），并提供了一种结构化诊断格式，同时充当人类可读的错误报告和机器可消费的奖励信号。

3. **信息表示假说**（§3）。一个可测试的主张，锚定于 RLVR 文献：物理工程中 AI 的瓶颈在于缺乏可计算的设计表示，而非模型能力不足。本文从该假说推导出预测，并描述了 **SD-HWE-Bench**，一个设计用于评估生成式 Agent 在 ADL 创作任务上性能的评测基准。

我们通过 **piki**——一个开源的 EaC 运行时——验证 ADL 和 ESA。在一个代表性电信机架部署中，piki 在 200ms 以内检查 64 条工程规则——接口兼容性、功率预算、U 位冲突、承重。原型集成 Git，支持 `--skip` 和 `warning_only` 规则豁免，并产出适用于 CI 管道消费的结构化诊断。我们在 §6 中报告原型当前范围与明确限制。

## 1.3 为什么投稿 FSE

本文投稿 FSE，因为它回答的是一项软件工程研究问题：**当一个领域缺乏源代码时，SE 社群如何为其设计一种源代码？** 答案涉及 DSL 设计（ADL 的正交子语言）、静态分析（ESA 的分层规则引擎）、版本控制语义（设计意图的文本差分）、CI/CD 集成（pre-commit 合规门禁）和实证评估（SD-HWE-Bench）。这些都是 SE 核心能力。目标领域是电信机架而非云基础设施并不改变贡献的性质——正如 IaC 静态分析 [@chiari2024iacstatic] 是一项 SE 贡献，尽管其目标领域是云运维。

## 1.4 论文结构

第 2 节综述物理工程中当前使用的表示，分析模型 ACC 的结构性缺陷——几何碰撞假阳性和命名依赖导致的翻模开销，以及为何它们是 AI 的瓶颈。第 3 节阐述信息表示假说及其 RLVR 基础。第 4 节介绍 ADL：三层架构、设计理由以及与 SE 工具链的集成。第 5 节介绍 ESA：分层规则层次、相对于 ACC 的语义优势、操作性纲领、结构化诊断输出以及下游验证协议。第 6 节报告 piki 原型和评估计划——电信机架样本的当前结果，后续实验正在进行中，预计在投稿截止日前完成。第 7 节讨论与相关工作的差异——ACC、SysML v2、BIM/IFC、IaC 静态分析和现有基准。第 8 节总结，确认限制，概述未来工作包括 SD-HWE-Bench 和 RLVR 训练。
