# 3. Engineering as Code 方法

EaC 是关于基础设施的一项命题。它说：在我们问 AI 是否能设计物理系统之前，我们应该先问这些系统是否以可被软件工程工具链操作的形式表示。本节定义该范式，阐述其核心假说，并将该假说与 RLVR 文献以及 ESA 的分层设计联系起来。

## 3.1 EaC 是什么

**Engineering as Code（EaC）** 是将结构化、文本化的设计声明作为物理工程系统的唯一真相源，并围绕该表示构建软件工程工具链——版本控制、静态分析、CI/CD 和自动验证——的实践。

EaC 不是 CAD 或 BIM 的替代品。物理工程永远需要几何建模、仿真和制造输出。EaC 分离关注点：*设计意图*（使用哪些部件、它们如何连接、它们位于何处）以文本形式在 ADL 中表达，而*几何实例化*和*分析*消费该声明作为输入。这与芯片设计以硬件描述语言实现的分离相同：Verilog/VHDL 描述意图；布局工具生成几何；DRC 检查结果 [@ieee1364]。

EaC 有三根支柱，但构成一个整体命题：

1. **ADL**（§4）是*表示层*：一种被设计为可被 SE 工具链阅读、检查、差分和版本化的工程设计语言。
2. **ESA**（§5）是*验证层*：一个提供快速、确定性正确性反馈的规则引擎，使 RLVR 成为可能。
3. **信息表示假说**（§3.2）是*理论锚点*：将这些技术选择与一项可测试的主张相连接——为何物理工程中 AI 的进展停滞。

## 3.2 信息表示假说

我们主张：

> **信息表示假说**。物理工程中 AI 的滞后主要由缺乏一种结构化、文本原生、可被机器检查的设计表示引起，而非由模型能力不足或物理的内在复杂性引起。

该假说是**信息论**的，而非计算论的。它不主张物理很简单，或当前模型已足够强大以设计复杂系统。它主张*即使*模型具备能力，当前表示也会妨碍它们改进，因为这些表示不支持驱动代码、数学和芯片设计领域 AI 进步的学习信号——快速、确定性的正确性反馈。

### 3.2.1 来自相邻领域的证据

该假说基于三个领域中可度量的规律：

**代码**。SWE-bench 任务要求 Agent 修复真实 GitHub issue [@jimenez2024swebench]。奖励信号是测试套件：Agent 的补丁要么通过，要么失败。SWE-RL 证明仅凭该信号，无需人类示范，即可训练出解决 42% SWE-bench Verified 任务的 Agent [@sweagent2025swerl]。表示——带有测试套件的 Git 仓库——是使 RLVR 可行的脚手架。

**数学**。DeepSeekMath 和 DeepSeek-R1 在形式化定理证明上训练，其中每个证明步骤的可正确性是可验证的 [@shao2024deepseekmath; @deepseek2025r1]。Let's Verify Step by Step 的关键发现是过程监督的奖励模型（PRM）优于结果监督的奖励模型（ORM）[@lightman2024letsverify]。PRM 要生效，中间步骤必须可单独验证——这一属性只有结构化、可分解的表示才能提供。

**芯片设计**。Google 的芯片布局工作使用 Verilog 网表作为输入，设计规则检查（DRC）作为奖励信号 [@mirhoseini2020rlchip]。DRC 检查可制造性约束（间距、层规则、时序）并返回通过/失败裁决。表示（网表 + 标准单元库）将功能意图与几何实现解耦，使奖励可计算。

共同结构是：

```text
结构化表示 → 确定性检查器 → 快速反馈闭环 → RLVR 训练可行
```

物理工程当前缺失前两项。CAD/BIM 模型不是为检查而结构化的；下游审查不够快。闭环是断开的。

### 3.2.2 可测试预测

该假说产生具体预测：

- **P1.** 在其 ADL 输出上收到确定性 ESA 通过/失败反馈的 Agent，将比只收到自然语言任务描述的 Agent 产生更少违规的设计。
- **P2.** P1 的改进在控制模型能力（相同基座模型、相同任务难度）时仍然成立。
- **P3.** L0–L4 分层诊断格式将比平坦错误列表实现更快的错误纠正，因为结构化诊断允许 Agent 按层定位和优先排序失败。
- **P4.** 通过 ESA 的 Agent 设计将比通过更粗粒度检查器或完全无检查的设计产生更少下游错误（碰撞、装配失败）。

SD-HWE-Bench（§6.3）被专门设计以测试 P1–P3。P4 是一项长期实证目标。

## 3.3 为什么这是软件工程贡献

为一个新领域设计源代码表示是一项软件工程问题。它需要关于以下内容的决策：

- **语法与语义**。原子单元是什么（Part）？关系如何声明（配合）？如何在不嵌入部件身份的前提下表达空间布局？这些是语言设计问题，类比于设计类型系统或 IR。
- **分层验证**。哪些检查足够便宜可在每次按键时运行（L0）？哪些检查需要全局分析但仍保持确定性（L3–L4）？哪些检查本质上是非确定性的或过于昂贵（L5–L6）？这是静态分析管道设计问题。
- **工具链集成**。设计工件如何与 Git（diff、blame、bisect）、CI（pre-commit hook、门禁）和诊断（结构化输出、编辑器集成）交互？这些是软件工程基础设施问题。
- **可扩展性**。用户如何在不修改核心引擎的情况下新增 Part Family、规则或领域词汇？这是一个插件和 API 设计问题。

EaC 回答了物理工程的这些问题，但其框架——文本原生 DSL + 分层静态分析 + 工具链集成——是领域无关的。同一模板可应用于临床试验方案、法律合同或基础设施配置。在此意义上，EaC 既是一个面向物理工程的具体系统，也是一套将 SE 基础设施扩展到可文本化但当前欠结构化的领域的方法论。
