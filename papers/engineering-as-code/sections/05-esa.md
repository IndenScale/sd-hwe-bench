# 5. ESA：工程静态分析

Engineering Static Analysis（ESA）是使 EaC 的质量左移策略可执行的机制。它消费 ADL 声明并在设计提交前检查确定性规则，将合规审查从下游审查前移到上游门禁。

## 5.1 从 ACC 到 ESA

传统自动合规检查（ACC）运行在已完成的 CAD/BIM 模型或 IFC 文件上 [@eastman2009acc; @zhang2019acc]。这造成三个结构性问题：

1. **信息损失**。设计逻辑被冻结为几何；逆向提取功能关系是有损的。
2. **高反馈成本**。违规被发现时设计已耦合为跨专业模型。
3. **无 RLVR 信号**。ACC 只能判断已完成的设计，不能奖励生成中的 Agent。

除了这些时序问题，模型 ACC 还存在两项更深层的语义缺陷（见 §2.2.1）。首先，纯几何碰撞检测产生大量假阳性——管道穿墙、桥架交叉——因为模型对**预期**关系没有表示。其次，基于规则的 ACC 依赖标准化命名约定，迫使昂贵的检查前翻模，侵蚀了 ACC 本应提供的效率。

ESA 从根本上解决这些问题。它检查 ADL 声明——而非几何——因此能够获取：

- **Part 身份和类型**（Family schema），使规则在语义类别上运行（如"任意 `PDUFamily` 实例"）而非命名模式（如"任意名为 `PDU-*` 的对象"）；
- **显式 Mate 关系**（PML），使检查器知道哪些对象对是**预期**交互的——并能将它们从碰撞检测中排除（管道穿墙是一个声明的 Mate，而非需要抑制的碰撞）；
- **分层检查**（L0–L4），使语法错误、引用错误和业务规则违规在不同阶段被截获，每层有针对性诊断。

结果不是更快的 ACC。它是一个以根本上不同的输入运行的检查器：设计**意图**而非设计**实例化**。这一区分——检查声明 vs. 检查模型——与使编译器类型检查器在本质上不同于事后二进制分析工具的区分相同。编译器拥有类型系统；二进制分析工具必须推断类型。

### 5.1.1 具体示例：机柜功率预算

以下对比检查某机柜内设备总功率是否超过其 PDU 容量。

**基于模型的 ACC 方法**。检查器必须：(a) 解析 BIM 模型以识别机柜对象，(b) 识别服务器对象，(c) 通过几何包含或命名约定判断哪些服务器在哪个机柜中，(d) 从属性集中提取功率额定值（如果已填充），(e) 提取 PDU 额定值（如果已填充），(f) 比较。步骤 (a)–(e) 依赖模型质量和命名规范。如果某台服务器名为 `DELL-R740-SRV01` 而非 `SERVER-R740-01`，规则引擎可能遗漏它。如果功率额定值存储在特定于项目的共享参数中，规则必须按项目定制。

**ESA 方法**。规则遍历所有 `PDUFamily` 实例，遍历其 `outputs`，累加所连负载的 `rated_power_w`，与每路输出的 `rated_power_w` 比较。规则是 12 行 Python，独立于命名约定，对使用相同 Part Family 的任何项目有效。诊断精确指向具体 PDU 输出和超限负载。

这不是速度的对比。这是规则能否在不依赖每项目定制条件下**被编写**的对比。

## 5.2 为什么叫"静态分析"

ESA 借用软件工程中的"静态分析"一词，因为它共享三个特征：

1. **不执行物理仿真**。推理的是声明和约束，而非模拟物理。
2. **结果确定性**。相同的 ADL 声明始终产生相同的通过/失败裁决。
3. **成本低廉**。毫秒到百毫秒完成，适合 pre-commit hook 和 CI 门禁。

如同 lint 和类型检查不能取代单元测试，ESA 也不能取代 CAE/CFD 或人工审查。它将大量低级、确定性错误拦截在前端，使昂贵验证专注于真正需要专家判断的问题。

## 5.3 四条操作性纲领

ESA 遵循四条原则，旨在使其在真实工程实践中可用。

**纲领 I：规则可豁免**。物理边界条件无法被有限规则集穷尽。ESA 允许获得授权的工程师豁免特定规则，但每次豁免必须记录为设计历史的一部分并触发下游强化验证（例如更高精度的 CAE 仿真或物理测试）。目标是将"超规范"行为从非正式决策转化为可审计的风险管理。

**纲领 II：聚焦底线规则**。初期部署应聚焦于机械性、确定性的条款：防火间距、疏散净宽、最小净高、功率预算、U 位冲突。这些规则具有二值裁决、争议空间小、自动化收益高。规则库应支持模块化配置，使项目可组合国家规范、地方标准和企业内控。

**纲领 III：左移与信噪比优化**。软件工程通过 CI/CD 实践验证了 lint 和类型检查在提交前拦截低价值错误，使代码审查聚焦于架构和逻辑 [@chiari2024iacstatic]。ESA 将同一逻辑应用于工程设计：在提交前拦截功率越界、U 位冲突和接口不匹配，使下游人类专家和昂贵分析工具专注于真正需要专业判断的问题。

**纲领 IV：AI 辅助规则库构建**。规则库构建是 EaC 落地的关键瓶颈。在专家监督下，大语言模型可从规范文本、历史审查意见和设计指南中提取初步形式化规则草稿 [@yang2024llmacc; @nakhaee2024kgllm]，降低规则工程的初始成本。长期看，规则库应采取开源社区治理模式：核心规则由权威机构维护，领域规则由从业者贡献，通过版本控制和认证机制保证质量。

## 5.4 规则目录

截至本文撰写时，piki 实现 64 条规则，其中 62 条由插件注册，2 条为内置接口检查。

| 插件 | 规则数 | 主要领域 |
|------|-------|---------|
| telecom | 28 | 电信机房、机架、服务器、PDU、线缆、光纤 |
| keyboard | 14 | 机械键盘装配、DFX、矩阵 |
| datacenter | 9 | 模块化集装箱、电源单元 |
| environments | 4 | 运行环境、材料兼容性 |
| manufacturing | 5 | 工艺、材料约束 |
| consumer_electronics | 3 | 小型电子产品网表式连接 |
| 内置接口检查 | 2 | 接口类型存在性、方向一致性 |
| **合计** | **64** | — |

电信插件是最成熟的。代表性规则包括：

| 规则 ID | 名称 | 层级 | 说明 |
|---------|------|-----|------|
| `INTERFACE-COMPAT-001` | 接口类型兼容性 | L2 | Mate 或 Connection 端点接口类型可配对 |
| `INTERFACE-CABLE-001` | 线缆-接口匹配 | L2 | 所选 `cable_type` 与端口接口类型一致 |
| `TELECOM-POWER-001` | PDU 功率预算 | L3 | PDU 负载不超过容量阈值 |
| `TELECOM-RACK-001` | U 位冲突 | L3 | 同一机柜设备 U 位不重叠 |
| `TELECOM-RACK-002` | 机柜容量 | L3 | 设备总高度不超过机柜可用 U 位 |
| `TELECOM-COLLISION-001` | 机柜内 3D 碰撞 | L4a | 基于 AABB 的空间冲突检测 |
| `TELECOM-WEIGHT-001` | 机柜承重 | L3 | 总重量不超过机柜承重限制 |
| `TELECOM-FLOOR-002` | 维护通道宽度 | L4a | 同排机柜间距满足维护通道要求 |

规则通过 `@rule(rule_id, name, priority, severity)` 装饰器注册。分层注册使项目可在逻辑设计阶段仅启用 L2，在详细布局阶段再增加 L3–L4a。

## 5.5 豁免机制

结构化豁免系统是 ESA 设计的一部分，当前 piki 原型仅支持 `--skip <rule_id>` 和 `warning_only` 配置。目标设计将每条豁免记录为包含以下字段的 YAML 文件：

| 字段 | 含义 | 示例 |
|------|-----|------|
| `rule_id` | 被豁免的规则 | `TELECOM-FLOOR-002` |
| `target` | 受影响对象 | `RACK-A01`, `RACK-A02` |
| `scope` | 范围（单文件、分支、全局） | `branch:feature/legacy-room` |
| `rationale` | 技术和业务理由 | "受既有建筑柱网限制" |
| `author` | 申请人 | 工程师 ID |
| `approver` | 授权人 | 高级工程师 / 合规负责人 |
| `expires_at` | 过期时间（可选） | `2027-06-20` |
| `downstream_tasks` | 下游强化验证任务 | `["CFD-thermal-sim", "site-survey"]` |
| `created_at` / `commit` | 审计追踪 | 时间戳和 Git 哈希 |

豁免与 ADL 文件一起提交并在 PR 中审查。豁免不是绕过规则，而是将责任从确定性规则引擎转移到更高成本的验证活动。如果下游任务未通过，豁免被重新考虑。

## 5.6 诊断输出

ESA 产出一个统一的诊断结构，可被终端、CI 面板、IDE 和 PR bot 消费。在电信机架样本上，`piki check` 在笔记本上 200ms 内完成，返回 1 个警告和 29 个通过：

```text
[PASS] INTERFACE-COMPAT-001: 接口类型兼容性检查
...
[FAIL] TELECOM-FLOOR-002: 机柜维护通道宽度检查
       机柜 RACK-A01 与 RACK-A02 同排间距 -600.0mm 小于要求 600.0mm
============================================================
总计: 0 错误, 1 警告, 29 通过
============================================================
```

JSON 格式暴露相同数据，兼容 LSP，可驱动终端摘要、JUnit XML 面板、IDE 叠加层和 GitHub Checks API 评论。

## 5.7 CI/CD 与 Pre-commit 集成

ESA 被设计为在软件工程师已使用的工作流中运行。典型的 EaC CI/CD 管道包含以下阶段：

| 阶段 | 触发 | 内容 | 失败策略 |
|------|-----|------|---------|
| Lint | 每次提交 | YAML/TOML 格式、行尾空白、命名约定 | 拦截提交 |
| Parse | 每次提交 | ADL 语法（L0） | 拦截合并 |
| Schema 检查 | 每次提交 | pydantic / JSON Schema（L1） | 拦截合并 |
| Link 检查 | 每次 PR | 引用完整性、Mate 一致性（L2） | 拦截合并 |
| 规则检查 | 每次 PR | 业务与快速几何规则（L3–L4a） | 拦截合并 |
| 交付物构建 | 合并后 | BOM、图纸、端口映射 | 报告警告 |
| Nightly 回归 | 定时 | 全样本套件、几何、CAE/CFD | 报告并通知 |

Pre-commit 钩子在本地镜像较便宜的检查，使工程师在提交前获得亚秒级反馈。同一配置文件驱动 pre-commit 和 CI，防止"本机通过、CI 失败"的不一致。

## 5.8 下游验证协议：ESA 的边界

ESA 的范围被有意限制为可仅从声明进行确定性检查的内容。它覆盖 L2–L3（核心）和 L4a（轻量几何扩展）。L0–L1 由 ADL 加载器处理。L4b（精确几何碰撞检测，需要 CAD 内核）、L5（物理仿真，需要 CAE/CFD 求解器）和 L6（人工/专家签审）属于**下游验证**——对工程正确性至关重要，但不属于 ESA。

这些下游阶段的集成遵循一项协议，该协议是 EaC 工作流架构的架构主张，而非当前 piki 原型的功能：

1. **触发**。`piki check` 通过后，CI Pipeline Action 调用下游工具——将 ADL 声明导出为 STEP/USD，以 CAD 内核运行精确碰撞检测（L4b），或生成网格文件并调用 CFD/FEA 求解器（L5）。
2. **反馈回流**。下游发现以与 ESA 兼容的 Diagnostic 格式回流到 ADL 项目，引用特定 Instance、LayoutEntry 或 Mate 标识符。这意味着下游工具只需产出结构化诊断输出，无需理解 ADL 内部实现。
3. **关注点分离**。下游发现不反馈到 ESA 规则库。它们形成独立的 L4b/L5 诊断集合，正如芯片设计中 SPICE 仿真结果和 DRC 报告是独立、互补的验证通道。基于下游发现的豁免引用下游诊断，而非 ESA 规则 ID。

这一协议定义了 EaC 的设计时验证与更广泛的物理验证生态之间的**接口**。它不要求 piki 成为 CAD 或 CAE 平台；它要求 piki 产出可消费的输出和标准化的诊断封套。
