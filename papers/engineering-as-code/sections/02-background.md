# 2. 背景与动机

本章通过追溯一条已在三个相邻领域推动转型的因果链，论证 Engineering as Code 的动机，并说明工程设计为何至今无法参与这一链条。

## 2.1 RLVR 因果链

在代码、数学和芯片设计三个领域中，一个显著模式反复出现：结构化的正确性表示实现自动、确定性的检查；该检查提供快速奖励信号；这些信号使可验证奖励强化学习（RLVR）得以生效。

- **数学**。DeepSeekMath 在最终答案自动可验证的问题上训练。使用群组相对策略优化（GRPO），它在不需要单独奖励模型的情况下改进了推理 [@shao2024deepseekmath]。Lightman 等人证明过程奖励模型（PRM）——对中间推理步骤打分——优于结果奖励模型（ORM），凸显了步骤级可检查性的价值 [@lightman2024letsverify]。
- **代码**。SWE-RL 使用测试套件的通过/失败作为奖励信号训练 Agent 修复 GitHub issue，相比监督微调取得了显著提升 [@sweagent2025swerl]。Claude Code 及类似 Agent 在编译和测试提供即时反馈的仓库中运行 [@anthropic2025claude]。
- **芯片**。基于深度强化学习的芯片布局将设计规则检查视为宏单元布局中的快速、可微分信号 [@mirhoseini2020rlchip]；AMS-IO-Bench 评估 LLM 生成的 I/O ring 设计在 DRC/LVS 下的表现 [@liu2025amsio]。

该模式可以概括为：

```text
结构化、可检查的表示 → 亚秒级确定性反馈 → RLVR 有效 → Agent 能力跃迁
```

我们称之为**RLVR 因果链**。

## 2.2 工程设计为何受阻

物理工程设计——建筑、机械装配、电信基础设施——至今未能参与这一链条。障碍不在于模型规模不足或物理过于复杂，而在于**设计表示本身不可被机器按所需方式检查**。

当前主流表示是 CAD 和 BIM。它们将三件事耦合在同一工件中：

1. **功能身份**（组件是什么，做什么）
2. **几何实现**（位于何处，外观如何）
3. **工具特定状态**（厂商格式内部的约束、历史、参数）

这种耦合对 AI Agent 造成三个问题。

**信息损失**。设计意图一旦被冻结为几何，恢复原始功能关系——哪个端口连接到哪个、哪个机柜承载哪个服务器、哪些房间构成防火分区——就需要逆向提取。Eastman 等人和 Zhang 等人记录了 ACC 必须解析 IFC 或图纸模型并推断意图，这是一个有损且易出错的步骤 [@eastman2009acc; @zhang2019acc]。

**反馈延迟**。ACC 在设计工件完成后运行。发现违规时，设计已经是一个紧耦合的跨专业模型；修复成本高且缓慢。

**无训练信号**。因为检查是下游的且成本高昂，Agent 无法在生成期间将其作为奖励。它无法尝试一种布局，在毫秒级收到通过/失败奖励，并更新策略。RLVR 闭环在第一支箭头处断裂。

近期研究尝试将 LLM 直接应用于建筑规范解读 [@fuchs2024llmregs; @yang2024llmacc; @nakhaee2024kgllm]，但这些方法仍然运行在设计工件的下游。它们改进了审核方，而非生成方。

### 2.2.1 模型 ACC 的两项结构性缺陷

即使在其运行范围内——检查已完成设计——基于模型的 ACC 仍存在两项结构性而非算法性的缺陷。这些缺陷解释了为何仅仅加速 ACC 并不能弥合差距。

**纯几何碰撞检测产生的假阳性**。BIM 碰撞检测将每次体量交叉视为违规。一根管道穿越墙体触发碰撞，因为墙体模型缺少开槽。一条电缆桥架穿越梁被报为碰撞，即使两者在设计上存在物理间距。这些不是检测错误——算法正确识别了几何重叠。问题在于模型**没有语义信息**来描述两个对象之间的**预期**关系。按 EaC 的术语，模型缺失了一条 Mate 声明，说明"这根管道穿过这堵墙"或"这条桥架置于该梁上方指定偏移位置"。没有该声明，检查器无法区分有意的贯穿和真正的布局错误。结果是设计师学会忽略的大量假阳性，将真正违规淹没在噪音中。

**命名依赖与翻模开销**。基于规则的 ACC 工具——无论作用于 IFC、Revit 还是专有格式——依赖标准化命名约定来识别对象。一条规则如"所有命名为 `DUCT-SUPPLY-*` 的风管必须与结构构件保持 50mm 间隙"仅当每个项目中的每条风管遵守该命名约定时才有效。实践中，命名约定因事务所、项目甚至专业而异。ACC 运行前，模型必须被**翻模**——对象重命名、重组，有时甚至重建——以匹配规则引擎期望的命名模式。这一翻模步骤是人工的、昂贵的，且本身引入错误。更糟的是，它侵蚀了 ACC 本应提供的效率：如果必须在检查前翻模设计，你并没有自动化检查——你只是将人工劳动移到了上游。

这两项缺陷源于同一个根源：模型是设计的**唯一**结构化描述，但它是为几何表示而非语义查询而结构化的。一个必须通过检查几何来推断意图的规则引擎将永远受限于几何的语义贫乏。

## 2.3 软件工程与芯片设计的启示

两个成熟领域提供了一种不同的模型。

**基础设施即代码（IaC）**。Terraform 和 Ansible 等工具将基础设施表达为可版本化、可差分和可静态分析的文本声明。Chiari 等人实证表明 IaC 静态分析工具可在秒级检测数百条安全和合规违规 [@chiari2024iacstatic]。IaC 证明了"X as Code"加静态分析是一种实用、可规模化的组合 [@morris2022iac; @quattrocchi2023iacsurvey]。

**芯片设计**。Verilog 网表是逻辑设计的真相源；物理布局是下游视图。设计规则检查在布局和布线过程中运行，而不仅在 tape-out 之后。该表示被有意设计为文本化、层次化和在可视化之前可检查 [@ieee1364]。

工程设计没有等价于 Verilog 网表或 Terraform 文件的东西。EaC 的目标正是提供它。

## 2.4 研究缺口

该缺口可以精确表述。现有工作提供了：

- 丰富的**几何**表示（CAD/BIM/IFC）；
- 下游的**合规检查**（ACC），受困于假阳性噪音和命名依赖开销；
- 孤立的**AI 基准测试**用于工程任务 [@engdesign2025; @galanos2026aecbench; @maatouk2023teleqna]。

缺失的是一种**上游的、文本原生的、可被机器检查的设计表示**——可作为生成式 Agent 的训练目标和人类工程师的真相源。EaC 填补了这一缺口。
