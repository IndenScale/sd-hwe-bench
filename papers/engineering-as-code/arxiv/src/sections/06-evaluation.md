# 6. 评估

本章报告评估结果。全部样本检查和违规注入实验已完成；SD-HWE-Bench 处于设计阶段。

## 6.1 研究问题与方法论

我们回答三项研究问题：

- **RQ1.** ADL 能否以文本原生、可被机器检查的形式表达真实工程设计？
- **RQ2.** ESA 能否在设计时检测常见设计违规，延迟满足 CI/CD 门禁要求，且信噪比高于基于几何的 ACC？
- **RQ3.** 该表示是否提供了 RLVR 式 Agent 训练所需的可验证奖励信号？

评估使用 piki 仓库中的样本项目和一项受控的违规注入实验。测量指标包括：

1. **表达力**：每个样本能否完全以 ADL 声明；
2. **检查延迟**：`piki check` 完成 L0–L4a 验证的时间；
3. **规则覆盖**：适用的规则数量和类型；
4. **检测准确率**：ESA 是否在语义合理的配置中识别出故意注入的违规而不产生假阳性；
5. **交付物生成**：下游工件（BOM、面板视图、端口映射、线缆清单）能否产出。

所有测量在笔记本级机器上以热缓存进行；延迟为中位数五次运行。

**样本 03：机械键盘**。该样本展示了跨电子消费品领域的装配体类骨骼建模：CASE-01 为装配根，PCB、定位板、电池、开关、键帽和稳定器通过 parent-transform 链声明。样本通过所有 L2–L4a 检查（27 项通过，0 错误）。`LAYOUT-001` 空字符串判空缺陷修复后，连续 2D 网格布局的全部条目正确解析。该样本的通过为 ADL 三层架构从电信出发、跨越数据中心基础设施、延伸到电子消费品的跨领域通用性提供了证据。

## 6.3 违规注入实验

为测试 ESA 的检测准确率并对基于模型的 ACC 进行结构性对比分析，我们在电信样本上进行受控违规注入实验。

**实验设计**。在正确的 ADL 声明中注入 15 个已知违规，覆盖四个类别：

| 类别 | 数量 | 示例 |
|------|-----|------|
| L2：接口不匹配 | 4 | SFP28 光模块与 SFP+ 端口配对 |
| L2：引用完整性 | 3 | Mate 引用不存在的 Instance |
| L3：功率预算 | 4 | PDU 输出超载超过额定容量 |
| L3：空间冲突 | 2 | 两台服务器分配到重叠 U 位 |
| L4a：通道间距 | 2 | 机柜间距低于维护通道最小值 |

Table: 电信样本违规注入实验的违规类别。{#tbl:violation-categories}

然后运行 `piki check` 进行 ESA 检查。

**结果**：

1. **ESA 检测率**：15/15 违规全部检出（100%），假阳性为零——因为 ESA 检查语义类别（Family 类型、Mate 关系）而非几何。
2. **检查延迟**：`piki check` 完成全部 30 条规则的 L0–L4a 验证在 200ms 以内（笔记本，热缓存，五次运行中位数）。

### 6.3.1 与 ACC 的结构性对比

我们没有对 ACC 工具进行定量对比实验，因为这类对比在当前技术条件下意义有限且执行困难。原因有三。

**L2 语义违规对 ACC 是结构性不可达的。** 15 项违规中有 7 项属于 L2 类别——接口类型不匹配和引用完整性违规。这些违规的检测依赖 Part Family 类型系统和 Mate 关系声明，而这些信息在 IFC 几何模型中根本不存在。无论 ACC 工具（Solibri Model Checker、Navisworks 或学术原型）多么成熟，它无法检查输入表示中没有编码的属性。这不是精度的问题，是表示的边界。

**空间违规检测的信噪比差异是定性的。** 其余 8 项 L3/L4a 空间违规（功率预算、U 位冲突、通道间距）在原理上可被基于模型的 ACC 检出，但 ACC 必须通过命名约定或几何包含推断语义关系——步骤（a）识别哪些对象是服务器，（b）判断哪些服务器属于哪个机柜，（c）提取可能未填充的额定功率字段。命名模式因项目而异，功率参数可能存储在不同属性集中。同时，ACC 将在语义合理的几何交点上产生假阳性——线缆穿过理线孔、管道穿过墙体开槽——因为模型没有表示预期关系 [@eastman2009acc; @zhang2019acc]。最近的两个 ACC 系统验证了这一基本困难：BIM-Graph ACC 在 679 道防火规范问答题上达到 84.3% 的准确率，但在涉及多步空间推理链的规则上仍频繁失败 [@xiao2025bimgraph]；LLM-BIM Code Checking 管道依赖 LLM 生成 Revit Python 脚本处理语义歧义，但受限于特定项目定制 [@madireddy2025llmbim]。这两个系统都作用于建筑领域，而非电信机架领域——将它们适配到电信机架需要重新定义全部的命名约定和属性映射。

ESA 通过检查声明而非几何规避了这些限制。规则在语义类别（`ServerFamily`、`PDUFamily`、`rack-mount` Mate）上运行，而非命名模式。`TELECOM-FK-001` 直接从 Part Family 引用中解析 PDU identity，无需推断。`TELECOM-COLLISION-001` 通过 Mate 关系区分预期的设备兼容（服务器插入机柜）和真正的空间冲突（两个设备占据重叠的 U 位）。ESA 的优势不在于运行更快，而在于输入表示捕获了 ACC 必须从几何模型逆向推断的语义。

**ACC 社区缺乏标准化评测基准，这本身就是表示层问题的症状。** 经过 15 年的 ACC 研究，该领域仍未形成一个可与 SWE-bench 或 VerilogEval 相比的标准化评测基准。ACC 的输入——IFC/BIM 模型——高度异构：命名约定、建模习惯和 LOD 因项目而异，规范体系因地域而异。每次评测都需要定制化的翻模和规则适配。这恰恰验证了本文的核心命题：当设计意图被困在几何中时，不仅 Agent 无法获得训练信号，连基本的人类评估复现都变得困难。

ESA 的违规注入实验不需要这类定制化。15 项违规在 ADL 声明中以语义清晰的 YAML 编辑注入，`piki check` 提供了统一、可复现的评分函数。同样的违规注入方法可应用于任何遵循 ADL 三层架构的样本——正如样本 02 和 03 已展示的跨领域可移植性——因为规则以 Family 类型而非项目命名模式为索引。

这一对比直接验证了 §2.2.1 和 §5.1 中的核心主张：ESA 的优势源于它检查的是设计声明而非几何实例化产物。命名依赖翻模和纯几何碰撞假阳性不是 ACC 的实现缺陷，而是其输入表示的根本局限。

## 6.4 SD-HWE-Bench：Agent 评测基准 [设计阶段]

试点和违规注入实验验证可行性和检测准确率，但不测试 Agent 能否从自然语言需求**生成** ADL。我们正在设计 **SD-HWE-Bench** 以填补这一空白。它将由配套论文报告；此处描述其设计以供审稿人评估更大的评估计划。

**任务范式**：

```text
输入：  自然语言工程需求
输出：  结构化 ADL 声明（piki YAML）
评分：  L0–L4 规则检查通过率 + 交付物质量 + L6 签审评估
```

与 SWE-bench 不同——后者测试对已有代码库的补丁生成 [@jimenez2024swebench]——SD-HWE-Bench 测试从零创建，因为声明式工程设计作为一种实践尚未规模化存在。

**初始领域：电信机架部署**。选择该领域是因为：

- 空间离散（机柜 U 位），避免连续 3D 约束求解；
- 约束以代数为主（功率预算、重量、U 位冲突）；
- 接口类型可枚举（SFP28、RJ45、IEC-C13/C14）；
- `piki check` 约 200ms 完成，满足 RLVR 对快速奖励信号的需求。

**计划指标**：

| 指标 | 定义 |
|------|-----|
| `Pass@1` | 首次 ADL 输出通过全部启用规则的百分比 |
| `Pass@k` | 至少一次通过 k 次采样的百分比 |
| `Latency` | 每次输出的 `piki check` 中位耗时 |
| `Coverage` | 任务需求在生成 ADL 中的覆盖率 |
| `Human sign-off` | 领域专家评定可接受的百分比（L6 代理） |

Table: SD-HWE-Bench 计划评估指标。{#tbl:metrics}

**与 RQ3 的连接**。SD-HWE-Bench 直接测试 RLVR 因果链：如果 ADL + ESA 提供了结构化、确定性的奖励信号，则使用该信号训练的 Agent 应优于无该信号的 Agent。这是信息表示假说的预测 P1（§3.2.2）。Benchmark 设计已完成；任务编写和基线 Agent 实验正在进行中。
