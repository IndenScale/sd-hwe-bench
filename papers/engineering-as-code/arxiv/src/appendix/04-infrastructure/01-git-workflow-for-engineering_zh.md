# Git Workflow：将 ISO 变更管理落实为可执行规则

在 Engineering as Code 范式下，Git 不仅是版本控制工具，更是工程变更管理、版本控制与资料管理的工作流基础设施。ISO 9001、ISO 19650、ISO 10007 等标准对变更可追溯性、审批记录与文件版本控制提出了明确要求；EaC 通过 Git 的 diff、branch、pull request 与 CODEOWNERS 机制，把这些要求从纸质流程转化为可自动执行的规则。

## 为什么传统工程变更管理难以落地

传统 CAD/BIM 工作流中，变更管理面临三重障碍：

- **版本不可比**。IFC 等交换格式将身份、几何与关系耦合在一张图中，允许多种等价序列化，使得 Git 差异比较失去语义 [@liu2023ifcversion]。
- **审批流程与数据脱节**。变更申请、审批单、图纸版本往往分散在邮件、PLM 系统或纸质文件中，难以与设计数据自动关联。
- **责任边界模糊**。谁对哪个专业、哪个构件的变更负责，通常靠口头约定或项目章程，而非嵌入工作流的可执行规则。

结果上，ISO 要求的变更控制常常沦为事后补录，而非事前门禁。

## Git 作为可执行变更管理

ADL 的文本原生特性使 Git 可以直接作用于工程意图。每一次提交都是一次变更记录，自动包含：变更内容、作者、时间戳、关联的审查意见与合并决策。这比传统变更单更完整，且无法被绕过。

### Diff：语义化的变更识别

ADL 的正交性使 Git diff 获得工程语义：

| diff 发生的目录 | 对应的工程决策 | 典型审查关注点 |
|---|---|---|
| `instances/` | Part 身份、型号、属性变更 | 设备规格、接口能力、电气参数 |
| `mates/` | 装配关系、接口配对、机械耦合变更 | 配合可行性、受力路径、连接逻辑 |
| `layouts/` | 空间位置、机架 U 位、坐标变换变更 | 布局合理性、维护空间、碰撞风险 |
| `connections/` | 信号流、能量流、物质流拓扑变更 | 拓扑正确性、冗余策略、线缆选型 |

例如，仅修改 `layouts/layout.yaml` 中的一行 `position_u`，审查者即可判断这是一次纯布局调整，无需检查几何模型。这种可解释性使 diff 本身成为变更影响分析的工具。

### Branch：设计方案的版本化分支

由于 Instance 文件不包含布局信息，同一组 Part 可以在不同分支中拥有完全不同的布局方案，而无需复制 Part 定义。这使得工程师可以像比较软件 feature branch 一样比较设计方案：

```text
main          A──B──C
                         \
branch-scenario-2         D──E
```

`branch-scenario-2` 可以只修改 `layouts/` 而保留 `instances/` 与 `mates/`，合并时冲突只可能出现在布局维度。这种方案级分支在传统 CAD/BIM 中几乎不可行，因为布局变更通常隐含在整体模型中。

### Pull Request：变更审批与讨论载体

PR 是工程变更的审批单元。它天然包含：

- **变更描述**：提交信息、关联需求或问题单；
- **审查讨论**：针对具体行或文件的评论；
- **自动化检查**：CI 运行的 ESA 规则结果；
- **审批记录**：谁批准了合并、何时批准。

这些信息自动沉淀在版本历史中，满足 ISO 对变更可追溯性的要求，无需额外填写变更单。

### CODEOWNERS：责任边界的可执行规则

CODEOWNERS 文件将工程责任映射到文件路径，例如：

```text
# 电气工程师负责 instances/ 下的 PDU 与服务器
instances/pdus/       @electrical-lead
instances/servers/    @electrical-lead

# 布局工程师负责 layouts
layouts/               @layout-lead

# 机械工程师负责 mates
mates/                 @mechanical-lead
```

当 PR 修改这些文件时，必须获得对应 CODEOWNER 的批准才能合并。这把“谁对什么负责”从项目章程转化为可执行的门禁规则，直接落实 ISO 中的职责分配与审批控制要求。

## 与 ISO 要求的对应

| ISO 要求 | Git 机制 | 可执行形式 |
|---|---|---|
| 变更控制 | Commit + PR | 所有变更必须经过 PR 审查与合并 |
| 版本识别 | Git tag + SemVer | 发布节点打标签，Part 包使用语义化版本 |
| 审批记录 | PR review + merge commit | 审批人、时间、讨论自动记录 |
| 可追溯性 | `git blame` + 提交历史 | 每行声明可追溯到人与变更 |
| 资料管理 | 文本文件 + LFS | 源文件入版本库，二进制产物用 LFS 或制品仓库 |
| 职责分配 | CODEOWNERS | 文件级负责人强制参与审查 |

## 与下游二进制产物的区隔

Git 管理的是**设计意图的声明**，而非 CAD 渲染、CFD 网格、仿真结果等二进制产物。后者由 ADL 声明派生，应被 `.gitignore` 或通过大文件存储（LFS）单独管理。这一区隔保证了行级差异比较始终作用于人类可读的工程决策，而非工具生成的噪声，也使版本控制符合 ISO 对“受控文件”与“派生文件”分离的管理思想。
