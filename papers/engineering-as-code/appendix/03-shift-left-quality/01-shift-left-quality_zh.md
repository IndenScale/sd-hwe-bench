# 工程设计中的质量左移

质量左移（Shift-Left Quality）是指将缺陷发现与修复活动尽可能前移到设计流程的早期阶段。在软件工程中，这一理念已被 CI/CD、lint、类型检查和单元测试广泛验证：越早发现问题，修复成本越低，反馈循环越快 [@chiari2024iacstatic]。Engineering as Code 的核心主张之一，就是把同样的左移逻辑引入工程设计。

## 工程设计的质量光谱

工程设计的质量保证活动可以排成一条从左到右的光谱：

```text
语法检查 → 风格检查 → 静态分析 → 几何验证 → 多物理场仿真 → 受控测试 → 售后召回
```

左侧活动成本低、反馈快、可完全自动化；右侧活动成本高、反馈慢、依赖物理原型或专家判断。理想情况下，设计应尽可能在左侧被验证，把右侧资源留给真正需要物理直觉与复杂仿真的问题。

在软件与芯片领域，"X as Code" 表示已将大部分质量活动压缩到光谱左端：代码提交即触发 lint、类型检查、单元测试；Verilog 编写后立即执行语法检查、形式验证与逻辑综合。错误在设计最早、成本最低的阶段被发现并修复。

传统工程领域则呈现**质量右移**：设计审查依赖人工读图；碰撞检测只能在几何模型完成后进行；合规审查通常在图纸发布阶段甚至施工阶段才发生。这导致错误发现成本高、反馈周期长，也让生成式 AI 无法获得训练所需的秒级确定性信号。

## 质量右移的根因

质量右移并非工程行业的主观选择，而是由设计表示形态决定的：

1. **逻辑–几何耦合**。CAD/BIM 将功能属性、接口规格与三维几何绑定在同一对象中，设计意图无法被独立提取和校验。
2. **数据–工具耦合**。设计真相源常常是专有格式，开放标准如 IFC 主要作为下游交换格式，而非可被规则引擎直接消费的源 [@liu2023ifcversion]。
3. **缺少文本原生表示**。没有一种机器可消费、可版本化的“Design as Code”形式，使得 lint、类型检查、静态分析等方法无从附着。

只要设计意图被困在几何模型中，质量活动就只能发生在模型完成之后。

## 左移的实现路径

EaC 通过三项设计实现质量左移：

1. **文本声明作为真相源**。ADL 将设计意图从几何中解耦，使规则引擎可以直接消费设计逻辑。
2. **分层校验体系**。L0–L1 在加载期完成解析与 schema 校验；L2–L4a 作为工程静态分析（ESA）在提交前运行；L4b–L6 作为下游验证按需触发。
3. **CI/CD 与 pre-commit 集成**。校验嵌入版本控制工作流，每次提交和 PR 都自动执行，形成与软件工程类似的快速反馈闭环。

## ESA 在左移中的角色

Engineering Static Analysis（ESA）是质量左移的**核心执行机制**。它负责在光谱左端完成可快速、确定性执行的规则检查（L2–L3 为核心，L4a 为快速几何扩展），在问题进入昂贵下游流程之前将其拦截。ESA 不是全部验证活动，而是左移策略中“用自动化规则引擎替代事后审查”的具体实现。

后续章节将详细展开 ESA 的原则、分层边界、规则目录、豁免机制与输出形态。
