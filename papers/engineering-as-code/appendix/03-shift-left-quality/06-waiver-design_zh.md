# 豁免机制设计

ESA 的“规则可豁免”纲领要求一种结构化、可审计的豁免机制。当前 piki 原型尚未实现完整的豁免子系统；规则只能通过 `--skip <rule_id>` 临时跳过，或通过 `piki.toml` 将特定规则降级为 `warning_only`。本节描述目标设计，作为未来实现的规格。

## 豁免记录的数据模型

一条豁免记录至少应包含以下字段：

| 字段 | 含义 | 示例 |
|---|---|---|
| `rule_id` | 被豁免的规则标识 | `TELECOM-FLOOR-002` |
| `target` | 豁免作用的对象或对象组 | `RACK-A01`, `RACK-A02` |
| `scope` | 豁免范围（单文件、单分支、全局） | `branch:feature/legacy-room` |
| `rationale` | 技术与业务理由 | “受既有建筑柱网限制，通道无法扩展” |
| `author` | 申请人 | 工程师 ID / 签名 |
| `approver` | 授权人 | 高级工程师 / 合规负责人 |
| `expires_at` | 过期时间（可选） | `2027-06-20` |
| `downstream_tasks` | 下游强化验证任务 | `["CFD-thermal-sim", "site-survey"]` |
| `created_at` | 创建时间 | 提交时间戳 |
| `commit` | 关联 Git 提交 | `a1b2c3d` |

## 豁免与 Git 工作流的集成

豁免记录以 YAML 文件形式存放于项目目录（例如 `.piki/waivers/`），并随 ADL 声明一起提交。典型工作流如下：

1. **本地发现**：运行 `piki check` 发现违规；
2. **申请豁免**：工程师创建 `waiver-<id>.yaml`，说明理由与下游任务；
3. **PR 审查**：豁免记录作为 PR 的一部分被审查，合规负责人可要求补充仿真或测试；
4. **合并与追踪**：豁免随代码合并进入主分支，`piki report waivers` 可列出所有活跃豁免及其到期状态；
5. **过期提醒**：CI 可在豁免接近 `expires_at` 时发出提醒，防止临时偏差永久化。

## 豁免与下游 CAE 的联动

豁免不是“绕过规则”，而是将规则责任从自动化规则引擎转移到更高成本的验证手段。因此，每个豁免必须关联一个或多个下游任务：

- 几何违规 → CAE 仿真或物理原型测试；
- 功率预算越界 → 热仿真或现场测量；
- 材料兼容性例外 → 实验室加速老化测试。

这些任务的执行结果应反向写入豁免记录，形成闭环。若下游任务未通过，则豁免应被撤销或重新评估。

## 当前局限

如前所述，piki 目前仅提供 `--skip` 和 `warning_only` 两种临时机制，尚未实现上述结构化豁免记录。论文正文将这一局限作为当前边界诚实披露，并列入未来工作。
