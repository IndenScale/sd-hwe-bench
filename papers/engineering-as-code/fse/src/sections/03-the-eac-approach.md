# 3. Engineering as Code 方法

## 3.1 EaC 是什么

**Engineering as Code（EaC）** 将结构化、文本化的设计声明作为物理工程系统的唯一真相源，并围绕该表示构建版本控制、静态分析、CI/CD 等 SE 工具链。它不是 CAD/BIM 的替代品——几何建模和仿真仍是必需的——而是分离关注点：设计意图在 ADL 中以文本表达，几何实例化和分析消费该声明，类似芯片设计中 Verilog 与 DRC 的关系 [@ieee1364]。

EaC 由三根支柱构成：ADL（§4，表示层）、ESA（§5，验证层）和信息表示假说（§3.2）。

## 3.2 信息表示假说

> **信息表示假说**。物理工程中 AI 的滞后主要由缺乏结构化、文本原生、可被机器检查的设计表示引起，而非模型能力不足或物理内在复杂性。

该假说是**信息论**的：即使模型足够强大，当前表示也不支持驱动代码、数学和芯片领域 AI 进步的学习信号——快速、确定性的正确性反馈。

相邻领域证据：**代码**（SWE-bench/SWE-RL 以测试为奖励 [@jimenez2024swebench; @sweagent2025swerl]）、**数学**（可验证步骤上的 PRM [@shao2024deepseekmath; @deepseek2025r1; @lightman2023letsverify]）、**芯片**（Verilog + DRC [@mirhoseini2020rlchip]）。共同结构是：结构化表示 → 确定性检查器 → 快速反馈 → RLVR 可行。物理工程当前缺失前两项。

### 3.2.1 可测试预测

- **P1.** 收到 ESA 通过/失败反馈的 Agent 比仅收到自然语言描述的 Agent 产生更少违规。
- **P2.** 在控制模型能力时 P1 仍成立。
- **P3.** L0–L4 分层诊断比平坦错误列表更快纠错。
- **P4.** 通过 ESA 的 Agent 设计产生更少下游错误。

SD-HWE-Bench（§6.4）测试 P1–P3；P4 为长期目标。

## 3.3 为什么这是软件工程贡献

为新领域设计源代码表示涉及语法语义、分层验证、工具链集成和可扩展性。EaC 的框架——文本原生 DSL + 分层静态分析 + 工具链集成——是领域无关的。
