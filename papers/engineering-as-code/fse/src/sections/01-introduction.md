# 1. 引言

软件工程的核心基础设施是文本源代码：可被版本化、静态分析和测试，为 RLVR 提供结构化、可验证反馈 [@lightman2023letsverify; @deepseek2025r1; @sweagent2025swerl; @mirhoseini2020rlchip]。物理工程——机柜、结构、HVAC——仍依赖 CAD/BIM/IFC 等几何中心工件，将设计**意图**与几何**实例化**耦合 [@buildingsmart2023ifc]。自动合规检查（ACC）因此必须在设计完成后从几何逆向推断关系，导致假阳性高、每项目翻模昂贵，且无法为生成式 Agent 提供快速奖励信号。瓶颈不在算法，而在**表示基础设施**。

本文提出 **Engineering as Code（EaC）**：为物理工程设计文本原生、可被机器检查的设计意图表示及配套 SE 工具链。EaC 不是"AI for engineering"，而是**为工程领域实现源代码表示的软件工程方法论**。

@tbl:se-gap 对比软件工程与物理工程的可验证反馈能力。

| 维度 | 软件工程 | 物理工程（现状） |
|------|---------|---------------|
| 主工件 | 文本 | 几何（.stp/.rvt/.ifc） |
| 正确性检查 | 编译器 / lint / 测试 | 下游 / 人工审查 |
| 检查延迟 | 毫秒至分钟 | 小时至周 |
| 版本控制 | Git 行级 diff | PDM/PLM 二进制 diff |
| CI 管道 | push → lint → test | 人工审查门禁 |
| 关系编码 | 显式 | 隐式于几何或命名约定 |
| 信噪比 | 语义类别 | 命名依赖；几何碰撞假阳性 |

Table: 软件工程与物理工程在可验证反馈维度上的差距。{#tbl:se-gap}

## 1.1 本文贡献

1. **ADL：装配体定义语言**（§4）。三层文本原生设计语言，将部件身份（PDL）、配合（PML）和布局（PLL）正交分离。
2. **ESA：工程静态分析**（§5）。确定性分层规则引擎（L0–L4），将合规检查前移到提交前门禁。
3. **信息表示假说**（§3）。可测试主张：工程 AI 的瓶颈在于缺乏可计算设计表示。本文给出预测并描述 SD-HWE-Bench 用于评估 Agent 的 ADL 创作能力。

开源运行时 **piki** 在电信机架部署中 200ms 内完成接口兼容性、功率预算、U 位冲突、承重等检查。

## 1.2 为什么这是 SE 问题

为新领域设计源代码表示是软件工程问题：DSL 设计 [@hudak1996dsl; @fowler2010dsl; @volter2013dsl]、静态分析 [@cousot1977abstract]、CI 与诊断格式。ADL 继承这一传统；ESA 复现编译器"语法→类型→语义→空间"分层，只是此前缺少表示层使其成为可能。IaC 已证明"X as Code + static analysis"在云基础设施可行 [@chiari2024iacstatic]；EaC 将其外推到物理基础设施。
