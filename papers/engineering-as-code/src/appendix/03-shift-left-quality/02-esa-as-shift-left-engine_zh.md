# ESA：质量左移的静态分析引擎

Engineering Static Analysis（ESA）是实现质量左移的核心机制。它的作用是把目前发生在设计完成之后的合规审查，前移到设计生成阶段，并以确定性规则引擎的形式持续运行。ESA 不是独立的验证体系，而是“质量左移”理念在 EaC 中的具体执行层。

## 为什么需要 ESA

在传统工程工作流中，Automated Compliance Checking（ACC）通常在设计图纸或 BIM 模型完成之后运行 [@eastman2009acc; @zhang2019acc]。它检查的对象是已经固化的几何产物，而不是设计意图本身。这带来三个结构性问题：

1. **信息损失不可逆**。设计逻辑被冻结在几何线条或三维体素中，反向提取疏散拓扑、防火分区、设备关系等逻辑时会出现严重信息损失。
2. **反馈成本过高**。发现问题时，设计已经耦合为跨专业的整体，局部修改可能触发连锁变更。
3. **无法支持生成式 AI**。ACC 只能判定“已完成的设计是否合规”，不能为正在生成设计的智能体提供毫秒级反馈，因此无法嵌入 RLVR 训练闭环。

ESA 通过直接消费 ADL 文本声明，把规则检查从“事后审计”转向“生成时门禁”。这与芯片设计中的 Design Rule Checking（DRC）理念一致：规则不是等版图完成后再检查，而是嵌入布局与布线工具，在生成过程中持续拦截违规构型 [@ieee1364]。

## ESA 在左移中的位置

在质量光谱上，ESA 占据左端：

```text
语法检查 → 风格检查 → ESA（L2–L4a） → 几何验证 → 多物理场仿真 → 受控测试 → 售后召回
              ↑
           ESA 负责这一段
```

它负责拦截确定性的、可规则化的错误，例如接口不匹配、功率预算越界、U 位冲突、维护通道宽度不足等。这些问题原本需要在图纸审查或碰撞检测阶段才能发现，现在可以在提交前被自动拦截。

## ESA 的范围

ESA 不是一切验证活动的统称。它特指**可在设计阶段快速、确定性执行的规则检查**：

- **核心范围：L2–L3**
  - L2：引用完整性、Mate 自洽性、接口兼容性；
  - L3：跨记录业务规则，如功率预算、承重、容量、间距。
- **边界扩展：L4a**
  - 快速几何规则，如 U 位冲突、AABB 碰撞、维护通道宽度。这些规则使用轻量几何方法，不调用 CAD 内核，因此可视为静态分析向几何领域的有限延伸。

以下活动**不属于 ESA**：

- L0–L1：ADL 加载期的解析与 schema 校验；
- L4b：精确几何干涉检查（需要 CAD 内核）；
- L5：物理仿真（CFD/FEA，需要数值求解器）；
- L6：AI / 人工审批（需要专家判断或训练模型）。

这一范围界定基于**计算成本与确定性**。ESA 层具备毫秒级、可重复、二值化的特点，适合嵌入 CI/CD 与 pre-commit；L4b 之后成本高、参数敏感，应作为下游验证环节。

## 为什么叫“静态分析”

借用软件工程中 static analysis 的命名，是因为 ESA 与代码静态分析共享三个特征：

1. **不执行物理仿真或实际运行**：只基于声明之间的关系和约束进行推理；
2. **结果确定性**：给定相同的 ADL 声明，总是产生相同的判定；
3. **成本低廉**：可在提交前高频率运行。

正如同 lint 和类型检查不会取代单元测试与集成测试，ESA 也不会取代 CAE/CFD 或人工审查。它负责在质量光谱左端拦截大量低级、确定性错误，让昂贵的验证手段聚焦于真正需要专业判断的问题。
