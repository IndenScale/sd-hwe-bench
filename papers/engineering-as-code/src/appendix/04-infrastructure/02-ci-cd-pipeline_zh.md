# EaC 的 CI/CD 流水线

Continuous Integration / Continuous Delivery（CI/CD）是软件工程中验证代码变更的标准实践。在 EaC 范式下，由于设计意图以文本声明表达，同样的流水线可以直接迁移到工程设计：每次提交 ADL 文件都触发一层自动化的校验、报告与门禁。

## 流水线阶段

一个典型的 EaC CI/CD 流水线包含以下阶段：

| 阶段 | 触发条件 | 执行内容 | 失败策略 |
|---|---|---|---|
| Lint | 每次提交 | YAML/TOML 格式、尾随空白、文件命名规范 | 阻止提交 |
| Parse | 每次提交 | ADL 语法解析（L0） | 阻止合并 |
| Schema Check | 每次提交 | pydantic / JSON Schema 校验（L1） | 阻止合并 |
| Link Check | 每次 PR | 引用完整性、Mate 自洽性（L2） | 阻止合并 |
| Rule Check | 每次 PR | 业务规则与快速几何规则（L3–L4a） | 阻止合并 |
| Deliverable Build | 合并后 | 生成 BOM、施工图、端口映射等 | 报告警告 |
| Nightly Regression | 定时 | 大规模样本、几何干涉、CAE/CFD 回归 | 报告并通知 |

这一分层结构与软件 CI/CD 高度一致：低成本、高频率的检查放在提交阶段；高成本、需要更多上下文的检查放在 PR 或夜间阶段。

## 门禁与合并策略

ESA 规则引擎在 CI 中以门禁形式运行。任何 ERROR 级别诊断都应阻止 PR 合并，这与单元测试失败阻止代码合并的逻辑相同。WARNING 级别诊断可以配置为：

- **硬门禁**：不允许合并，直到 warning 被修复或显式豁免；
- **软门禁**：允许合并，但要求在后续 PR 中处理；
- **信息性**：仅生成报告，不阻止合并。

门禁策略应根据项目阶段与风险等级调整。对于安全关键领域（如航空航天、核电），建议采用硬门禁；对于早期方案探索，可采用软门禁或信息性策略。

## 报告与可观测性

CI 流水线的输出应服务于三类受众：

- **工程师**：人类可读的摘要，指出违规文件、行号与修复建议；
- **审查者**：PR 中的检查评论，将诊断直接绑定到变更行；
- **管理者**：趋势仪表盘，显示规则通过率、技术债务积累与豁免到期情况。

诊断格式采用 LSP 兼容结构（`severity`、`code`、`message`、`location`、`related_information`），使得同一份输出可以被终端、IDE、GitHub Checks、JUnit 仪表板同时消费。

## 与基础设施即代码的类比

IaC 社区已经证明，对声明式配置运行 CI/CD 静态分析是可行且高效的：tfsec、checkov 等工具可以在数秒内检测出数百项安全与合规违规 [@chiari2024iacstatic]。EaC 将同一实践扩展到工程设计，区别仅在于规则库的内容从安全策略变为工程规范。
