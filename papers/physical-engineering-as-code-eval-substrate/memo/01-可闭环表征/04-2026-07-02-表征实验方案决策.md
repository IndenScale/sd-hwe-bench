# 表征实验方案决策：完成判定、模型矩阵与建模软件

日期：2026-07-02

本 memo 回答当前表征实验设计中的四个具体问题，并记录 2026-07-02 后续决策：CUA/GUI 不进入主实验条件，只在相关工作与威胁讨论中处理。

1. 怎么判定完成。
2. 为什么不把 CUA 模式纳入主实验。
3. OpenSCAD MCP / OpenSCAD-only / ADL + OpenSCAD 三组条件是否合适。
4. 使用什么建模软件，是否使用 Blender。

## 结论摘要

建议将表征实验收敛为一个小而硬的 controlled representation study：

- 最终提交对象统一为可归档的工程工件包，而不是 GUI 截图或自然语言说明。
- 最终正确性统一由离线 checker 判定，不由中间工具、模型自评或人工视觉判断判定。
- 主实验条件收敛为三组：OpenSCAD code-only、OpenSCAD MCP/tool-mediated、ADL + OpenSCAD。
- CUA 不进入主实验分母。DeepSeek Flash/Pro 缺少视觉能力不再是实验矩阵问题；Kimi/GPT 这类 visual-capable actor 的 GUI 操作能力改在相关工作和威胁讨论中处理。
- OpenSCAD 作为主建模与评分工件；Blender 不作为主建模软件，只作为可选的 CUA 前端或 mesh/export 辅助工具。

这组实验的论文主张不是“某个工具最好”，也不是重新评测通用 GUI agent，而是：在同等工程意图下，不同 artifact-bearing 表征接口形成 evaluation loop 的能力不同。

## 1. 怎么判定完成

完成判定分三层：attempt 完成、条件完成、实验完成。

### 1.1 Attempt 完成

一次 attempt 只有在产生完整 artifact 后才算完成。完整 artifact 至少包含：

- `manifest.json`：记录 task id、condition、model、actor、工具版本、timeout、isolation、run id。
- 最终提交工件：
  - OpenSCAD code-only：最终 `.scad` 源码。
  - OpenSCAD MCP/tool-mediated：工具调用日志、最终设计状态和导出的 `.scad`。
  - ADL + OpenSCAD：最终 ADL 工程文件、生成的 `.scad`、生成映射表。
- `score.json`：统一 checker 输出。
- `diagnostics.json`：结构化错误定位，至少能定位到 constraint id；ADL 条件下应尽量定位到 object/field。
- `actor_output.log` 或等价 transcript。

Attempt 的完成状态分为：

| 状态 | 含义 | 是否进入统计 |
|---|---|---|
| `pass` | 工件完整，离线 checker 全部通过 | 是 |
| `fail` | 工件完整，但 checker 未通过 | 是 |
| `invalid` | 工件缺失、无法解析、无法导出或无法重评 | 是，作为 invalid submission |
| `timeout` | 超时自然触发，artifact 保留 | 是，作为 runtime failure |
| `infrastructure_error` | 工具安装、runner 或 checker 自身故障 | 不计入模型能力统计，但必须报告 |

关键原则：不能用“看起来像”“GUI 中显示正确”“模型说完成了”作为 pass。最终 pass 只看归档工件是否能被同一 checker 复现通过。

### 1.2 条件完成

一个 representation condition 完成，需要满足：

- 至少完成一次 smoke run，证明该条件能从空 scaffold 产生可评分 artifact。
- 每个入选模型达到预设 attempt 数，例如每个任务 3-5 次。
- 所有失败样本都有 `score.json` 或明确的 `invalid/timeout` 原因。
- 可以从 artifacts 自动重建该条件的汇总表。

CUA 不作为条件完成的一部分。若保留本地 GUI smoke，只作为基础设施记录或 appendix note，不进入主实验分母。

### 1.3 实验完成

整组表征实验完成，需要满足：

- 三组主条件覆盖同一批任务、同一批最终 checker、同一套 token/time/repair budget。
- CUA 仅在相关工作与威胁讨论中说明，不作为本文自行报告的 pass-rate 条件。
- 主表可以由脚本从 run artifacts 生成，至少包含：
  - pass rate
  - invalid submission rate
  - feedback latency
  - localization granularity
  - repair success rate
  - scoring coverage
  - artifact reproducibility
- 至少提供一个 case study，展示同一个 pseudo-correctness 错误在三种主路径中的定位和修复差异。

若时间不足，最低可投稿版本应保留 OpenSCAD-only 与 ADL + OpenSCAD 的主对照；MCP 可降级为 appendix/case study。不能用未完成的模拟数字填正文结果。

## 2. 模型矩阵与 CUA 处理

DeepSeek Flash/Pro 没有视觉能力这一事实说明：CUA 主要是 actor/interface 能力问题，容易把“模型是否能看屏幕”和“表征是否可闭环”混在一起。因此本文不把 CUA 纳入主实验矩阵。

建议模型矩阵如下：

| 模型/Actor | MCP | OpenSCAD-only | ADL + OpenSCAD | 备注 |
|---|---:|---:|---:|---|
| Kimi + Kimi Code | 可跑 | 可跑 | 可跑 | 主实验矩阵 |
| Code + GPT 5.5 | 可跑 | 可跑 | 可跑 | 主实验矩阵 |
| DeepSeek-v4-Flash + Claude Code | 可跑 | 可跑 | 可跑 | 主实验矩阵 |
| DeepSeek-v4-Pro + Claude Code | 可跑 | 可跑 | 可跑 | 主实验矩阵 |

报告时只保留一张 comparable matrix：OpenSCAD MCP / OpenSCAD-only / ADL + OpenSCAD，覆盖所有模型。这样可以避免把“模型是否有视觉通道”和“表征路径是否闭环”混在一起。

## 3. 三组主条件是否合适

OpenSCAD MCP / OpenSCAD-only / ADL + OpenSCAD 三组是当前主实验的合适边界。CUA 作为相关工作和威胁项，不作为主实验组。

### 3.1 CUA（不进入主实验）

CUA 表示 agent 主要通过 GUI/CAD-like 操作建模，再导出可评分工件。它回答的是“agent 如何操作既有软件界面”，不是“最终工程表征是否形成 evaluation substrate”。

已有通用 CUA / GUI agent 研究可作为相关工作引用。本文主实验不自行评测 CUA，因为这会引入视觉能力、焦点控制、GUI 权限、操作延迟和窗口状态等 confounder。报告中应强调：CUA 可以改善交互效率，但如果最终状态不能成为稳定、可归档、可检查、可诊断的 artifact，它仍只是 interface layer，而不是 representation substrate。

### 3.2 MCP

MCP 表示 agent 通过工具/API 调用完成几何生成、检查或转换。

它回答的问题是：工具接口是否足以替代统一提交对象？

建议 MCP 条件不要让 agent 直接编辑 ADL，也不要直接自由写 OpenSCAD；否则会和 OpenSCAD-only 或 ADL 条件混淆。MCP 条件可以提供类似以下工具：

- `create_fixture(params)`
- `add_mounting_holes(params)`
- `add_slot(params)`
- `export_openscad()`
- `run_checker()`

最终 artifact 包含工具调用日志和导出的 `.scad`。这能测试工具调用历史是否足以支撑归档、diff、诊断和 repair。

### 3.3 OpenSCAD-only

OpenSCAD-only 表示 agent 直接编辑 `.scad` 源码。

它回答的问题是：code-like geometry 是否已经足以成为闭环表征？

这是关键对照组。OpenSCAD 已经具备文本化、可 diff、可执行和参数化优势，因此它是对 ADL/EaC 的强 baseline。若 ADL + OpenSCAD 优于 OpenSCAD-only，优势必须体现在语义约束覆盖、错误定位、跨对象引用、repair 粒度和交接成本，而不是简单说“ADL 更像代码”。

这一组还承担一个更重要的报告任务：说明为什么“在几何/B-rep/特征层写 checker”不足以成为通用 evaluation substrate。对于单个 fixture，OpenSCAD 或 B-rep 后处理可以检查孔径、孔距、槽宽、壁厚、包围盒和干涉；但这些检查通常绑定当前零件的坐标系、特征命名、设计意图和工况假设。它们能检查工艺约束的局部几何后果，却很难承载“这个孔属于哪个安装接口”“这个槽服务于哪个夹持对象”“这条边距规则来自哪个制造方法或组织标准”等工艺语义。因此 OpenSCAD-only 的价值不是作为弱 baseline，而是作为强 baseline：它证明几何层可检查，同时暴露几何层 checker 跨零件、跨项目复用困难。

### 3.4 ADL + OpenSCAD

ADL + OpenSCAD 表示 agent 编辑语义工程声明，由 generator 生成 OpenSCAD，并由 checker 同时检查语义层和几何层。

它回答的问题是：在 code-like geometry 之上增加对象、引用、约束和接口语义，是否改善 evaluation loop？

这一组应提供：

- ADL 源文件。
- ADL 到 OpenSCAD 的生成器。
- ADL object/field 到 OpenSCAD feature 的映射表。
- 同一最终 geometry checker。
- 额外 semantic checker，例如孔位属于哪个安装接口、槽宽对应哪个夹持对象、边距约束来自哪个制造规则。

ADL + OpenSCAD 的核心不是多加一种格式，而是把可执行约束放在更高的抽象层：对象、接口、角色、约束来源、制造方法和项目阶段。几何层负责落地和验证，语义层负责承载工艺约束。这样同一条规则才有机会跨零件、跨夹具族、跨项目复用，而不是每换一个 B-rep 拓扑或坐标系就重写 checker。

### 3.5 为什么不建议加入更多组

暂不建议加入 Revit、FreeCAD、Grasshopper、Excel 等额外主条件。它们会把实验变成工具横评，分散论文主张。

当前三组覆盖核心轴：

- 工具调用：OpenSCAD MCP
- code-like geometry：OpenSCAD-only
- semantic engineering substrate：ADL + OpenSCAD

这正好对应论文中“工具接口不等于 substrate、可计算不等于可闭环”的论证。

## 4. 使用什么建模软件

建议主建模工件选择 OpenSCAD，不以 Blender 作为主建模软件。

当前本机可用工具：

- `/opt/homebrew/bin/openscad`
- `/opt/homebrew/bin/blender`
- `/Applications/OpenSCAD.app`
- `/Applications/Blender.app`

### 4.1 OpenSCAD 作为主工件

OpenSCAD 适合作为主工件，原因是：

- 文本原生，天然可 diff、可版本化、可归档。
- 参数化能力足够支撑单零件工装夹具任务。
- CLI 可运行，适合隔离 runner 与自动评分。
- 可导出 STL/CSG，便于后续几何 checker。
- 与 ADL + OpenSCAD 的 generator 路径自然衔接。

建议最终标准化为：

```text
design.scad
metadata.json
score.json
diagnostics.json
```

ADL 条件再增加：

```text
design.adl.yaml
adl_to_scad_map.json
generated/design.scad
```

### 4.2 Blender 的定位

Blender 不建议作为主建模软件。原因是：

- 它偏 DCC/mesh 工作流，不是工程 CAD 约束建模的最佳代表。
- GUI 状态和操作序列难以成为稳定提交对象。
- 对孔位、基准面、装配接口、制造约束等工程语义表达较弱。
- 如果用 Blender Python 脚本建模，它又会变成另一种 code-like geometry，和 OpenSCAD-only 的实验含义重叠。

Blender 可以作为辅助工具，但不进入主实验：

- 导入 STL 后做 mesh 体积、包围盒、连通性等后处理。
- 生成非评分用预览，但预览不能作为 pass/fail 证据。

因此推荐表述是：

> OpenSCAD is the canonical representation artifact; GUI/CUA systems are treated as interaction layers in related work and threats, not as the scoring substrate of the main experiment.

### 4.3 是否考虑 FreeCAD

FreeCAD 比 Blender 更像工程 CAD，但当前本机没有检测到可用安装。即使后续安装，也建议不要作为第一版主条件，除非 CUA 的 reviewer-facing value 明显不足。

若未来讨论 FreeCAD，定位应是 GUI/CUA 相关工作的实现工具，而不是第四种主表征路径。

## 推荐任务形态

第一版任务建议使用“单零件工装夹具 / fixture”而不是完整 AIDC 布置。原因是它足够小，可以让 MCP、OpenSCAD-only 和 ADL + OpenSCAD 都 work；同时又能设置 pseudo-correctness 陷阱。

任务规格建议包含：

- 固定外形尺寸，例如 120 mm × 80 mm × 16 mm。
- 两个或四个安装孔，含孔径、孔距、边距、沉孔或避让。
- 一个夹持槽，含槽宽、槽深、中心线和公差。
- 两个定位销或限位台阶。
- 最小壁厚与加强筋。
- 坐标系/镜像陷阱，例如孔位视觉对称但相对基准面错误。
- 语义陷阱，例如槽宽满足几何尺寸但不满足被夹持对象的 clearance rule。

最终 checker 分两层：

1. Geometry checker：OpenSCAD 是否可编译，导出 mesh 是否满足尺寸、孔位、槽宽、包围盒、连通性等。
2. Semantic checker：设计是否满足规格条款、基准面、接口、制造规则和可解释映射。

OpenSCAD-only 可以通过 geometry checker，但可能在 semantic checker 上暴露弱点；ADL + OpenSCAD 应该能把 semantic checker 的诊断定位到对象和字段。

报告中应避免把这个 fixture checker 写成“几何层已经足够”。更准确的表述是：我们故意为特定零件构建强 geometry checker，让几何/B-rep 路径获得公平机会；如果即使在这种有利条件下，checker 仍表现出语义恢复、错误定位和跨任务复用困难，就能自然推出为什么可执行表征应构建在高于几何特征的工程语义层。

## 推荐最小实验矩阵

若先做 smoke：

| 阶段 | 任务数 | 模型 | 条件 | attempts |
|---|---:|---|---|---:|
| Smoke | 1 | Kimi + Kimi Code, Code + GPT 5.5 | MCP, OpenSCAD-only, ADL + OpenSCAD | 1 |
| Main | 1-2 | 四个模型 | MCP, OpenSCAD-only, ADL + OpenSCAD | 3 |

若资源紧张，优先级是：

1. OpenSCAD-only vs ADL + OpenSCAD。
2. MCP 作为工具接口对照。
3. CUA 不进入主实验；只在相关工作和威胁讨论中引用通用 GUI/CUA 研究。

## 论文写法建议

正文中不要写成：

> ADL 比 OpenSCAD / MCP 更好。

应写成：

> 在低耦合几何任务中，OpenSCAD-only 已经是强 baseline；差异主要不在生成能力，而在提交边界、语义覆盖、错误定位、repair 和重评能力。MCP 可以增强操作能力，但若最终工程状态不能成为稳定、可检查、可诊断的 artifact，它仍只是工具接口层，而不是 evaluation substrate。CUA 属于交互层，本文在相关工作中讨论，不作为主实验对照。

这能更稳地回应 reviewer 对参数化 CAD、OpenSCAD、GUI agent 和 MCP 工具调用的质疑。
