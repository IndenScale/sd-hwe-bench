# 可闭环表征：基础设施要求

本 memo 记录为了支撑“可闭环表征”实验，需要在 SD-HWE-Bench 基础设施中具备的能力。

2026-07-02 更新：主实验条件已收敛为 **OpenSCAD code-only / OpenSCAD MCP(tool-mediated) / ADL + OpenSCAD** 三组。CUA/GUI、NL-only、Revit/Grasshopper/Excel 等路径可作为相关工作、威胁讨论或未来扩展，不进入当前主实验分母。以下基础设施要求以三组 artifact-bearing 路径为准。

## 表征路径的标准化执行环境

正式实验需要为每条表征路径提供统一、可复现的执行容器，避免因为工具版本、安装方式、脚本依赖不同而引入噪声。

### NL-only（非主实验）

- Agent 直接输出 Markdown 或纯文本方案。
- 需要人工或半自动协议判定方案是否满足需求（用于对比 baseline）。

### CUA/GUI（非主实验）

- 需要录制/回放工具或 API 操作序列。
- 需要确定性的初始 GUI 状态。
- 评分依赖后处理：导出文件 → 解析 → DTS 检查。

### MCP/Tool

- 需要定义统一的工具集合和调用协议。
- 评分依赖工具返回 + wrapper，需记录所有调用日志。

### OpenSCAD code-only

- 需要统一 OpenSCAD 版本、库路径、渲染参数。
- 需要标准流程：**修改 `.scad` → 渲染/导出 STL → 进入同一 checker**。
- 需要记录 `.scad`、渲染日志、导出工件和 score。

### ADL + OpenSCAD

- 需要 ADL 源文件、ADL 到 OpenSCAD 的 generator、ADL object/field 到 OpenSCAD feature 的映射表。
- 生成的 `.scad` 与 OpenSCAD code-only 进入同一 checker。

## 跨路径评分统一

最终评分必须对所有路径使用同一套完整约束集合。不同路径只影响“Agent 如何生成和提交”，不影响“什么算正确”。

为此需要：

- 一个统一的**约束目录**（constraint catalog），包含每条约束的 id、族、所属层、可执行状态、定位粒度。
- 一个统一的**评分器入口**，接收不同路径的产物并输出同构的 `ScoreResult`。
- 一个统一的**诊断渲染器**，把同一底层 critic 结果转换成不同路径 Agent 能消费的格式。

## 闭环能力度量工具

### 生成—验证闭环

- 记录从提交到完整评分的 wall-clock 时间。
- 记录是否能自动检测：语法错误、引用错误、静态约束违规、动态模型违规、几何/可建造性违规。
- 记录 Agent 是否能基于评分结果自动发起 repair。

### 专业—综合闭环

- 需要能按约束族（电气、热、结构、布局、施工、调度、交付物）单独统计违规。
- 需要能比较“本专业检查通过但跨专业检查失败”的方案比例。
- 需要能在任务 metadata 中显式标注跨专业依赖关系。

### 设计—建造闭环

- 需要 ConstructabilityCritic 和 EPCCritic 检查吊装、施工通道、VDC 工作面、资源约束、天气窗口、SLA。
- 需要把设计产物（ADL 或参数化导出）自动输入到施工排程仿真器中。

### 当前—未来闭环

- 需要构造“迁移实验”：同一表征应用于新场景（不同地点气候、不同设备库、不同规范版本）。
- 记录迁移所需的人工修改量：参数数量、文件数量、硬编码值数量、隐式依赖数量。

### 个人—组织闭环

- 需要构造“交接实验”：让未参与原始建模的 Agent 或工程师基于表征文档修改任务。
- 记录理解表征所需的时间/轮次、误改率、需要询问原作者的次数。
- 需要统计表征中的隐式依赖数量（例如硬编码 id、未声明的假设、脚本中的魔法数字）。

## Artifact 与指标

每个 run 的 manifest 应记录：

- `representation_path`：`openscad-code` / `openscad-mcp` / `adl-openscad`。
- `closed_loop_coverage`：五重闭环中哪些被启用。
- `formalization_cost`：人工建模步骤数、时间。
- `determinism_score`：重复运行同一提交是否得到同一评分。
- `feedback_latency_ms`：提交到首次结构化反馈的时间。
- `localization_granularity`：错误定位到 task / object / file / field 的比例。
- `repair_feasibility`：能否基于反馈自动修复，以及修复成功率。
- `cross_domain_conflict_count`：跨专业冲突检出数。
- `buildability_violation_count`：可建造性违规数。
- `migration_cost`：迁移实验的人工修改量。
- `handoff_cost`：交接实验的理解和修改成本。

聚合脚本需要输出：

- 三组主路径在闭环指标上的对比表。
- 闭环覆盖度与 pass rate / omission density 的关系曲线。
- 按任务复杂度分层的结果。
- 跨专业冲突和设计—建造违规的典型案例。

## 任务 metadata 扩展

任务 metadata 需要增加或派生以下字段：

- `scale_level`：low、medium、high。
- `coupling_level`：low、medium、high。
- `task_family`：layout、connection、cross-domain、co-design、epc 等。
- `constraint_families`：任务涉及的约束族列表。
- `requires_cross_domain_check`：是否需要跨专业联合检查。
- `requires_buildability_check`：是否需要设计—建造反馈。
- `expected_saturation`：是否低耦合饱和任务。

## 参数化路径的公平对照

为了让参数化路径与 ADL 公平比较，需要：

- 为参数化路径提供与 ADL 相同的 scaffold 信息（工程需求、约束描述、参考规范）。
- 允许参数化路径的 Agent 使用任何工具或脚本生成工件，但最终必须导出为可被 DTS 检查的中间格式。
- 记录参数化路径独有的成本：渲染时间、格式转换、脚本调试、隐式依赖管理。

## Actor 隔离与样本可信度

正式实验必须保留 actor 级隔离：

- actor workspace 迁出仓库。
- macOS 上启用 seatbelt 或等价内核级 repo read deny。
- run artifact 记录 `isolated=true` 与实际 `work_dir`。

如果 Agent 超时、卡住或没有落盘，只要 artifact 完整、timeout 自然触发、隔离成立，该样本应作为失败样本或运行时失败进入统计。不能手动终止后计入 pass/fail。

## 进入正式实验前的最低门槛

- 每个入选任务都有显式约束目录。
- 三组主表征路径都有最小可行执行协议。
- 评分器能接收不同路径的产物并输出同构结果。
- 能记录生成—验证、专业—综合、设计—建造、当前—未来、个人—组织五重闭环的度量。
- 至少完成一个 Kimi 或 Codex 小矩阵 smoke，确认 artifact、timeout、隔离和聚合链路可用。
- 至少完成一次参数化路径（如 OpenSCAD 或 Excel）与 ADL 的对照 smoke，确认闭环能力差异可被稳定测量。
