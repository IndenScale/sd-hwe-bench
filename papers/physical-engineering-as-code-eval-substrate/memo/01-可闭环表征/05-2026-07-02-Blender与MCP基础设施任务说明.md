# Blender 与 MCP 基础设施探索记录（已降级）

日期：2026-07-02

本 memo 记录早期探索阶段对 CUA / MCP / OpenSCAD / ADL + OpenSCAD 四组条件的基础设施判断。该四组矩阵已被后续决策降级：当前主实验只保留 OpenSCAD code-only / OpenSCAD MCP(tool-mediated) / ADL + OpenSCAD 三组，CUA/GUI 不进入主实验分母。

1. Blender 是否支持无头模式，还是必须有活跃窗口。
2. MCP 如何安装、封装和联调，才能进入表征实验矩阵。

2026-07-02 后续决策：本文主实验不再包含 CUA/GUI 条件，也不以 Blender 作为主建模软件。当前主实验收敛为 **OpenSCAD code-only / OpenSCAD MCP(tool-mediated) / ADL + OpenSCAD**。本 memo 保留为基础设施探索记录；其中关于 Blender GUI/CUA 的任务不进入主实验分母，可作为 appendix 或未来工作素材。

## 当前实测结论

### Blender

本机可用：

```text
/opt/homebrew/bin/blender
/Applications/Blender.app
Blender 5.1.2
```

已完成最小无头 smoke：

```bash
blender --background --factory-startup --python create_cube.py -- /tmp/.../cube.stl
```

结果：

```text
Info: File exported successfully
cube.stl exists, 684 bytes
```

结论：

- Blender 支持无头 `--background` 模式。
- 无头模式足以完成 Python 脚本建模、导入/导出、mesh 后处理和 STL 生成。
- 无头模式不需要活跃窗口。
- 但 CUA/GUI 条件如果定义为“agent 通过视觉和鼠标键盘操作 GUI”，仍然需要可见且可交互的 Blender/OpenSCAD/CAD 窗口。

因此应区分两种 Blender 用法：

| 用法 | 是否需要活跃窗口 | 是否适合正式评分 |
|---|---:|---:|
| Blender headless backend | 否 | 适合导出、转换、mesh 检查、smoke |
| Blender Python 建模 | 否 | 可作为工具后端，但会和 code-like geometry 条件重叠 |
| Blender GUI / CUA | 是 | 已降级为相关工作/未来扩展，不作为主实验条件 |

报告口径：

> Blender can run headlessly for deterministic export and mesh processing. A live window is only required when the experimental condition explicitly evaluates GUI/CUA interaction. Therefore Blender should not be treated as the scoring substrate; it is either a headless backend or a GUI frontend for the CUA subset.

### OpenSCAD

本机可用：

```text
/opt/homebrew/bin/openscad
/Applications/OpenSCAD.app
OpenSCAD version 2026.06.12
```

OpenSCAD 仍建议作为 canonical representation artifact：

- OpenSCAD-only 条件：agent 直接编辑 `.scad`。
- ADL + OpenSCAD 条件：agent 编辑 ADL，由 generator 生成 `.scad`。
- 历史 CUA / MCP 条件：最终也要导出 `.scad` 或可转换为 `.scad` / `.stl` 的工件，并进入同一 checker。当前主实验只保留 MCP。

### MCP

当前项目环境中未安装 Python MCP 包：

```text
mcp: missing
fastmcp: missing
```

仓库当前也没有表征实验专用 MCP server、MCP client smoke 或 actor 联调脚本。

由于项目全局约束禁止 `npm` / `npx` / `pip`，MCP 安装与依赖管理必须走 `uv`。第一版建议使用 Python MCP SDK 或 FastMCP，而不是 Node MCP server。

## 设计判断

### Blender 不应成为主建模软件

Blender 的 headless 能力可以降低基础设施风险，但它不改变表征实验的主判断：

- 如果用 Blender GUI，让 agent 点击建模，它属于 CUA 条件。
- 如果用 Blender Python，让 agent 写脚本建模，它属于 code-like geometry 工具条件，和 OpenSCAD-only 含义重叠。
- 如果用 Blender 做导出、导入、mesh 统计，它只是后处理 backend。

因此 Blender 不作为主建模软件，不作为 pass/fail 证据来源。正式 pass/fail 仍来自归档工件和统一 checker。

### 历史 CUA 条件必须拆成两个 smoke（不进入主实验）

CUA 的基础设施应拆成：

1. **Backend smoke**：不启动 GUI，验证 Blender/OpenSCAD 可以无头导出与检查工件。
2. **GUI smoke**：启动可见窗口，验证 visual-capable actor 能看到窗口、操作 UI、导出工件，并保留操作日志。

Backend smoke 通过不代表 CUA 成立。它只能证明工具链可运行。CUA 成立必须证明 actor 能通过 GUI 完成从 scaffold 到 artifact 的路径。

DeepSeek Flash/Pro 没有视觉能力，跳过 GUI smoke；Kimi + Kimi Code 与 Code + GPT 5.5 可进入 GUI smoke。

### MCP 条件应保持窄接口

第一版 MCP 不应封装完整 Blender/CAD 能力。否则实验会变成“谁写了更强工具”的比较。

建议 MCP 工具只覆盖 fixture 任务所需的最小操作：

- `create_fixture_base(length, width, height, material)`
- `add_mounting_holes(pattern, diameter, countersink, edge_clearance)`
- `add_clamping_slot(width, depth, centerline, clearance_rule)`
- `add_locator_pins(count, diameter, positions)`
- `export_openscad(path)`
- `run_geometry_checker(path)`
- `run_semantic_checker(path)`

MCP 条件的提交对象应包括：

```text
mcp_call_log.jsonl
design_state.json
exported/design.scad
score.json
diagnostics.json
```

MCP 的关键观察不是“工具能不能生成几何”，而是：工具调用历史和导出状态能否形成稳定、可归档、可定位、可 repair 的提交对象。

## 任务拆解

### T1. Blender headless 基线固化

目标：把当前手工 smoke 固化为仓库脚本。

建议产物：

```text
tools/representation/blender_headless_smoke.py
tools/representation/run_blender_headless_smoke.sh
```

注意：若写 shell 脚本，不得依赖截图，不得要求人工窗口操作。

完成标准：

- `blender --background --factory-startup --python ...` 成功运行。
- 生成一个确定性的 STL。
- 输出 JSON manifest，记录 Blender 版本、命令、输出路径、文件大小、耗时。
- 失败时返回非零退出码。

### T2. OpenSCAD smoke 固化

目标：确认 OpenSCAD CLI 可以作为 canonical artifact backend。

建议产物：

```text
tools/representation/openscad_smoke.scad
tools/representation/run_openscad_smoke.py
```

完成标准：

- `openscad` 可从 `.scad` 导出 `.stl`。
- 记录 OpenSCAD 版本。
- 检查输出文件存在且非空。
- 输出 `openscad_smoke_manifest.json`。

### T3. Fixture checker 最小版

目标：构建第一版 geometry checker，作为 OpenSCAD-only 与 ADL + OpenSCAD 的共同评分入口。

建议产物：

```text
src/sd_hwe_bench/representation/fixture_checker.py
tests/test_representation_fixture_checker.py
```

第一版 checker 可以先检查：

- OpenSCAD 是否可编译。
- STL 是否可导出。
- mesh 包围盒是否满足外形尺寸。
- 基本连通性或三角面非空。
- 根据 scaffold metadata 检查孔位、槽宽、边距等可从设计参数或导出 metadata 得到的约束。

注意：如果 STL 直接解析难度过高，第一版允许要求 agent 同时提交 `metadata.json`，checker 先对 metadata + artifact 做联合检查。论文中要明确这是强 geometry baseline，不是假装 B-rep 能天然承载全部工艺语义。

### T4. Python MCP 依赖安装方案

目标：用 `uv` 引入 MCP SDK，不触碰 `npm/npx/pip`。

候选方式：

```bash
uv add mcp
```

或如果选择 FastMCP：

```bash
uv add fastmcp
```

完成前需要确认：

- 依赖是否支持当前 Python 版本。
- 是否引入过大的非必要依赖。
- 是否需要放在 optional extra，例如 `representation` 或 `mcp`，避免污染核心 benchmark runtime。

建议先不直接写入主 dependencies，而是评估以下两种方案：

| 方案 | 优点 | 缺点 |
|---|---|---|
| 主依赖 `dependencies` | runner 简单 | 影响所有用户安装 |
| optional extra `mcp` | 边界清晰 | 运行命令需 `uv sync --extra mcp` |

建议采用 optional extra。

### T5. Fixture MCP server

目标：提供一个最小 stdio MCP server，暴露 fixture 建模工具。

建议产物：

```text
src/sd_hwe_bench/representation/mcp_server.py
tests/test_representation_mcp_server.py
```

工具集合：

- `create_fixture_base`
- `add_mounting_holes`
- `add_clamping_slot`
- `add_locator_pins`
- `export_openscad`
- `run_checker`

实现原则：

- server 内部维护一个明确的 `design_state.json`，不要只依赖工具调用历史。
- 每个工具调用都 append 到 `mcp_call_log.jsonl`。
- `export_openscad` 必须生成可归档 `.scad`。
- `run_checker` 调用同一个 fixture checker，不为 MCP 条件另写评分标准。

完成标准：

- 本地 Python client 能通过 stdio 启动 server。
- client 连续调用上述工具后生成 `.scad`。
- checker 能返回结构化 `score.json`。
- 调用日志可重放或至少可审计。

### T6. MCP client smoke

目标：不依赖任何大模型，先用脚本联通 MCP server。

建议产物：

```text
tools/representation/mcp_fixture_smoke.py
```

流程：

1. 启动 stdio MCP server。
2. 调用 `create_fixture_base`。
3. 调用 `add_mounting_holes`。
4. 调用 `add_clamping_slot`。
5. 调用 `export_openscad`。
6. 调用 `run_checker`。
7. 写出 smoke manifest。

完成标准：

- 单命令可跑通。
- 无需 GUI。
- 无需模型。
- 输出 artifact 与 score。

### T7. Actor-MCP 联调

目标：让模型 actor 在 MCP 条件下使用工具完成同一个 fixture 任务。

建议分两步：

1. **本地 agent shell smoke**：把 MCP 工具说明写入 prompt，让 actor 通过可用客户端调用 MCP server。
2. **正式 runner 接入**：在 SD-HWE-Bench run manifest 中记录 `representation_path=mcp`、MCP server command、工具版本、调用日志路径。

完成标准：

- Kimi + Kimi Code 或 Code + GPT 5.5 至少完成 1 次 MCP attempt。
- artifact 包含 `mcp_call_log.jsonl`、`design_state.json`、`exported/design.scad`、`score.json`。
- 同一 checker 可用于 OpenSCAD-only 与 MCP 条件。

### T8. CUA GUI smoke

目标：验证 visual-capable actor 能通过可见 GUI 路径完成导出。

建议先选一个 GUI 前端：

| 前端 | 优点 | 风险 |
|---|---|---|
| OpenSCAD GUI | 和主工件一致 | GUI 自动化价值有限 |
| Blender GUI | 本机已安装，视觉操作明显 | 偏 DCC，不是工程 CAD |
| FreeCAD GUI | 更像 CAD | 当前未安装，增加依赖 |

第一版建议：

- 不把 GUI smoke 放进主矩阵前置条件。
- 先用 Blender GUI 做 CUA feasibility case。
- 最终导出 `.stl` 或 `.scad` 进入同一 checker。

完成标准：

- 只在 Kimi + Kimi Code / Code + GPT 5.5 上跑。
- DeepSeek Flash/Pro 标记为 `not_applicable_no_vision`。
- 保留操作日志、最终导出工件和 score。
- 不使用截图作为评分依据。

## 推荐优先级

第一阶段，先完成无 GUI、无模型的可复现链路：

1. T1 Blender headless smoke。
2. T2 OpenSCAD smoke。
3. T3 Fixture checker。
4. T4/T5/T6 MCP server + client smoke。

第二阶段，再接模型：

1. OpenSCAD-only actor attempt。
2. MCP actor attempt。
3. ADL + OpenSCAD generator attempt。

第三阶段，最后做 CUA：

1. Blender/OpenSCAD GUI smoke。
2. visual-capable actor CUA attempt。

这样可以避免一开始被 GUI 权限、窗口焦点、视觉能力和工具安装同时卡住。

## 风险与处理

| 风险 | 处理 |
|---|---|
| Blender GUI 需要焦点或 macOS 权限 | CUA 独立为 subset，不阻塞主矩阵 |
| Blender Python 太强，混淆 MCP/OpenSCAD baseline | Blender Python 只做 backend，不让 actor 直接写 Blender 脚本作为主条件 |
| MCP 工具过强，变成手写 solver | 工具只提供建模原语，checker 仍统一 |
| MCP 依赖污染核心运行时 | 使用 optional extra |
| STL 几何解析复杂 | 第一版允许 metadata + artifact 联合评分，逐步加强 mesh/B-rep 检查 |
| GUI smoke 无法稳定复现 | 降级为 appendix case study，不进入主结果表 |
