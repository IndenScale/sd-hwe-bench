# 可闭环表征：实验设计

本 memo 记录“可闭环表征”章节的实验设计。实验目标不是证明 ADL 比参数化设计“更好”，而是拆解：在同等工程意图下，不同表征路径能在多大程度上形成完整的评估闭环。

## 实验目标

实验需要支撑以下问题：

- 参数化设计是否已经能独立支撑工程 AI 的评估闭环？
- 缺少哪些闭环能力时，Agent 会稳定产出 pseudo-correct 方案？
- 生成—验证、专业—综合、设计—建造、当前—未来、个人—组织五重闭环，哪一重对任务成败影响最大？
- 这些闭环能力差异在不同任务复杂度下是否稳健？

## 比较路径

我们将同一工程需求交给五类表征路径完成，比较其闭环能力。

| 路径 | 典型工具/形式 | 核心表征 | Agent 操作 | 预期特点 |
|---|---|---|---|---|
| NL-only | 自然语言方案 | 文本描述 | 直接生成说明 | 无 diff、无验证、无 repair |
| CUA/GUI | CAD/BIM/表格 GUI | 视觉/交互状态 | 点击、输入、导出 | 交互脆弱、难复现、后处理评分 |
| MCP/Tool | 外部 API、数据库、插件 | 工具调用参数 | 查询、调用、汇总 | 接口层，无统一提交对象 |
| Parametric/Config-driven | OpenSCAD、CAD families、Revit families、Excel、Grasshopper/Dynamo | 参数、族、配置表、脚本 | 修改变量、族参数、配置表 | 可计算、可 diff、可复用，但通常只覆盖生成 |
| ADL/EaC | 声明式工程对象与关系 | Part、属性、引用、连接、配合、约束、调度 | 修改工程声明文件 | 支持生成—验证—repair 完整闭环 |

实验重点不在“谁能生成”，而在“谁能闭环”。

## 轴 A：闭环能力覆盖度

把五重闭环能力作为独立变量，观察缺失哪一重会导致什么后果。

| 条件 | 缺失/保留的闭环 | 操作方式 | 目的 |
|---|---|---|---|
| Gen-only | 只保留生成能力，其余均缺失 | 参数化或脚本生成工件 | 测试“可计算但不闭环”的上限 |
| Gen + Verify | 保留生成 + 验证 | 生成后用独立检查器评分 | 测试验证闭环本身的价值 |
| Gen + Cross-domain | 保留生成 + 跨专业综合 | 要求电气/热/结构/布局联合检查 | 测试跨专业闭环的价值 |
| Gen + Buildable | 保留生成 + 可建造反馈 | 生成后检查吊装、施工、排程 | 测试设计—建造闭环的价值 |
| Gen + Transferable | 保留生成 + 可迁移 | 同一表征用于不同项目/团队 | 测试当前—未来闭环的价值 |
| Gen + Documented | 保留生成 + 可交接 | 新人/Agent 无需作者解释即可理解和修改 | 测试个人—组织闭环的价值 |
| Full closed-loop | 五重闭环均保留 | ADL + DTS 完整评估 | 测试完整闭环的上限 |

注意：最终评分应使用完整约束集合，各条件只控制表征路径和 Agent 可见的闭环能力，不降低评分标准。

## 轴 B：任务复杂度分层

实验任务按“规模”和“耦合复杂度”分层，避免用单一总体平均值掩盖结构差异。

| 任务类型 | 规模 | 耦合复杂度 | 预期现象 |
|---|---:|---:|---|
| 简单几何/布置 | 低-中 | 低 | 参数化和 ADL 可能都能通过，差异主要在可审计性和可交接性 |
| 单专业参数化优化 | 中 | 低-中 | 参数化路径表现好，ADL 优势不明显 |
| 跨文件/跨专业约束 | 中 | 中-高 | 参数化开始漏跨专业引用和冲突，ADL 优势显现 |
| 选型-布置-调度联合优化 | 中-高 | 高 | 只有完整闭环才能稳定收敛，参数化显著落后 |
| 设计-施工联合排程 | 中-高 | 高 | 设计—建造闭环成为决定性因素 |

低耦合任务不是反例。它们用于证明：可闭环表征在简单任务上也能提供稳定性、可审计性和防回归价值；真正体现差距的是高耦合系统级任务。

## 核心指标

本实验不以 pass rate 为唯一指标，而关注闭环能力：

- **任务形式化成本**：从工程需求到可评测任务需要多少人工建模步骤。
- **提交确定性**：同一提交是否能稳定复现评分。
- **反馈延迟**：从提交到结构化错误反馈的时间。
- **错误可定位性**：错误能否定位到对象、字段、引用或规则。
- **repair 可行性**：Agent 是否能基于反馈进行局部修复。
- **评分覆盖度**：是否覆盖语法、引用、工程约束、几何、仿真、交付物、可建造性。
- **跨专业冲突检出率**：同一方案在本专业检查和跨专业联合检查下的差异。
- **可迁移成本**：把同一表征应用到新场景（新地点、新设备库、新规范）所需的人工调整量。
- **交接成本**：新成员或 Agent 在没有原作者解释的情况下理解、修改、复现表征所需的时间/轮次。

## 关键对照

实验设计应避免把“表征文本化程度”和“闭环能力”混为一谈。至少需要以下对照：

- 同一任务、同一 Agent、同一 scaffold，在五类表征路径下比较。
- 同一参数化模型，在“仅生成”和“生成 + DTS 验证”两种条件下比较。
- 同一 ADL 任务，在只反馈本专业错误 vs. 反馈跨专业冲突 vs. 反馈设计—建造冲突三种诊断条件下比较。
- 对低耦合与高耦合任务分别报告结果，不只给一个总体平均值。

## 预期章节图表

- 表征路径闭环能力矩阵：五类路径在七个闭环维度上的定性/定量评分。
- 覆盖率-质量曲线：x 轴为闭环能力覆盖数（0-5），y 轴为 pass rate 或 omission density。
- 任务复杂度分层表：低/中/高耦合任务在不同表征路径下的表现。
- 跨专业冲突 case studies：本专业检查通过、跨专业联合检查失败的案例。
- 可迁移/交接成本对比：同一参数化模型与 ADL 模型在新项目上的适配过程。

## 与“可执行约束”实验的关系

“可闭环表征”实验关注的是“表征形态”：不同的工程提交形态能在多大程度上支持完整闭环。

“可执行约束”实验关注的是“约束反馈”：在同一表征形态下，约束是否可执行、诊断是否可定位、repair 是否高效。

两者是正交但互补的：

- 没有可闭环表征，可执行约束就缺少附着对象和引用语义。
- 没有可执行约束，可闭环表征就只停留在“可提交、可 diff”，无法产生真实正确性反馈。

因此，两个实验应共用同一套任务池和评分器，但从不同角度切入。
