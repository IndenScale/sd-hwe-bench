# 可闭环表征：实验设计

本 memo 记录“表征鸿沟 / representation experiment”的实验设计。实验目标不是证明 ADL 比 MCP 或 OpenSCAD “更好”，而是拆解：在同等工程意图下，不同表征路径能在多大程度上形成可提交、可检查、可诊断、可修复、可归档的 evaluation loop。

2026-07-02 决策更新：**CUA/GUI 不进入主实验条件**。CUA 解决的是“agent 如何操作既有软件界面”的交互问题，而本文关注的是“操作完成后留下的工程表征能否进入评测闭环”。因此主实验聚焦 OpenSCAD code-only、OpenSCAD MCP/tool-mediated、ADL + OpenSCAD 三组 artifact-bearing 表征路径；CUA 作为相关工作与威胁讨论引用通用 GUI/CUA agent 研究，不在本文自行跑分。

表征实验不是事务优先级，而是论文三重鸿沟中的一个实证轴。

## 实验目标

实验需要支撑以下问题：

- OpenSCAD code-only、OpenSCAD MCP/tool-mediated 和 ADL + OpenSCAD 都能完成一个规格明确的单零件 / 工装夹具建模任务时，它们的 evaluation-loop 性质有什么差异？
- 一个几何体“看起来对”但规格语义错时，不同表征路径能否定位错误并支持 repair？
- OpenSCAD 作为 code-like geometry 已经提供可执行工件；ADL + OpenSCAD 额外提供的价值是否体现在语义约束、提交边界、诊断和归档重评上？
- 这些差异是否能用小规模、可复现、可审稿的 representation experiment 表达，而不是变成庞大的工具能力竞赛？

## 任务：单零件工装夹具 OpenSCAD 建模

当前表征实验收敛为一个受控任务：给定夹持对象、安装界面、制造约束和检查器，要求 Agent 生成一个规格良好的工装夹具或工程几何体，并最终产出可运行的 OpenSCAD 模型。

任务不追求复杂系统规模，而追求“几何可见正确”和“规格语义正确”之间的可测差异。建议规格包含：

- 外形尺寸、基准面、安装孔位、孔径和孔边距。
- 夹持槽、定位销、限位台阶、倒角或圆角。
- 最小壁厚、加强筋、工具可达性和简单可制造性约束。
- 一个容易产生 pseudo-correctness 的陷阱，例如孔位镜像、槽宽略小、孔边距不足、倒角遗漏、定位销与夹持槽坐标系不一致。

最终离线评分始终使用同一套规格 checker / geometry checker。条件之间只改变 Agent 可操作的表征路径和反馈方式，不改变最终正确性标准。

## 比较路径

我们将同一工程需求交给三类表征路径完成。重要前提是：三者都应当 work。实验比较的不是“谁能不能做”，而是谁能形成更稳定的评估闭环。

| 路径 | 典型形式 | Agent 操作 | 预期观察 |
|---|---|---|---|
| OpenSCAD code-only | code-like geometry | 直接编辑 `.scad` 源码 | 强 baseline：文本化、可 diff、可执行，但规格语义多依赖约定和后处理恢复 |
| OpenSCAD MCP / Tool | 工具/API/脚本接口 | 通过工具生成、修改或检查 OpenSCAD | 工具调用可审计，但提交边界和语义约束可能散落在调用历史与设计状态中 |
| ADL + OpenSCAD | 语义规格 + code-like geometry | 用 ADL 描述零件特征、约束和接口，再生成或约束 OpenSCAD | 同时保留语义提交对象和可执行几何工件，适合 localized feedback 与 repair |

### CUA 的处理

CUA/GUI agent 不作为本文主实验条件。已有通用 CUA 研究已经证明视觉 agent 可以操作软件界面；本文不重新评测“能否点击 CAD GUI”。我们把 CUA 视为交互层：它可能改善人机操作效率，但不自动解决提交边界、语义定位、归档重评和组织交接问题。论文中应在相关工作引用 CUA / GUI agent 研究，并在威胁讨论中说明：未来视觉 agent 与 CAD GUI 的结合可能改变操作成本，但最终仍需要 artifact-bearing representation 才能进入 evaluation loop。

实验重点不在“谁能生成”，而在“谁能闭环”。

## 控制变量

- 同一任务规格、同一最终 checker、同一模型和相近 token/time budget。
- 所有条件最终都必须交付可归档的 OpenSCAD 工件或可转换为 OpenSCAD 的工件。
- 最终评分不看中间工具是否声称成功，只看归档工件是否满足规格。
- repair budget 固定；若条件无法提供 localized feedback，应记录人工转译或状态恢复成本。

如果只做一个任务，建议每个条件 3-5 次 attempt。主结果应避免声称统计显著性，而强调受控 representation case study 和 artifact-level observability。

## 核心指标

本实验不以 pass rate 为唯一指标，而关注闭环能力：

- **任务形式化成本**：从工程需求到可评测任务需要多少人工建模步骤。
- **提交确定性**：同一提交是否能稳定复现评分。
- **反馈延迟**：从提交到结构化错误反馈的时间。
- **错误可定位性**：错误能否定位到几何特征、尺寸参数、坐标系、规格条款或语义约束。
- **repair 可行性**：Agent 是否能基于反馈进行局部修复。
- **语义约束覆盖度**：规格条款中有多少能进入自动检查，而不是停留在自然语言说明、OpenSCAD 约定或工具调用状态中。
- **invalid submission rate**：无法解析、无法导出、无法重评或缺少提交边界的比例。
- **归档与重评能力**：是否能在新环境中重新渲染、重新检查和 diff。
- **交接成本**：新成员或 Agent 在没有原作者解释的情况下理解、修改、复现表征所需的时间或轮次。

## 关键对照

实验设计应避免把“工具能完成任务”误写成“工具形成了 evaluation substrate”。至少需要以下对照：

- 同一夹具规格、同一最终 checker，在 OpenSCAD code-only、OpenSCAD MCP/tool-mediated、ADL + OpenSCAD 三条路径下比较。
- 同一 OpenSCAD 工件，在“几何可运行”和“规格语义满足”两个层面分别评分。
- 同一错误类型，例如孔位镜像或槽宽不满足夹持对象，比较三条 artifact-bearing 路径是否能定位、解释和 repair。
- 报告所有条件都能 work，避免把实验写成单方面胜负表。

## 预期章节图表

- 表征路径闭环能力矩阵：三条主路径在 deterministic artifact、semantic coverage、localized feedback、repairability、archive/re-score 上的表现。
- 单零件 case study：一个看似正确但规格失败的 OpenSCAD 工件，展示错误如何在三条路径中被发现和修复。
- 规格条款覆盖表：哪些条款进入了 checker，哪些只存在于 prompt、OpenSCAD 约定、工具调用历史或 ADL 语义层中。
- repair trajectory 表：每轮提交的失败条款、反馈粒度和修复结果。

## 与“可执行约束”实验的关系

表征实验关注的是“提交对象和表征路径”：不同的工程提交形态能在多大程度上支持完整闭环。

约束实验关注的是“约束反馈”：在同一表征形态下，约束是否可执行、诊断是否可定位、repair 是否高效。

两者是正交但互补的：

- 没有可闭环表征，可执行约束就缺少附着对象和引用语义。
- 没有可执行约束，可闭环表征就只停留在“可提交、可 diff”，无法产生真实正确性反馈。

因此，两个实验应共用同一套任务池和评分器，但从不同角度切入。
