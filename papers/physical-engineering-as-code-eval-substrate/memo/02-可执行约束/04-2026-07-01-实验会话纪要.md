# 2026-07-01 实验会话纪要

本 memo 记录 2026-07-01 围绕“可执行约束”实验的会话结论、代码修复、当前数据状态与下一步计划。它是后续继续实验和改写论文小节的交接文件。

## 一、实验语义已重新校准

本轮确认了三类条件的语义边界：

- **NL-only**：只给自然语言任务要求；不提供编译器式反馈；只有一次提交机会。这个条件模拟非专业甲方或需求方的自然语言委托。
- **Docs-only**：给任务要求和规范文档；不提供编译器式反馈；只有一次提交机会。这个条件模拟“文档齐全但没有自动检查”的工程流程。
- **Executable**：给完整可执行约束目录和编译器式反馈；设置最大提交预算；每次提交后评分并返回诊断；所有提交都保留用于后续分析。

关键原则：

- 自然语言约束不应获得编译器式反馈。
- Docs-only 也不应看到可执行约束目录或 repair diagnostic。
- Partial executable 中的约束隐藏或反馈隐藏不是污染，而是实验干预；最终评分仍使用完整约束集合。
- `max_repair` 的语义是“初始提交之后的 repair 次数”，因此总提交预算为 `max_repair + 1`。

## 二、已修复的基础设施问题

本轮对实验 harness 做了多处修复。相关代码尚未提交，位于当前工作区。

### 1. 断点续跑

`sd-hwe-bench batch` 新增 `--resume`。

- 只统计完整 manifest：需要包含 `success`，并且包含 `turn_scores`、`layers` 或 `overall_score`。
- incomplete/stub manifest 不计为完成，会被续跑补齐。
- dry-run 会显示 `remaining` 与 `complete`。

当前续跑命令：

```bash
uv run sd-hwe-bench batch --matrix scripts/batch/constraint-gap-p0.yaml --resume
```

### 2. 真正的提交循环

`run-repair` 修正为“编译器式提交循环”：

- `.sdhwe.done` 不再在失败时立即终止；如果预算未耗尽，会清除 marker 并继续给反馈。
- 非 done marker（`give_up`、`info_gap`、`no_solution`）仍终止。
- 成功只由 scorer success 决定。
- manifest 记录 `max_submissions`、`submissions_used`、每轮 `submission` 与 `remaining_submissions`。

### 3. 诊断噪声过滤

修复了 positive comment 被误当作 failure diagnostic 的问题。

曾出现的错误症状：

- `pass 100%` 但 `omission_density > 0`。
- repair prompt 中出现 `L0 passed` 或 `✓ power-budget found`。

现已过滤：

- 以 `✓` 或 `✅` 开头的正向评论。
- `L0 passed:`。
- `No deliverables required...`。

聚合脚本也修复为：成功样本的 omission density 必须为 0。

### 4. Provider-aware 调度

`batch` 新增 provider-aware 调度能力：

- plan 按 provider round-robin，避免队首被同一 provider 塞满。
- 支持 `provider_max_workers`，可以同时并发 DeepSeek、Kimi、Codex，但分别限制每个 provider 的并发。

示例：

```yaml
max_workers: 3
provider_max_workers:
  deepseek: 1
  kimi: 1
  codex: 1
models:
  deepseek-v4-flash-claude-code: claude:deepseek-v4-flash
  kimi-k2.7-kimi-code: kimi
  gpt-5.5-codex: codex:gpt-5.5
```

### 5. 长预算轨迹采样

聚合脚本新增 `submission_budget_rows`。

设计意图：不分别跑 5、20、50、100 四批，而是跑一次长预算轨迹，然后在同一条 `turn_scores` 上做前缀采样。

采样语义：

- 若第 37 次提交通过，则 budget=5 与 budget=20 记为 fail，budget=50 与 budget=100 记为 pass。
- 成功后的 omission density 记为 0。
- 未成功则取该预算内最后一次提交的 omission density。

这可以减少 provider 随机性、网络波动和重复实验成本。

## 四、当前证据判断

### 1. 短任务证据支持主张

核心 evidence 已经出现：

- `telecom-cross-001`
  - `nl-only`: 0/3
  - `docs-only`: 0/3
  - `executable`: 3/3，全部 2 次提交后通过
  - `partial-executable-l3-muted`: 3/3，全部 2 次提交后通过
  - `executable-coarse-diagnostic`: 0/3，全部 6 次提交耗尽
- `telecom-emergent-001`
  - `nl-only`: 1/3
  - `docs-only`: 1/3
  - `executable`: 2/3，其中出现 4 次、5 次提交后成功的样本
  - `partial-executable-l3-muted`: 2/3
  - `executable-coarse-diagnostic`: 3/3

这些结果支持以下较稳健表述：

> 在约束可执行且反馈足够定位时，Agent 能通过提交循环显著减少自然语言或文档条件下的遗漏，并把 pseudo-correct 方案修复为可评分通过的工程产物。

不应过早表述为“所有任务都接近 100%”。更稳妥的是说：短任务和中等耦合任务上已观察到明显收敛证据，长任务需要更高预算。

### 2. 诊断粒度有可解释影响

`executable-coarse-diagnostic / telecom-cross-001` 三个完整失败样本都用尽 6 次提交，最终仍未修好交付物路径/命名类问题。这支持：

> 诊断不仅要发现错误，还要定位到 Agent 可消费的文件、对象、字段或交付物位置。

coarse feedback 不是无效，但对路径、文件名、交付物位置这类错误可能不足。

### 3. Partial executable 是实验结果，不是泄露

`partial-executable-l3-muted` 中隐藏或不反馈某些约束，正是用于测试约束覆盖率下降后的遗漏模式。

应在论文中明确：

- partial 条件仍使用完整评分。
- mute 只控制 Agent 可见信息或反馈信息。
- hidden/muted violation 是结果变量。

### 4. AIDC 不应简单视为噪声

AIDC 当前表现为 long-horizon stress test，而不只是噪声。

合理解释：

- AIDC 任务 horizon 长、文件多、约束多，可能需要几十到上百次提交。
- 小预算失败不代表主张失败；它说明固定 6 次提交预算不足。
- 若小时级或长预算下 pass 率继续上升，反而更支持“工程合规是迭代编译/搜索过程，而非 one-shot generation”。

论文中建议将 AIDC 单列为 long-horizon 分析，不与短任务在同一主表里混合解释。

## 五、噪声与处理原则

本轮观测到至少两类噪声。

### 1. Provider/runtime 网络错误

出现过：

```text
API Error: The socket connection was closed unexpectedly
```

处理原则：

- 如果只是 incomplete/stub manifest，`--resume` 补跑。
- 如果完整 manifest 中记录为 actor/API 错误，应单独标记为 infra/runtime failure，不与工程约束失败混淆。
- 当前聚合仍可能把 actor_error 样本计入完整失败；正式论文统计前需要决定是否剔除或单列。

### 2. Walltime 长尾

后段 case walltime 明显拉长，但原因不是单纯“后跑就慢”，而是：

- 后段排到了 repair、partial、coarse 等多提交条件。
- AIDC 任务天然长。
- 网络异常产生长尾。

示例观察：

- `telecom-cross-002` 中位 actor 时间约 54s，且各条件 3/3，区分度弱。
- `aidc-60mw-002` 中位 actor 时间约 472s，且多为失败或 incomplete。
- 多提交失败样本可超过 1000s。

- budget exhausted count
