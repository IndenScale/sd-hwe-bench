# 可执行约束：基础设施要求

本 memo 记录为了支撑“可执行约束鸿沟”实验，需要在 SD-HWE-Bench 基础设施中具备的能力。

## 约束目录与覆盖率

正式实验前需要把任务中的约束显式登记为可统计对象，而不是只散落在 task.yaml、docs 和 critic 代码里。

每个任务至少需要能够导出：

- 适用约束列表。
- 约束 id。
- 约束族，例如 schema、reference、electrical、thermal、layout、construction、schedule、deliverable。
- 所属评分层，例如 L1-L5。
- 是否可执行。
- 对应 critic 或 rule。
- 错误定位粒度能力，例如 task-level、object-level、file-level、field-level。

如果某条任务要求目前无法被 critic 检查，应显式标记为 non-executable，不能混入 Full executable 条件的主证据。

`telecom-emergent-001` 的相位要求就是一个警示：任务要求“相位 L2”，但 scorer 没有强制捕获该字段，导致 Agent 把相位写入 `name` 仍然通过。这类任务要么补 critic/schema，要么从可执行约束主实验中剔除。

## Constraint mute 机制

需要支持按不同粒度 mute 约束：

- 按约束 id mute。
- 按约束族 mute。
- 按评分层 mute。
- 按随机比例 mute，并记录随机种子。

mute 应区分两种模式，但两者都不应把约束从最终评分中移除。mute 的目的不是降低评分标准，而是控制约束是否对 Agent 可见，从而分析 Agent 在不可见约束上在哪里丢分。

- **Prompt mute**：该约束不进入任务提示词、规范摘录或显式上下文，但仍参与最终评分。用于模拟约束没有被语言化传达，观察 Agent 是否靠世界知识或 scaffold 模式自然满足它。
- **Feedback mute**：critic 仍检测并计入最终评分，但不把对应 diagnostic 反馈给 Agent。用于模拟约束虽然可执行、但没有进入 repair 反馈，观察 Agent 在缺少定位信息时是否能修复。

实验 artifact 需要记录每轮对 Agent 可见、不可见、进入反馈、未进入反馈的约束集合。最终评分应始终基于完整约束集合，另行报告 Agent 在不可见约束和未反馈约束上的丢分。

## Diagnostic verbosity 机制

repair loop 需要把 critic diagnostic 渲染成不同粒度：

- `none`：只返回 failed。
- `coarse`：返回自然语言摘要，不含精确定位。
- `attributed`：返回约束族、对象 id、失败原因。
- `localized`：返回文件、字段、对象、期望值、实际值、修复建议。

同一个底层 critic 结果应能被不同 renderer 转换，避免不同条件实际看到的错误集合不一致。

## Artifact 与指标

每个 run 的 manifest 应记录：

- `context_mode`。
- `repair_mode`。
- `constraint_coverage_mode`。
- `enabled_constraints`。
- `muted_constraints`。
- `offline_constraints`。
- `diagnostic_verbosity`。
- `repair_rounds_used`。
- 每轮 diagnostic 条数、类型、定位粒度。
- 每轮 pass/fail、score、omission density。

聚合脚本需要输出：

- pass@1 / pass@k。
- repair 后 pass rate。
- pseudo-correctness rate。
- omission density。
- mute 约束上的违规率。
- repair saturation curve。
- diagnostic verbosity 对 repair 成功率和成本的影响。
- 按任务复杂度分层的结果。

## 任务分层元数据

任务 metadata 需要增加或派生以下字段：

- `scale_level`：low、medium、high。
- `coupling_level`：low、medium、high。
- `task_family`：layout、connection、cross-domain、co-design、epc 等。
- `constraint_families`。
- `expected_saturation`：是否低耦合饱和任务。

报告结果时不能只按全部任务平均。需要分别报告低耦合饱和任务、中耦合跨文件任务、高耦合系统级任务。

## Actor 隔离与样本可信度

正式实验必须保留 actor 级隔离：

- actor workspace 迁出仓库。
- macOS 上启用 seatbelt 或等价内核级 repo read deny。
- run artifact 记录 `isolated=true` 与实际 `work_dir`。

如果 Agent 超时、卡住或没有落盘，只要 artifact 完整、timeout 自然触发、隔离成立，该样本应作为失败样本或运行时失败进入统计。不能手动终止后计入 pass/fail。

Kimi 的 `-p` 模式在较耦合任务上可能出现长尾或不退出风险。正式实验需要设置合理 timeout，并把 timeout 作为结果而不是先验地当成 harness bug。只有当日志显示隔离、评分或 artifact 写入异常时，才归类为基础设施错误。

## 进入正式实验前的最低门槛

- 每个入选任务都有显式约束目录。
- Full executable 条件中的任务要求都能被 critic 检查，或明确标记为非可执行并从主统计中排除。
- 支持 constraint mute 与随机种子记录。
- 支持 diagnostic verbosity renderer。
- manifest 能记录约束覆盖、mute 和诊断粒度。
- 聚合脚本能按任务复杂度分层输出结果。
- 至少完成一个 Kimi 或 Codex 小矩阵 smoke，确认 artifact、timeout、隔离和聚合链路可用。
