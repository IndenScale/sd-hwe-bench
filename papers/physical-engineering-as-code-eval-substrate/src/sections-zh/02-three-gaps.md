# 2. 从三重鸿沟到 Evaluation Substrate

本章先分析现有 engineering-AI 路径为什么难以单独形成 evaluation substrate，再从这些失败中归纳出 representation、constraint、knowledge 三类缺口。三重鸿沟并非为推出 EaC 而预设的概念框架；它们来自已有评估路径反复暴露的失效模式。

## 2.1 现有路径为何无法单独构成 Evaluation Substrate

本文所说的 evaluation substrate 至少包含五件事：agent 提交什么；提交如何被 deterministic critic 检查；失败如何定位到对象、字段、关系或约束层；repair loop 如何消费诊断；结果如何被归档、复现和比较。许多现有路径解决了其中一部分问题，但很少同时闭合这五件事。

| 现有路径 | 已解决的问题 | 未闭合的问题 |
|---|---|---|
| Parametric CAD / BIM / OpenSCAD | 局部几何、参数化生成、部分工程对象表达 | 不自动定义统一提交对象、任务增量、跨文件 critic、诊断 repair loop 和归档协议 |
| Natural-language specifications / document QA | 提供规范知识、设计要求和解释性上下文 | 不能稳定判定提交是否满足约束，容易产生 pseudo-correctness |
| MCP / CUA / tool-use agents | 扩展 agent 对工具、GUI、数据库和文档的访问 | 工具接口不等于评估协议；不定义提交对象、critic、归档和 repair 环境 |
| Simulation wrappers | 提供高保真物理或工程反馈 | 昂贵、慢、难复现，且常缺少局部诊断和可组合任务结构 |
| Optimization benchmarks | 可度量目标函数和 Pareto tradeoff | 常假设状态、变量和约束已被预先形式化，不能评估 agent 如何构造工程状态 |
| IC/EDA HWE benchmarks | 在 RTL/EDA 场景中已有代码化表示、testbench、仿真和 datasheet/PDK-like 知识 | 证明三类条件在 EDA-rich domain 中可成立，但不能自动推广到 AIDC、施工、能源、机械装配等系统级工程 |

这个表格并不否定现有路径。相反，它说明 engineering-AI evaluation 已经有许多强组件：CAD/BIM 擅长工程表达，参数化设计擅长生成，MCP/CUA 擅长访问工具，仿真器擅长物理反馈，优化 benchmark 擅长目标权衡，IC/EDA HWE benchmark 展示了硬件 agent 评估可以 work。真正缺失的是把这些组件组织成统一的提交-检查-诊断-修复-归档闭环。

从这些失效模式可以归纳出三类缺口。表征缺口的核心在于：现有可计算表示并不自动成为评估提交对象。约束缺口来自规则的执行落点：大量规范没有统一进入可执行、可定位、可归因的 critic。知识缺口则出现在资料到模型的转换环节：datasheet、报价、气候、电价和施工经验若不能变成 agent 可消费、可更新、可优化的模型、曲线、库和目标函数，就难以改变设计上限。

## 2.2 表征鸿沟：从可计算到可闭环

表征鸿沟不能简化为“工程缺少可计算表示”。参数化设计、CAD family、BIM 族、OpenSCAD、Grasshopper、Dynamo 和 Excel 配置表都已经证明，工程工件可以被变量驱动、批量生成、版本化和复用。真正的问题是：**什么样的可计算表征才配称为 physical engineering evaluation 的 executable substrate？**

本文的回答是：可闭环表征。可计算只是必要条件，可闭环才是充分条件。一个可闭环工程表征至少需要支持五类闭环：

- **生成-验证闭环**：表征不仅能生成工件，还能被映射到可执行 critic。
- **本专业-跨专业闭环**：电气、热、结构、布局、施工和运营需要在同一语义层被联合检查，不能只停留在各专业内部自洽。
- **设计-建造闭环**：设计状态能接受吊装、施工通道、资源窗口、EPC 排程等建造反馈。
- **当前项目-未来项目闭环**：工程知识可以跨项目迁移，不被绑定在某个作者的脚本习惯里。
- **个人知识-组织知识闭环**：隐式约定被显式化为对象、关系、接口、约束和依赖。

因此，EaC 的定位不是替代 CAD/BIM/参数化设计，而是提供 agent 提交与评估所需的语义层。CAD 可以表达几何，BIM 可以组织工程文档，OpenSCAD 可以表达 code-like geometry，MCP 可以暴露工具接口，CUA 可以操作 GUI；但 evaluation substrate 还必须规定：工程状态如何被提交，引用关系如何被检查，错误如何定位，repair 如何发生。

更具体地说，参数化设计和 BIM 解决了局部表达和生成问题，却通常不定义 agent evaluation 所需的提交边界、diff 语义、任务增量、critic 接口、repair 反馈和归档协议。没有这些协议，一个强大的工程表示仍可能停留在设计工具层，无法成为 evaluation substrate。

## 2.3 约束鸿沟：从规范文本到可执行正确性

约束鸿沟指工程正确性大量存在于自然语言规范、人工经验和隐式审查流程中，而没有变成可执行、可定位、可归因的 critic。自然语言规范可以告诉人“应当满足什么”，但不能稳定回答三个评估问题：本次任务涉及哪些约束，提交物是否真实满足这些约束，错误发生在哪个对象、字段、关系或约束层。

这里不能简单照搬软件工程中的测试套件逻辑。软件工程的测试通常与代码仓、构建系统和 CI 流程共同存在；系统级工程的约束却分散在 PDF 规范、Excel 表、供应商手册、CAD/BIM 模型、商业 GUI、仿真软件和人工审查流程中，缺少统一的可执行入口。工程约束的困难不只在规则数量，更在四个结构性问题：

- **多源异构**：约束来自规范、供应商手册、表格、图纸、经验审查和商业工具。
- **跨对象关系**：约束经常跨越设备、接口、连接、空间、施工和运营，无法被单文件单字段规则覆盖。
- **多层执行**：有些约束是 schema，有些是静态规则，有些需要动态仿真、几何检查或排程。
- **诊断困难**：即使知道提交失败，也不一定能定位到对象、字段、连接、活动或约束层。

这导致物理工程评估中最常见的失败形态并非明显胡说，而是 pseudo-correctness：方案在语言上完整、专业、结构合理，却在可执行检查下失败。例如：

- 引用不存在的设备、接口或端口。
- 连接器、线缆、光模块或接口类型不兼容。
- 功率预算、结构载荷、接地、防雷、消防或维护间距被遗漏。
- 布局满足视觉直觉，却违反吊装半径、施工路径或 VDC 工作面前置条件。
- 调度计划看似完整，却违反资源容量、天气窗口、交期或 SLA。

EaC 的约束层将正确性从“模型是否记得规则”转化为“系统是否能判定并反馈规则”。这并不意味着模型不重要；它意味着评估对象从单次自然语言输出变成 agent 与 critic 共同构成的闭环系统。同样的错误，如果只反馈“设计不满足电气要求”，repair 很可能退化为猜测；如果反馈到文件、对象、字段、期望值、实际值和约束族，agent 才能进行局部修复。

## 2.4 知识鸿沟：从背景资料到可优化知识

知识鸿沟比前两者更深。即使工程状态可提交、约束可执行，agent 仍可能被保守、静态或滞后的知识边界限制。这里的“知识”并不泛指更多上下文、更多检索文档或更多领域语料；本文关注的是能进入设计变量、模型参数、目标函数、约束接口或搜索空间，并改变 Pareto frontier 的 **可优化知识**。

可执行约束定义可行域：接口是否兼容、容量是否超限、间距是否满足、资源是否冲突。可优化知识塑造目标景观：不同设备曲线、气候、电价、水价、报价、交期、故障率和调度策略如何共同影响 TCO、PUE、SLA、工期和风险。简言之，可执行约束证明方案没有错；可优化知识解释方案为什么更好。

半导体 PDK 是一个重要对照。PDK 的价值远超过资料汇编：它把工艺能力转化为 EDA 可消费的规则、模型、参数和库，使设计工具能够在真实工艺边界内搜索。许多系统级工程领域缺少 PDK-like knowledge infrastructure：设备厂商知识没有变成候选设备库和效率曲线，地点知识没有变成气候与电价时序，施工经验没有变成资源容量、天气窗口和风险模型。

PDK-like infrastructure 没有在所有工程领域自然出现，原因在于形成条件不同。半导体制造高度标准化、重复度高、自动化价值极大、工具链边界清晰，并且有强烈的 verification 和 optimization pressure。许多系统级工程项目更像一次性或低重复度项目，参与方分散，知识由业主、EPC、供应商、施工方和运营方分别持有，商业工具封闭，数据格式异构，目标函数还会随地点、气候、电价、供应链和施工窗口变化。因此，普通 datasheet、报价单、气候数据和电价表不会自然沉淀成统一的、agent 可消费的模型库。

AI Data Center Design 适合作为知识鸿沟的 reference domain。设备选型、冷却布局、储能容量、光伏配置、当地气候、电价结构、负载曲线和调度策略之间存在强耦合。局部满足规则不等于全局最优；只有当这些知识进入 PEaC substrate 的模型、目标和搜索空间，物理工程评估才可能从合规走向前沿。
