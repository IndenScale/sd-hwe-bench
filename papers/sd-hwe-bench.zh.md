# SD-HWE-Bench：面向软件定义硬件工程的声明式设计能力评测基准

**作者**：SD-HWE-Bench 贡献者  
**状态**：初稿  
**日期**：2026-06-14

---

## 摘要

当前工程领域的 AI 评测基准（Benchmark）主要测量模型对已有工程文档的阅读和判断能力，但无法回答一个更根本的问题：**AI 能否像工程师一样，将模糊需求转化为正确、合规、可交付的设计声明？** 本文提出 SD-HWE-Bench（Software-Defined Hardware Engineering Benchmark），一个面向声明式工程设计能力的开放式评测基准。SD-HWE-Bench 要求模型/智能体从自然语言需求出发，生成结构化的工程设计声明（YAML），并通过 L0-L4 多层规则引擎自动校验。本文首先定义 SD-HWE（软件定义硬件工程）的核心概念——PDL（部件定义语言）、PLL（部件布局语言）、PML（部件配合语言）三层子语言及其之间的正交关系；进而将工程领域的验证体系映射到软件工程中 linter、单元测试和集成测试的等价概念；最后分析现有 AI4E benchmark 的范式局限，阐述 SD-HWE-Bench 的设计原理、评分体系和与现有工作的关系。

---

## 1. 引言

大模型在代码生成领域取得了令人瞩目的进展。从 SWE-Bench [1] 到 Code RLVR（基于可验证奖励的强化学习）[2]，一条清晰的路径已被证实：**当模型输出可以被自动、快速、确定性地验证时，模型在该领域的能力就会发生系统性跃迁。**

然而，这条路径在实体工程领域尚未走通。这不是因为工程领域缺少可验证的约束——恰恰相反，工程领域的隐含约束比代码更多、更"硬"（物理定律不讲情面）。问题在于，这些约束没有被暴露为模型可以消费的形式。

当前工业软件（CAD、BIM、PLM）的默认假设是：**使用者是视力良好、受过专业训练的人类工程师** [3]。设计数据被埋藏在 GUI 菜单、专有二进制格式和云端数据库里。模型无法直接读取设计意图，无法获得秒级验证反馈，因此无法在工程领域建立有效的 RLVR 飞轮。

这导致了一个结构性错配：一边是试图理解物理世界的大模型，一边是为人类眼手协调而优化的 GUI 工业软件。让前者通过模拟鼠标点击来操作后者——所谓的"VLM 点 CAD 菜单"路线——成本高、错误多、难以规模化。

本文主张一条不同的路径：**让工程设计本身变得对机器友好**。具体而言：

1. 将工程设计表达为文本原生的声明式语言（PDL / PLL / PML），使 Agent 可直接读写、diff 和版本控制；
2. 将工程约束表达为多层规则引擎（L0-L4），提供秒级、确定性、可解释的验证反馈；
3. 在此外基础设施之上，构建 SD-HWE-Bench——一个标准化、可自动评分、可复现的能力竞技场。

本文的贡献如下：

- **定义 SD-HWE** 的核心概念体系，阐明 PDL、PLL、PML 三层子语言的设计原理及其与 SysML v2 的认识论差异；
- **建立工程验证与软件验证的映射**，明确 linter、单元测试、集成测试在声明式工程设计中的对应物；
- **分析现有 AI4E benchmark 的范式局限**，解释为何 AEC-Bench 等"真实数据提取"模式不适用于 SD-HWE；
- **提出 SD-HWE-Bench 的设计**，包括任务分类、分层评分体系和与 RLVR 训练的关系。

## 2. SD-HWE：软件定义硬件工程

### 2.1 核心主张

**软件定义硬件工程（Software-Defined Hardware Engineering, SD-HWE）** 的核心主张是：

> 工程设计应当像软件工程一样，以结构化文本为唯一真相源，以自动规则引擎为质量门，以版本控制为协作基座。

这并不意味着否定 CAD 和仿真工具的价值。它们仍然是可视化和深度分析不可或缺的下游工具。SD-HWE 主张的是**入口层的范式转移**：设计意图的表达和验证应当发生在文本层，而不是 GUI 层。

### 2.2 与 SysML v2 的关键差异

SD-HWE 常被与 SysML v2 [4] 进行比较。两者都在解决"如何用形式化语言描述复杂工程系统"，但心智模型有根本差异：

| 维度 | SysML v2 | SD-HWE (piki) |
| ------------ | --------------------------------------- | ------------------------------------------------ |
| 真相源形态 | 模型（model repository） | 文本文件（YAML） |
| 版本控制对象 | 模型版本 | Git 行级历史 |
| 核心操作单元 | Model element（part, port, connection） | 文件（Instance, Layout, Mate, Connection） |
| 空间与身份 | 混合在 part/occurrence 中 | Instance 与 Layout 文件级分离 |
| 部件间关系 | connection/interaction（二元） | Mate（物理拟合）+ Connection（信号/物质流），二元分离 |
| 验证方式 | 模型检查（model checking） | 规则引擎（L0-L6），分层、秒级、可解释 |
| 目标使用者 | 人类系统工程师（GUI 建模） | AI Agent + 人类工程师（文本读写） |

关键差异在于三个设计决策：

**（1）文件作为真相源，而非模型。** SysML v2 的真相源是一个模型仓库——一个封闭的、版本化管理的数据结构。piki 的真相源是分布在文件系统中的 YAML 文件——开放、可被任何文本编辑器读写、可被任何 `grep`/`diff` 操作、可被任何 LLM tokenize。这不是工程上的妥协，而是刻意的 Agent 友好设计：Agent 不需要学习模型仓库的 API，它只需要读写文本文件。

**（2）空间描述与身份声明的分离。** SysML v2 中，一个 part 的空间位置（occurrence）和身份定义（definition）在语言层面是耦合的。piki 中，Instance（声明"这台服务器存在"）和 Layout（声明"这台服务器在 RACK-A01 的 U10-U11"）是两种不同文件类型。这使 piki 天然支持"同一设备在不同方案中位于不同位置"的 diff，也使得布局规则（PLL）可以独立于实例声明（PDL）进行演进。

**（3）Mate 与 Connection 的二元分离。** SysML v2 用 connection 统一表达部件间的所有关系。piki 将"机械上的配合"（Mate：rack-mount、power-iec）与"信号/物质上的流动"（Connection：光纤、网线）分离为两种独立的子语言。配合决定"什么东西能装在一起"，连接决定"装好后什么东西流向什么东西"。前者是物理拓扑约束，后者是功能拓扑约束——在设计流程中，前者通常先于后者确定。

这三个设计决策不是随意为之——它们反映了 SD-HWE 对"工程知识如何结构化"的认识论立场。

### 2.3 工程知识成熟度的四个阶段

SD-HWE 将工程知识按成熟度分为四个阶段 [5]：

| 阶段 | 特征 | 验证方式 | SD-HWE 的覆盖 |
| ------ | ------------------ | ------------------------ | ------------- |
| 第一阶段 | 口传心授 | 无 | 不覆盖 |
| 第二阶段 | 语言化 | 人类共识判断 | 不覆盖 |
| 第三阶段 | 规则化 | 确定性规则引擎（L0-L4） | ✅ 全覆盖 |
| 第四阶段 | 仿真可解 | 数值求解器（L5） | ⚠ 未来扩展 |

SD-HWE-Bench 的当前覆盖边界精确对应第三阶段——**规则化可自动验证的工程知识**。这一定位的设计意图是：如果模型连确定性规则都无法满足，那更复杂的模糊约束和仿真验证就没有讨论的基础。

## 3. PDL、PLL、PML：三层声明式子语言

### 3.1 设计原则

piki 的声明式语言体系遵循三条设计原则 [6]：

**（1）Text-Native（文本原生）。** 所有设计声明都是纯文本 YAML 文件。Agent 可以直接 tokenize、生成、diff 和版本控制。没有二进制格式、没有专有数据库、没有 GUI 依赖。

**（2）Agent-Oriented（面向智能体）。** 语言设计优先考虑机器的可读写性和可验证性，其次才是人类的手工书写便捷性。这意味着 Schema 严格、字段语义明确、错误信息精确到行号。

**（3）Orthogonal Concerns（正交关注点）。** 设计的三个维度——"有什么"（PDL）、"放哪里"（PLL）、"怎么配合"（PML）——在语言层面严格正交。修改一个 Instance 的属性不影响 Layout；修改一个 Layout 不影响 Mate 定义。

### 3.2 PDL：部件定义语言（Part Definition Language）

PDL 回答"有什么部件"。它包含两个核心概念：

- **Family**：设备家族（如 ServerFamily、SwitchFamily、PduFamily），定义一类设备的共享属性和接口清单。
- **Model**：具体型号（如 R740、S5735），继承 Family 的属性并赋予具体数值。
- **Instance**：设计中的具体实例（如 SRV-01），从 Model 实例化，拥有身份标识和可选的位置引用。

```yaml
# models/server.yaml
family: ServerFamily
models:
  - name: R740
    height_u: 2
    tdp_w: 750
    depth_mm: 715
    interfaces:
      - port: eth0
        type: SFP28
        speed_gbps: 25
```text

```yaml
# instances/srv-01.yaml
model: R740
name: SRV-01
```text

关键设计：Instance 文件**不包含布局信息**。设备的位置在 Layout 中单独声明。这种分离意味着同一设备可以在不同设计方案中占据不同位置——这对于工程变更管理和多方案对比至关重要。

### 3.3 PLL：部件布局语言（Part Layout Language）

PLL 回答"部件放在哪里"。piki 中的空间是层次化的：

```text
机房（Room）→ 通道（Aisle）→ 机柜列（CabinetRow）→ 机柜（Rack）→ U 位（Slot）
```text

每个 Layout 声明一个 Instance 在物理空间中的位置：

```yaml
# layouts/srv-01-layout.yaml
instance: SRV-01
location:
  rack: RACK-A01
  start_u: 10
```text

对于电信机柜部署，空间是一维离散的（U 位）。对于更复杂的场景（如暖通管道），PLL 需要扩展为连续 3D 坐标——这是 SD-HWE 当前的已知边界（见 §5.3）。

### 3.4 PML：部件配合语言（Part Mating Language）

PML 回答"部件之间怎么配合"。它包含两个正交的子概念：

**Mate（配合）**：物理上的装配关系。例如 "SRV-01 通过 rack-mount 方式装配到 RACK-A01 上"、"SRV-01 通过 power-iec 方式接入 PDU-A 的 C13 接口"。Mate 是物理拓扑约束。

```yaml
# mates/srv-01-rack-mate.yaml
type: rack-mount
source: SRV-01
target: RACK-A01
```text

**Connection（连接）**：信号或物质的流动关系。例如 "SRV-01 的 eth0 端口通过 SFP28 光纤连接到 SW-01 的 Gi1/0/1 端口"。Connection 是功能拓扑约束。

```yaml
# connections/srv-01-to-sw-01.yaml
source: {instance: SRV-01, port: eth0}
target: {instance: SW-01, port: Gi1/0/1}
media: fiber-single-mode
```text

Mate 与 Connection 的分离对应工程设计中的两个独立决策阶段：(1) 机械层面，"能装上吗？"；(2) 功能层面，"装好后能通信吗？"。在 GUI 工具中，这两个维度常常混在一起；在 piki 中，它们被强制分离，使 Agent 可以分步推理。

### 3.5 正交性的价值

PDL、PLL、PML 的正交设计有三个工程价值：

1. **增量验证**：Agent 可以先生成 Instance（只通过 L1-L2），再生成 Layout（通过 L3-L4），每步都获得独立反馈。
2. **并行编辑**：多个 Agent 可以同时修改 Instance 和 Layout 而不会产生文件级冲突。
3. **语义清晰的 diff**：`git diff` 的输出直接对应"改了设备"还是"挪了位置"——这对人类评审和 RLVR 奖励归因都有价值。

## 4. 工程领域的 Linter、单元测试与集成测试

软件工程师对 linter、单元测试、集成测试的概念非常熟悉。SD-HWE 将这一验证体系完整地映射到了工程领域。

### 4.1 L0-L2：工程 Linter（静态分析）

软件工程中的 linter 检查代码的语法、风格和基本正确性。对应到 SD-HWE：

| 软件工程 | SD-HWE | 检查内容 | 执行时机 |
| ----------------------------- | ------------------------ | ------------------------------------------------------- | -------------- |
| Parser（语法错误） | **L0**（文件格式合法性） | YAML 是否合法 | 加载时，< 1ms |
| Type checker（类型错误） | **L1**（Schema 校验） | height_u 是否在 1-48 之间？tdp_w 是否为正数？ | 加载时，< 10ms |
| Unused import / undefined ref | **L2**（引用完整性） | rack_id 引用的机柜是否存在？interface_type 是否已注册？ | 加载时，< 10ms |

这些检查的共性是：**无需理解业务语义，纯粹从结构和类型层面即可判定对错。** 它们相当于设计文件的"静态分析"——你没有权力提交语法不合法的 YAML，就像你不能提交有类型错误的代码。

### 4.2 L3：工程单元测试与集成测试

软件工程中的单元测试和集成测试检查业务逻辑的正确性。在 SD-HWE 中，这对应 L3 层——**跨记录业务规则**：

| 软件工程 | SD-HWE | 示例 |
| ---------------------- | -------------- | ------------------------------------------------------- |
| 单元测试（单函数逻辑） | 单设备约束检查 | "SRV-01 的功耗不超过其额定 TDP" |
| 集成测试（多模块交互） | 跨设备约束检查 | "PDU-A 上所有设备的总功耗不超过 3000W × 80%" |
| 契约测试（接口兼容） | 接口兼容性检查 | "SRV-01/eth0（SFP28）能否连接 SW-01/Gi1/0/1（SFP28）？" |

L3 规则用 Python 函数编写，像 pytest 一样运行：

```python
@rule("TELECOM-POWER-001", "PDU 功率预算检查")
def check_pdu_budget(ctx: Context):
    for pdu in ctx.query("instances", family="PduFamily"):
        devices = ctx.query("instances", pdu_id=pdu.id)
        load = sum(d.resolved.tdp_w for d in devices)
        ratio = load / pdu.resolved.capacity_w
        assert ratio <= 0.8, \
            f"{pdu.id} 负载率 {ratio:.1%}，超过 80% 阈值"
```text

执行成本：10-100ms 每个规则。反馈速度足以支撑 RLVR 的快速迭代循环。

### 4.3 L4：工程端到端测试（物理层）

L4 检查几何和空间上的正确性：

| 软件工程 | SD-HWE | 示例 |
| ---------- | -------- | ------------------------------------------------------ |
| 端到端测试 | 碰撞检测 | "SRV-01 和 SW-01 是否在同一 U 位发生空间重叠？" |
| 部署验证 | 布局约束 | "SRV-01 的深度 715mm 是否超出 RACK-A01 的深度 800mm？" |

L4 通过 AABB 包围盒碰撞检测实现，成本 100ms-1s。比 FEA 快 3-4 个数量级，适合作为 RLVR 的"快速拒绝"层。

### 4.4 L5-L6：仿真验证与 AI 评估

L5（物理仿真）和 L6（AI 综合评估）不在 SD-HWE-Bench 的默认评分中，但作为可选扩展：

| 软件工程 | SD-HWE | 成本 | 启用方式 |
| ------------------- | ----------------------------- | ------- | ------------------- |
| 性能基准测试 | **L5**：CFD、热仿真、电磁仿真 | 1s-1min | `piki check --deep` |
| 代码评审 / 架构评审 | **L6**：AI 综合评估 | 1-10s | `piki check --ai` |

这两层成本高，不适合作为 RLVR 训练的核心验证器，但适合作为高质量样本的最终奖励信号。

### 4.5 映射总结

```text
软件工程                      工程领域 (SD-HWE)
══════════════════════════    ════════════════════════════
Linter（语法 + 类型检查）  →  L0-L2：格式 / Schema / 引用完整性
   ↓                              ↓
单元测试 + 集成测试          →  L3：跨记录业务规则
   ↓                              ↓
端到端测试                   →  L4：几何碰撞检测
   ↓                              ↓
性能基准                     →  L5：物理仿真（可选）
   ↓                              ↓
代码评审                     →  L6：AI 评估（可选）
```text

这个映射不仅是一个类比——它是 SD-HWE 验证体系的设计蓝图。理解了软件工程中 linter、单测、集成测试的分层逻辑，就理解了 piki 的 L0-L6 检查分层。

## 5. 实验设置

SD-HWE-Bench 的实验设置需要同时回答三个问题：**测什么场景**、**为什么是这些场景**、**任务怎么拆解**。本章依次展开。

### 5.1 任务域选择：为什么从电信机柜开始

SD-HWE-Bench 的初始任务集选择电信机柜部署（rack-level deployment），不是随机的，而是因为这个场景恰好满足了 SD-HWE 范式当前运行所需的全部条件。这不是 SD-HWE 范式的普遍证明，而是一个精心选择的"最少依赖"验证场景。

#### 5.1.1 干净切片论证

| 维度 | 电信机柜部署 | 为什么干净 |
| ------ | ------------- | ----------- |
| **物理空间** | 一维离散（U 位 1-42） | 没有连续坐标、曲面、自由形体——碰撞检测简化为区间重叠 |
| **约束类型** | 不等式为主（功率 ≤ 80%、深度 ≤ 机柜深度） | 可表达为简单的比较运算符，不需要求解偏微分方程 |
| **接口体系** | 已穷举（SFP28、SFP+、QSFP28、IEC-C14、LC） | 电信接口类型数量有限，兼容性矩阵可完全枚举 |
| **设备树** | 浅层（设备→机柜→机房，2-3 层） | 没有多层装配、子组件、BOM 爆炸 |
| **生命周期** | 单阶段（设计部署方案） | 不考虑安装顺序、运维变更、退役拆除 |
| **多专业耦合** | 单一专业（通信）为主 | 电力只需粗粒度的功率检查，暖通和结构完全不在覆盖范围 |

电信机柜部署是一个**被高度简化的工程场景**——它保留了工程设计的核心要素（物理约束、接口兼容、多目标权衡），但把所有维度都压缩到了最"干净"的形式。这使得它可以作为 SD-HWE 的"验证性实验"（proof-of-concept），但不能作为范式通用性的"有效性证明"。

#### 5.1.2 三层空白的现实约束

选择电信机柜作为起点还有一个更根本的原因：**SD-HWE 面临三层结构性空白**，这严格限制了 benchmark 的初始规模。

| 供应链层级 | SWE-Bench | AEC-Bench | SD-HWE-Bench |
| ----------- | ----------- | ----------- | -------------- |
| **上游**（需求） | GitHub issue（百万级） | 施工图+规范文档（行业标准） | **零**——需求文本需人工编写 |
| **中游**（实现） | 开源仓库（百万级）+ git history | CAD/BIM 图纸 + 审图流程 | **零**——声明式设计仓库不存在 |
| **下游**（验证） | 编译器 + pytest（数十年） | 人类专家审图（行业标准） | **萌芽**——piki 0.1.0；传统仿真器不可用 |

这不是学术上常见的"数据稀缺"——后者可以通过标注来解决。这是**生态位的缺失**：SD-HWE 试图评测的能力——声明式工程设计——在当前工业体系中根本不存在。SWE-Bench 评测的"改 bug"这个能力在软件工程中已存在几十年，只是以前没人拿它做 AI 评测。SD-HWE-Bench 评测的能力即使在人类工程师中也几乎不存在。

后果直接体现在 benchmark 规模上：每个 task 的 scaffold、solution 和评分规则都必须人工构造，无法从真实工业数据中采样。SD-HWE-Bench 的 MVP 目标为 50-100 个任务——这约是 SWE-Bench（2294）的 2-4%，但代表了当前范式下从零构建 benchmark 的真实上限。

正因为三层空白，SD-HWE-Bench 的定位不是"便利型 benchmark"——从现成数据中提取任务的便利——而是**"定义型 benchmark"**。软件工程花了四十年从编译器走到 SWE-Bench。SD-HWE 的机会在于跳过了这四十年：piki 的规则引擎从第一天起就是 **verifier-first** 设计的。类似于 ImageNet 是人工标注、GLUE 是人工构建，SD-HWE-Bench 的价值不在于"从真实数据中提取"，而在于**定义了一个领域的问题形式和评判标准，从而激活整个研究社区**。

### 5.2 任务细粒度设计

#### 5.2.1 任务分类与能力映射

电信机柜部署被拆解为六类子任务，每类针对设计流程中的一种原子能力：

| 类型 | 说明 | 考察能力 | 难度 | 规划样本量 |
| ------ | ------ | --------- | ------ | ----------- |
| **实例声明** | 选择 Family/Model，声明 Instance 属性 | 型号选择、参数赋值、属性推导 | 简单 | 10-15 |
| **布局设计** | 将 Instance 分配到机柜 U 位 | 离散空间分配、区间不重叠约束 | 简单 | 10-15 |
| **连接设计** | 创建端口-端口连接 + 定义端口配合 | 接口兼容性判断、拓扑正确性 | 中等 | 10-15 |
| **配合设计** | 定义物理耦合约束（rack-mount、power-iec） | 机械约束匹配、耦合类型选择 | 中等 | 10-15 |
| **综合设计** | 端到端：实例 + 布局 + 连接 + 配合 | 全链路协调、多约束同时满足 | 困难 | 10-15 |
| **增量修改** | 在已有设计基础上增/删/改 | 变更影响分析、约束保持 | 中等 | 5-10 |

总规模：55-85 个任务（MVP）。

#### 5.2.2 难度梯度设计

每类任务内部设有难度梯度：

- **简单**：需求文本直接给出型号名、U 位号、端口名，Agent 只需填入正确的 YAML 结构；
- **中等**：需求文本给出一组约束条件，Agent 需要推理出正确的选择（例如"选择功耗最低且深度不超过 700mm 的 1U 服务器"）；
- **困难**：需求文本给出高层设计目标，Agent 需要自行分解为子任务并协调多个文件（例如"设计一个 42U 机柜，容纳 8 台服务器和 2 台交换机，功率不超过 80%，所有服务器双上联"）。

难度梯度的存在使得 benchmark 不仅能给出"通过/未通过"的二值判定，还能诊断出 Agent 能力的天花板在哪里——是卡在简单的参数填写，还是卡在复杂的多约束协调。

#### 5.2.3 任务结构

每个任务实例的自包含结构如下：

```text
tasks/telecom/rack-deploy-XXX/
├── task.yaml          # 任务元数据：类型、难度、需求文本、交付物清单
├── scaffold/          # 预置上下文：型号库、已有实例（Agent 可读）
│   ├── piki.toml
│   ├── models/
│   └── instances/
├── solution/          # 参考答案（对 Agent 隐藏）
│   ├── instances/
│   ├── layouts/
│   └── mates/
└── expected/          # 期望交付物：用于面向任务的评估
    ├── bom.csv
    └── port-map.csv
```text

其中 `scaffold/` 提供模型访问型号库和已有实例的锚点，模拟真实工程中的"已有设计上下文"；`solution/` 存储人类编写的参考设计声明；`expected/` 存储期望的工程交付物，用于面向任务的评估（见 §6）。

### 5.3 场景覆盖与范式边界

#### 5.3.1 为什么没做更多场景

SD-HWE-Bench 当前只覆盖电信机柜部署，不是由于资源投入有限，而是由于声明式建模本身的表达能力边界。以下分析不是"暂时没有数据"的工程问题，而是 SD-HWE 范式当前的**结构性缺口**。

**一、连续空间与自由形体。** 电信场景中，机柜是离散的槽位（U1 到 U42），设备高度是整数倍 U。布局的本质是在一维网格上分配区间。但真实工程场景——暖通管道在三维连续空间中走向、厂房设备有任意朝向和实数坐标——要求 PLL 支持连续 3D 坐标、自由朝向和非整数约束。piki 的 PLL 当前只支持离散的 rack-id + U 位定位，连续空间语义在语言层面就是缺失的。

**二、非线性物理约束与仿真依赖。** 电信场景的功率是标量加法（sum(tdp) ≤ capacity × 0.8），所有约束都是代数可判定的。但热设计涉及 CFD 求解、结构分析涉及 FEA 模态分析、电磁兼容涉及电磁场求解——这些需要数值求解器的输出作为验证信号。而 piki 的 L5 层目前空白，且从声明式设计到传统仿真器（Ansys、Cadence）的输入转换路径标准并不存在。更根本的是：CFD 仿真可能需要数十分钟到数小时，RLVR 的秒级反馈循环在这里会断裂。

**三、时序与动态行为。** PDL + PLL + PML 三层子语言都是**静态声明**——描述"最终态是什么"，不描述"从初始态到最终态的过渡过程"。施工顺序（先装哪个、后装哪个）、运维变更（扩容、替换、退役）、控制系统（暖通自控状态机）都需要时序和状态转换的表达能力，piki 当前完全不具备。

**四、多专业耦合与语义鸿沟。** 电信场景中跨专业交互只有一项——功耗汇总——属于弱耦合。但数据中心建设涉及通信、电力、暖通、结构、消防、安防六个专业深度耦合。跨专业的因果链（工艺温度 → 管道材质 → 焊接工艺 → 探伤要求）需要语义互操作协议，而 piki 的 telecom 和 datacenter 插件独立开发，没有机制表达跨专业规则引用。

**五、模糊需求与非确定解空间。** 电信场景的需求明确——"8 台 2U 服务器，功率不超过 80%"。但概念设计阶段的需求本身模糊（"大概容纳 500 个机柜，PUE 尽量低，预算未定"），且正确方案不唯一。SD-HWE 当前假设"需求 → 设计声明"是单向映射，无法处理需求与方案的共同演化。

#### 5.3.2 SD-HWE 的诚实边界

综上，SD-HWE 当前的能力边界如下：

| SD-HWE 擅长的 | SD-HWE 当前不能的 |
| -------------- | ----------------- |
| 离散约束（U 位、槽位） | 连续空间与自由形体 |
| 代数可判定的规则（比较、求和） | 需要数值求解的物理约束 |
| 静态声明（最终态是什么） | 时序过程（怎么过渡） |
| 单一专业为主的场景 | 深度多专业耦合 |
| 需求明确的详细设计 | 需求模糊的概念设计 |
| 秒级可验证的设计检查 | 需要小时级仿真反馈的约束 |

SD-HWE-Bench 的初始数据集选择电信机柜部署，恰恰因为这个场景完全落在"擅长"的那一列。这不是作弊——每个 benchmark 都需要从可操作的起点开始。SWE-Bench 的初始任务也是从 Python 项目的 bug fix 开始，不是从分布式系统的一致性 bug 开始。但必须在论文中坦诚承认：当前 benchmark 的覆盖边界与 SD-HWE 愿景的完整版图之间存在巨大差距。这个差距不是"更多数据"能填补的——它需要在语言设计（连续空间语义）、工具链（仿真器接入）、方法论（时序表达）上都做出基础性的扩展。

#### 5.3.3 扩展路线图

SD-HWE-Bench 的场景扩展将遵循"从干净到脏"的渐进策略：

| 阶段 | 领域 | 新增维度 | 关键挑战 |
| ------ | ------ | --------- | --------- |
| **v1** | 电信机柜 | 基准线 | 验证声明式设计 + 规则引擎的可行性 |
| **v2** | 数据中心 | 多专业弱耦合（通信 + 电力 + 暖通功率） | 跨专业规则引用、多插件协调 |
| **v3** | 机械（键盘） | 3D 离散空间（PCB + 键轴 + 外壳） | 三维装配约束、零部件配合链 |
| **v4** | 暖通/流体 | 连续空间 + 管道网络拓扑 | 连续坐标、非线性约束、仿真接入 |
| **v5** | 建筑施工 | 深度多专业耦合 + 时序 | 多专业语义互操作、施工顺序建模 |

每一阶段不是简单"加数据"，而是**验证 piki 语言和规则引擎在新增维度上的表达能力和验证覆盖率**。如果某个阶段的语言扩展证明不可行，也就发现了 SD-HWE 范式的真实边界。

## 6. 评估方案

SD-HWE-Bench 的评估分为两个维度：**面向过程的规则验证**（Agent 的 YAML 是否通过 piki check）和**面向任务的交付物评估**（Agent 产出的工程交付物是否可用）。前者是自动的、秒级的、确定性的；后者需要人工评审或 LLM-as-judge。两者互补。

### 6.1 面向过程的规则验证（L0-L4）

规则验证沿 piki 的 L0-L4 检查层级逐层评分：

| 层级 | 权重 | 检查内容 | 软件工程对应 | 反馈延迟 |
| ------ | ------ | --------- | ------------- | --------- |
| L0 | 0%（门槛） | 文件格式合法性（YAML parse） | Parser | < 1ms |
| L1 | 10% | Schema 校验（字段类型、值域） | Type checker | < 10ms |
| L2 | 15% | 引用完整性（rack_id 存在？interface_type 已注册？） | Import resolution | < 10ms |
| L3 | 40% | 业务规则（功率预算、U 位冲突、接口兼容） | 单元测试 + 集成测试 | 10-100ms/规则 |
| L4 | 20% | 几何检查（AABB 碰撞检测、深度越界） | 端到端测试 | 100ms-1s |
| 交付物 | 15% | 生成器输出有效性（BOM 格式、端口图完整性） | Build artifacts | 1-10s |

**Pass@k**：k 次尝试中最优一次通过全部 L0-L4 检查的任务比例。

**部分得分**：各层独立评分。一个 Agent 即使未通过 L3，仍可通过 L0-L2 获得 25% 的基础分——这允许精细化的能力诊断。

**与 RLVR 训练的关系**：piki check 的分层设计使 SD-HWE-Bench 不仅是被动评测工具，更是 RLVR 训练场。秒级反馈、确定性奖励、可解释的失败（精确到文件和行号）、分层奖励信号——这四项特性共同构成了工程 Agent 的 RLVR 飞轮。

### 6.2 面向任务的交付物评估

规则检查通过不代表工程可交付。一个设计可能 piki check 全绿，但 BOM 漏掉了光纤跳线——因为规则引擎没有检查光纤跳线的规则。因此 SD-HWE-Bench 引入面向任务的交付物评估。

#### 6.2.1 四种执行路径

同一业务需求，可通过四种路径完成，面向任务的评估对四种路径统一适用：

| 路径 | 交互模式 | 工具 | 在 benchmark 中的角色 |
| ------ | --------- | ------ | --------------------- |
| **人类基线** | 熟练工程师直接完成任务 | 任何其熟悉的工具 | 上界参考——当前最佳工程师能做什么 |
| **GUI 点击** | 人类用鼠标操作传统工业软件 | AutoCAD、Revit、Eplan | 旧范式对照——证明 GUI-only 路径的效率天花板 |
| **MCP 接口** | Agent 通过工具调用操作多源数据 | Excel、CAD API、PDF 解析器 | 当前 AI4E 主流路线 |
| **piki 工具链** | Agent 生成 YAML，piki check 自动校验 | piki CLI + 规则引擎 + 生成器 | SDE 主张的声明式路径 |

四条路径不是为了互斥竞争，而是为了回答一个核心问题：**在交付物质量和效率两个维度上，声明式路径（piki）是否显著优于 MCP 路径和 GUI 路径？**

#### 6.2.2 交付物清单

每个任务要求产出四类工程交付物：

| 交付物 | 内容 | 验收方 |
| -------- | ------ | -------- |
| **设备布置表 + 线缆连接表** | 设备位置、端口对端关系 | 设计评审 |
| **面板图 + 端口图** | 机柜面板布局、设备端口占用 | 施工 |
| **采购清单（BOM）** | 物料清单，含型号、数量 | 采购 |
| **施工交底工单** | 可交付施工的作业指导 | 施工 |

#### 6.2.3 评估维度

| 维度 | 指标 | 测量方法 |
| ------ | ------ | --------- |
| **完成度** | 四类交付物各是否产出 | 文件存在性 + 内容非空 |
| **正确性** | 布置表（数量、型号、U 位）、连接表（端口对端、接口兼容）、面板/端口图（与布置表一致性）、BOM（完整性、可采购性）、交底工单（可执行性、风险覆盖） | LLM-as-judge + 人工抽查 |
| **效率** | 任务完成耗时、修改-验证迭代次数 | 计时 + 操作日志 / git log |
| **可解释性** | 设计决策是否可追溯（为什么选这个 U 位？为什么选这个型号？） | commit message / 操作者阐述 |

#### 6.2.4 过程指标与结果指标的区分

piki 路径中，Agent 可以（也应该）主动运行 `piki check` 迭代修正设计——就像软件工程师提交 PR 前先跑测试。规则引擎的通过率因此是一个**过程指标**而非**结果指标**：

- **过程指标**（piki check 通过率）：Agent 是否善用了规则引擎来发现和修正错误？
- **结果指标**（交付物评估）：最终产出是否正确、完整、可用？

一个 Agent 可能 piki check 全部通过但交付物仍有问题（BOM 漏项未在规则覆盖范围内）。区分过程指标和结果指标，才能真实反映 Agent 的工程设计能力。

### 6.3 指标汇总

| 维度 | 一级指标 | 二级指标 | 测量方法 |
| ------ | --------- | --------- | --------- |
| **过程正确性** | L0-L4 各层通过率 | Pass@1 / Pass@k | piki check 输出 |
|  | 交付物生成成功率 | 生成器退出码 | piki generate 输出 |
| **结果完成度** | 交付物完整率 | 四类各是否产出 | 文件存在性 + 内容非空 |
| **结果正确性** | 布置表正确性 | 设备数量、型号、U 位 | LLM-as-judge + 人工 |
|  | 连接表正确性 | 端口对端、接口兼容 | LLM-as-judge + 人工 |
|  | 面板/端口图正确性 | 与布置表一致性 | LLM-as-judge + 人工 |
|  | BOM 完整性 | 物料项、型号、数量 | LLM-as-judge + 人工 |
|  | 交底工单可用性 | 步骤可执行、风险覆盖 | LLM-as-judge + 人工 |
| **效率** | 任务完成耗时 | 总人时/机时 | 计时 |
|  | 迭代次数 | 修改-验证循环次数 | git log / 操作日志 |
| **可解释性** | 设计决策可追溯 | 变更原因可追溯 | commit message |
|  | 约束意识 | 是否了解并避开已知约束 | piki check 调用记录 |

## 7. 相关工作

### 7.1 软件工程 Benchmark

**SWE-Bench** [1] 是 SD-HWE-Bench 最直接的设计参照。SWE-Bench 证明了四个原则：(1) 从真实仓库中抽取任务可以保证生态有效性；(2) 测试套件作为 ground truth 可以实现完全自动评分；(3) Pass@k 指标可以区分不同模型的能力层级；(4) benchmark 本身可以驱动 RLVR 训练。SD-HWE-Bench 继承这些原则，但面临一个 SWE-Bench 不需要回答的问题：当声明式工程设计的实践本身尚不存在时，如何定义"真实任务"？我们的回答是：从零人工构建、诚实承认边界、以定义型 benchmark 的定位来驱动范式建立。

**Code RLVR** [2]（以 DeepSeek-R1 为代表）证明了可验证奖励信号在代码生成中的有效性。SD-HWE-Bench 的 L0-L4 分层验证体系直接服务于 RLVR 训练：秒级反馈、确定性奖励、可解释失败、分层课程学习。这使 SD-HWE-Bench 不仅仅是排行榜，更是训练基础设施。

### 7.2 工程 AI Benchmark

**AEC-Bench** 是 2026 年最具影响力的工程 Agent benchmark，测的是 AI 对施工图纸和规范文档的阅读和判断能力——图纸审阅、交叉引用校验、提交文件比对。它与 SD-HWE-Bench 的关系是互补而非竞争：AEC-Bench 评测的是旧范式下的 AI 辅助能力（AI 在人类主导的审图流程中扮演助手），SD-HWE-Bench 评测的是新范式下的 AI 核心能力（AI 自己完成从需求到设计的全链路）。

**TeleQnA / TaleQnAD**：电信领域的问答型 benchmark，多选题形式，测量模型对工程术语的理解。属于知识层评测，不涉及设计生成。

**VerilogEval / ChipBench / AMS-IO-Bench**：EDA 领域的代码生成 benchmark，证明了"自然语言→结构化中间格式→工业规则校验"这条路径的可行性。AMS-IO-Bench 几乎就是 SD-HWE 在芯片 I/O ring 领域的预演，但输出格式局限于 RTL/网表，缺少跨领域统一的声明式语言。

### 7.3 声明式建模与形式化方法

**SysML v2** [4] 是 OMG 主导的下一代数模混合建模语言。SD-HWE 与 SysML v2 的差异已在 §2.2 中详细讨论：核心分歧在于真相源形态（模型仓库 vs. 文本文件）和版本控制对象（模型版本 vs. Git 行级历史）。

**Building Information Modeling (BIM)** 和 **Industry Foundation Classes (IFC)**：BIM 的目标与 SD-HWE 部分重叠——让建筑工程数据机器可读。但 BIM 的数据交换格式（IFC）是面向互操作性的序列化格式，而非面向 Agent 的读写和版本控制格式。一个 IFC 文件包含数千个 GUID 引用的嵌套实体，不适合 LLM 直接操作。

---

## 8. 结论

本文提出了 SD-HWE-Bench，一个面向软件定义硬件工程的声明式设计能力评测基准。其核心贡献在于：(1) 将工程设计表达为文本原生的声明式语言（PDL/PLL/PML），使 Agent 可直接读写、diff 和版本控制；(2) 将工程约束映射为 L0-L4 多层规则引擎，提供秒级、确定性、可解释的验证反馈；(3) 在此基础上构建了一个标准化、可自动评分、可复现的能力竞技场，服务于工程 Agent 的评测和 RLVR 训练。

SD-HWE-Bench 选择电信机柜部署作为初始任务域，不是因为它是 SD-HWE 范式通用性的证明，而是因为它是当前语言和工具链能力边界内最干净的验证场景。我们诚实地承认了连续空间、非线性物理、时序过程、多专业耦合、模糊需求五个结构性缺口——这些缺口不是"更多数据"能填补的，需要在语言设计、工具链和方法论上做出基础性扩展。

SD-HWE-Bench 的野心不在于做"最真实的 benchmark"，而在于做"最能激活工程 AI 研究的 benchmark"。类比 ImageNet 之于计算机视觉、SWE-Bench 之于代码生成，SD-HWE-Bench 的目标是定义一个领域的问题形式和评判标准，从而让工程 AI 的研究从"漫无目的的探索"进入"有明确目标、可量化进展、可复现对比"的阶段。
