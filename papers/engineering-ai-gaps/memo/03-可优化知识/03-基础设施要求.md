# 可优化知识：基础设施要求

本 memo 记录为了支撑“可优化知识鸿沟”实验，需要在 SD-HWE-Bench 基础设施中具备的能力。

## 知识库结构化

正式实验需要把知识从散文和临时参数中抽取为可版本控制、可追溯、可计算的知识库。

每个知识条目至少需要记录：

- `knowledge_id`：稳定 id。
- `knowledge_family`：equipment、climate、tariff、water、load、cost、construction、risk 等。
- `source`：datasheet、报价、规范、项目经验、合成场景或实测数据。
- `version`：版本或日期。
- `unit`：单位。
- `valid_range`：适用范围。
- `uncertainty`：确定值、区间、分布或场景集。
- `used_by`：被哪些 evaluator、critic、optimizer 或任务引用。

知识库不应只是给 Agent 阅读的文档，而应能被评估器和优化器直接消费。

## 动态时序输入

AIDC probe 至少需要以下时序数据：

- 全年或代表日气候曲线：干球温度、湿球温度、湿度。
- 电价曲线：峰谷电价、需量电费、季节电价或实时价格。
- 水价曲线：基础水价、阶梯水价、缺水或限水场景。
- IT 负载曲线：日内波动、季节波动、训练/推理负载形态。
- SLA 需求曲线：温度、容量、可用性或服务等级要求。

每条曲线需要统一时间粒度、时区、缺失值处理和聚合规则。

## 设备与供应商知识

候选设备库需要从静态容量表升级为可优化知识库。

冷却设备至少记录：

- 额定制冷量。
- COP/EER 随负载和湿球温度变化的曲线。
- 最小/最大运行范围。
- CAPEX、维保成本、寿命。
- 水耗曲线或水侧约束。

储能设备至少记录：

- 容量、功率、往返效率。
- 衰减模型。
- CAPEX、替换周期、运维成本。
- 充放电约束。

供配电设备至少记录：

- 额定容量。
- 效率曲线。
- 冗余模式。
- CAPEX、损耗和维护成本。

供应商与项目知识至少记录：

- 报价。
- 交期。
- 可用库存。
- 维保等级。
- 风险或不确定性场景。

## 时序经济评估器

需要一个统一的动态评估器，将设计方案、设备曲线、时序输入和运营策略转化为可比较指标。

评估器需要输出：

- feasibility：是否满足硬性约束。
- annual energy。
- annual water usage。
- annual electricity cost。
- annual water cost。
- maintenance cost。
- CAPEX。
- OPEX。
- TCO / NPV / LCOE 或类似生命周期指标。
- PUE。
- SLA violation rate。
- risk metrics。

评估器还需要支持场景批量运行，例如不同天气年、电价年、水价年、负载年和供应链扰动。

## 设计-运营联合优化接口

实验需要区分 Agent 的提交物和参考优化器的搜索空间。

Agent 提交物应包括：

- 设备选型。
- 容量配置。
- 布局或关键空间参数。
- 储能容量与功率。
- 光伏或其他能源配置。
- 运营调度策略。
- 温度设定点策略。
- 风险或应急策略。

参考优化器需要读取同一知识库，输出：

- 非支配解集。
- 每个解的设计变量。
- 每个解的动态评估结果。
- 与 Agent 方案的距离或 regret。

## Artifact 与指标

每个 run 的 manifest 应记录：

- `knowledge_condition`：static-compliance、static-cost、static-curves、dynamic-knowledge、oracle 等。
- `knowledge_sources`：本次可见或可用的知识条目。
- `time_series_sources`：气候、电价、水价、负载曲线版本。
- `equipment_library_version`。
- `cost_model_version`。
- `objective_set`。
- `scenario_seed`。
- `feasibility_passed`。
- `tco`、`capex`、`opex`、`pue`、`sla_violation_rate`、`water_cost`、`risk_metrics`。
- `pareto_rank`。
- `dominated_by`：被哪些参考或 Agent 解支配。
- `decision_variables`：关键设计变量快照。

聚合脚本需要输出：

- 各知识条件下的 feasibility rate。
- TCO/PUE/SLA/OPEX 分布。
- CAPEX-TCO 与 CAPEX-SLA Pareto frontier。
- Dynamic Knowledge 相对 Static Compliance 的 frontier shift。
- 设备组合和调度策略的 decision shift。
- Agent 相对 Oracle Optimizer 的 regret。

## 与可执行约束的边界

基础设施必须把硬性可行性检查和优化目标分开。

- 硬性约束：容量不能超限、接口必须兼容、SLA 不得超过失败阈值、施工资源不能冲突。这些属于可执行约束。
- 优化目标：在硬性约束内降低 TCO、OPEX、PUE、水耗、风险或工期。这些依赖可优化知识。

同一条知识可能同时参与两者，但角色不同。例如冷机容量下限是约束，COP 曲线对全年能耗的影响是优化知识；电价上限不是硬约束，电价时序对储能策略的影响是优化知识。

## 进入正式实验前的最低门槛

- AIDC 任务拥有可版本化的设备候选库。
- 至少有一套气候、电价、水价和负载时序曲线。
- 时序经济评估器能稳定输出 TCO、PUE、SLA、CAPEX、OPEX、水耗和风险指标。
- Static 与 Dynamic 条件使用相同硬性可执行约束。
- 所有条件能被完整动态评估器离线重评。
- 至少有一个参考优化器或枚举搜索器生成 baseline Pareto frontier。
- run artifact 能记录知识版本、曲线版本、目标函数、场景 seed 和关键设计变量。
- 至少完成一个小矩阵 smoke，确认 Dynamic Knowledge 条件能产生可解释的 decision shift 或 frontier shift。
