# 可优化知识：实验设计

本 memo 记录“可优化知识鸿沟”章节的实验设计。实验目标不是证明“给 Agent 更多资料会更好”，而是证明当工程知识被转化为时序模型、设备曲线、经济参数和搜索空间后，是否会改变设计决策和 Pareto frontier。

## 实验目标

实验需要支撑以下问题：

- 静态合规检查能否区分“可行方案”和“前沿方案”？
- 当气候、电价、水价、负载、设备曲线和报价进入评估器后，Agent 的设计选择是否发生系统性变化？
- 可优化知识是否会改善 TCO、PUE、SLA、OPEX、CAPEX、工期或风险等目标？
- 可优化知识的收益来自知识本身，还是来自更强的约束检查？
- 这些结论是否能通过 Pareto frontier 而不是单一 pass rate 表达？

## 比较条件

在同一 AIDC 设计-运营任务上设置多组知识条件。所有条件都使用同一套硬性可执行约束作为最终可行性闸门，差异只在可优化知识是否进入目标函数、模型参数和搜索空间。

| 条件 | Agent 可用信息 | 评估方式 | 目的 |
|---|---|---|---|
| Static Compliance | 静态容量、冗余、间距、PUE/SLA 阈值 | pass/fail + 静态评分 | 测试只靠合规能达到的上限 |
| Static + Cost | 增加设备报价、CAPEX/OPEX 参数 | LCC/TCO 评分 | 测试成本知识是否改变选型 |
| Static + Curves | 增加设备效率曲线、冷却曲线、负载曲线 | 准动态仿真 | 测试工程曲线是否改变设备组合 |
| Dynamic Knowledge | 增加全年气候、电价、水价、负载、设备报价和效率曲线 | 时序仿真 + TCO/PUE/SLA | 测试完整可优化知识是否推动 frontier |
| Oracle Optimizer | 使用同一知识库的参考优化器 | 搜索得到参考 Pareto frontier | 给 Agent 方案提供上限参照 |

核心控制原则：最终可行性约束不变。知识条件改变的是可优化信息和评分目标，而不是降低或提高工程底线。

## 可优化变量

AIDC probe 可以按变量权限分层，观察知识进入后是否真的触发设计-运营联合优化。

| 设置 | 可优化变量 | 观察对象 |
|---|---|---|
| Fixed Design | 设备和布局固定，只优化运营调度 | 调度策略对 OPEX/SLA 的影响 |
| Equipment-only | 优化冷机、储能、变压器、光伏、机柜/服务器候选 | CAPEX/OPEX 权衡 |
| Schedule-only | 设备固定，优化温度设定点、储能充放电、负载迁移 | 峰谷套利、free cooling、SLA 风险 |
| Joint DTCO | 设备选型 + 布局 + 运营调度联合优化 | Pareto frontier 是否显著外移 |

如果 Joint DTCO 明显支配单独优化条件，说明可优化知识的价值来自跨层组合，而不是某一条规则或某一个参数。

## 核心指标

本实验不以 pass rate 为主指标。pass/fail 只用于确保方案处于可行域内，真正的观测对象是 frontier 和目标权衡。

- **Feasibility rate**：满足硬性可执行约束的比例。
- **TCO**：生命周期总成本。
- **CAPEX / OPEX**：投资与运营成本拆分。
- **PUE / annual energy**：全年能耗和冷却效率。
- **SLA violation rate**：温控、供电、容量或服务水平违规率。
- **Water usage / water cost**：用水量与水费。
- **Risk metrics**：极端天气、价格波动、供应延迟或设备故障下的风险。
- **Pareto dominance**：不同知识条件下方案之间的支配关系。
- **Frontier shift**：Dynamic Knowledge 相对 Static Compliance 的 Pareto frontier 外移程度。
- **Decision shift**：设备组合、储能容量、冷却策略、调度策略是否发生可解释变化。

## 实验判断标准

可优化知识的重要性可以通过三类证据证明。

### 1. 同可行性下的性能提升

所有方案都通过相同硬性约束，但 Dynamic Knowledge 条件在 TCO、OPEX、PUE 或 SLA 风险上显著更好。

示例判断：

- 相同 SLA 下，TCO 更低。
- 相同 CAPEX 下，全年 OPEX 更低。
- 相同 TCO 下，SLA violation 更低。
- 相同 PUE 阈值下，极端天气风险更低。

### 2. 设计决策发生可解释变化

引入动态知识后，Agent 不只是微调参数，而是做出不同设计选择。

可能的变化包括：

- 选择更高 CAPEX 但低 OPEX 的冷却设备。
- 增大储能容量以利用峰谷电价。
- 调整冷却设定点以平衡 free cooling 和 SLA 风险。
- 因水价或气候条件改变冷却技术路线。
- 因设备报价或交期改变冗余策略。

### 3. Pareto frontier 外移

Dynamic Knowledge 或 Joint DTCO 条件产生的非支配解集相对静态条件外移。只要外移稳定出现，就说明可优化知识改变了可达设计上限。

## 关键对照

实验需要避免把“知识更多”和“约束更强”混在一起。至少需要以下对照：

- 同一任务、同一 Agent、同一硬性约束，在不同知识条件下比较。
- 所有条件最终都用完整动态评估器离线重评，避免静态条件因为评分较弱而虚高。
- Static 条件也允许 Agent 满足所有可执行约束，只是不提供动态曲线和经济模型。
- Dynamic 条件中的知识必须结构化进入模型，而不是只作为长文档提示。
- Oracle Optimizer 使用同一知识库，作为 Agent 可达到上限的参照，而不是额外真值来源。

## 与前两个实验的关系

“可闭环表征”实验关注提交对象能否支撑闭环。

“可执行约束”实验关注工程正确性能否被判定和定位。

“可优化知识”实验关注正确方案之间的性能上限。

三者的递进关系是：

```text
可闭环表征 -> 让工程状态可提交、可检查、可修复
可执行约束 -> 把错误挡在可行域之外
可优化知识 -> 在可行域内推动解空间上限
```

因此，本实验不应被写成“知识文档越多越好”，而应写成：当知识成为可计算的目标、参数、曲线和搜索空间时，工程 AI 才能从合规设计走向前沿设计。
