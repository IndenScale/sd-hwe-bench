# 3. 可执行实验平台：SD-HWE-Bench

SD-HWE-Bench 在本文中作为三重鸿沟的实验平台，而不是唯一主张。它提供四个关键能力：可计算表征、可执行 critic、可复现任务增量和可观察 failure mode。

## 3.1 Engineering as Code 与 ADL

SD-HWE-Bench 建立在 Engineering as Code 范式上：将系统级工程设计转化为文本原生、可版本控制、可检查的声明式工件。ADL 用工程对象、属性、引用、连接、配合、布局和调度等结构表达设计状态，使 Agent 的输出不再是一段自然语言方案，而是一个可提交、可 diff、可评分的工程 patch。

## 3.2 DTS 分层约束

DTS 将工程正确性拆分为多个可执行层：

- L0：语法和文件格式。
- L1：schema 与类型。
- L2：引用完整性。
- L3：静态工程约束，例如功率、接口、载荷和规则断言。
- L4：动态模型、调度仿真或多方案决策。
- L5：几何干涉、空间约束和施工可建性。
- L6：预留高精度 FEM/CFD 等仿真。

这种分层使错误不只是 pass/fail，而能被定位到约束类别，为 repair loop、错误模式分析和 RLVR 奖励设计提供信号。

## 3.3 Canonical Lineage 与任务提取

SD-HWE-Bench 从 canonical 工程的 commit 历史中提取任务。每个任务可理解为一个工程增量：给定某个设计状态和需求，Agent 需要提交下一个状态的工程 patch。AIDC 60MW 任务采用 lineage 化 canonical，将概念设计、详细设计和 EPC 排程组织为同一工程仓的连续 tag，从而减少多阶段任务之间的内容漂移。

## 3.4 本文如何使用该平台

本文不把任务数量作为主要论据，而把 SD-HWE-Bench 用于受控对照：

1. 在同一类工程需求上比较不同表征底座是否能形成可复现评估闭环。
2. 在同一类约束下比较自然语言规范与可执行 DTS 的行为差异。
3. 在 AIDC 场景中比较局部优化与设计-调度联合优化的上限差异。

这种用法使小规模任务集仍然具有解释力：我们关心的是三类鸿沟的因果结构，而不是仅凭任务数量证明覆盖面。
