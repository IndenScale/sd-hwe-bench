# 2. 三重鸿沟

本章定义本文的三重鸿沟框架。三者不是并列的工程困难清单，而是递进关系：表征决定任务能否被机器提交与检查；约束决定提交是否能被判定为真实正确；知识决定正确解空间是否接近前沿实践。

## 2.1 表征鸿沟

表征鸿沟指工程系统缺少一种既能表达工程语义、又能被机器稳定操作的设计工件。自然语言描述适合交流意图，但不适合做 diff、类型检查和局部 repair。CAD/BIM 表达几何强，但通常对电气、热、供应链、施工、运营调度等语义支持不足，且难以作为高频 Agent 评测的文本原生接口。供应商 PDF 和 Excel 参数表包含大量权威知识，却不形成统一提交对象。

一个严肃的工程 AI benchmark 至少需要以下表征能力：

- 工程对象可枚举：设备、部件、接口、位置、连接、资源与活动可以被机器引用。
- 关系可检查：引用、配合、连接、依赖、前置条件和资源占用不是散文，而是结构化关系。
- 状态可 diff：一个设计增量能够像代码 patch 一样被审查。
- 反馈可定位：错误可以定位到字段、对象、关系或约束层。
- 修复可闭环：Agent 能基于反馈修改提交，而不是重新生成一段自然语言说明。

本文将比较 MCP、CUA、OpenSCAD 与 ADL 四类路径。MCP 与工具调用能帮助 Agent 访问外部系统，但往往不定义统一工程提交对象。CUA 能操作 GUI，但交互脆弱、慢且难复现。OpenSCAD 提供 code-like geometry，却主要表达参数化形状，缺少完整工程语义。ADL 的目标则是把 Part、属性、连接、配合、布局和调度等对象组织为可验证工程声明。

## 2.2 约束鸿沟

约束鸿沟指工程正确性大量存在于自然语言规范、人工经验和隐式审查流程中，而没有变成可执行 critic。自然语言规范可以告诉人“应当满足什么”，但不能稳定阻止 Agent 生成 pseudo-correct 方案。

Pseudo-correctness 是本文的核心观测对象之一。它指方案在自然语言上看起来完整、专业、合理，但在可执行检查下失败。例如：

- 引用不存在的设备或端口。
- 接口类型、连接器、线缆或光模块不兼容。
- 功率预算、结构载荷、接地、防雷或消防间距遗漏。
- 布局满足视觉直觉，但违反维护空间、吊装半径或施工路径。
- 调度计划看似完整，却违反资源容量、天气窗口或 SLA。

当约束可执行且 Agent 能在提交前运行检查时，许多任务会迅速饱和。这并不说明任务无价值，而是说明工程能力很大程度上来自 Agent 与约束系统构成的闭环。相反，当约束只以自然语言存在，即使给 Agent 多部规范文档，它仍可能遗漏跨文档、跨专业、跨尺度的关键条件。

## 2.3 知识鸿沟

知识鸿沟比前两者更深。即使我们拥有可计算表征和可执行约束，工程 AI 仍可能被保守、简化或滞后的知识边界限制。

工程规则常常来自安全冗余、静态假设和历史工艺窗口。结构规范会预留大量安全系数；DFM 规则可能没有吸收新材料、新设备或新工艺的能力边界；施工规范可能默认传统装备与组织方式；运营模型可能忽略当地气候、电价、供应链和供应商曲线。许多最有价值的知识不在公开规范里，而在设备厂商 datasheet、应用笔记、项目报价、现场经验和行业关系中。

半导体领域的 PDK 是一个重要对照。PDK 将 foundry 的工艺能力形式化为 EDA 可消费的规则、模型和参数，使 Fabless、EDA 与 Foundry 能快速共享新工艺进步。许多系统级工程领域缺少 PDK-like knowledge infrastructure。随着基础材料与单点器件进步放缓，性能改进越来越依赖设计-工艺-运营-施工的协同优化，也就是穿透抽象层级的 DTCO。

因此，知识鸿沟的实验不应只问“方案是否满足规则”，而应问“当情境化知识进入优化空间时，解空间上限是否改变”。AIDC 是合适的 probe：设备选型、冷却布局、储能容量、光伏配置、当地气候、电价结构和调度策略之间存在强耦合。局部满足规则不等于全局最优；跨层知识是否进入优化循环，决定了 PUE、TCO、SLA 和投资回收的 Pareto frontier。
