# 4. 实验一：表征鸿沟

本实验验证不同工程表征路径是否能够支持严肃的自动化评估闭环。比较对象包括 MCP 方法、CUA 方法、OpenSCAD 方法和 ADL 方法。

## 4.1 实验设计

我们选择若干等价工程需求，例如设备扩容、机柜布局调整、连接关系更新或 AIDC 子系统配置。每种方法都要求 Agent 完成同一意图，但允许的操作接口不同：

| 方法 | 工程表征 | Agent 操作 | 评分方式 |
|---|---|---|---|
| MCP | 工具 API、文档接口或外部系统资源 | 查询、调用、汇总或提交工具参数 | 依赖工具返回与人工设计的 wrapper |
| CUA | CAD/BIM/表格等 GUI | 点击、输入、导出文件 | 依赖视觉状态和后处理检查 |
| OpenSCAD | 参数化几何代码 | 生成或修改几何脚本 | 几何渲染、尺寸或碰撞检查 |
| ADL | 声明式工程对象和关系 | 修改工程声明文件 | DTS 分层 critic |

## 4.2 指标

本实验不以 pass rate 为唯一指标，而关注闭环能力：

- **任务形式化成本**：从工程需求到可评测任务需要多少人工建模步骤。
- **提交确定性**：同一提交是否能稳定复现评分。
- **反馈延迟**：从提交到结构化错误反馈的时间。
- **错误可定位性**：错误能否定位到对象、字段、引用或规则。
- **repair 可行性**：Agent 是否能基于反馈进行局部修复。
- **评分覆盖度**：是否覆盖语法、引用、工程约束、几何、仿真和交付物。

## 4.3 预期发现

MCP 和 CUA 能显著增强 Agent 对现有工具的访问，但它们本身不必然提供统一的工程提交对象。OpenSCAD 展示了 code-like geometry 的价值，却难以完整表达电气、热、施工、调度和供应链语义。ADL 的优势不在于替代所有工业软件，而在于提供一个可执行评估层：工程对象、引用关系和约束接口被统一到可检查 patch 中。

因此，表征鸿沟实验的目标不是证明某一种接口在所有工程场景中最好，而是证明：**只有当工程表征同时支持语义对象、关系、diff、critic 和 repair 时，benchmark 才能成为高频可复现的评估环境。**
