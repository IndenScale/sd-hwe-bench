# 1. 引言

软件工程已经经历了一个清晰的 AI 加速循环：代码是文本原生、可版本控制、可执行、可测试的工件；GitHub issue、pull request 与测试套件将真实开发活动转化为可复现任务；SWE-bench 等 benchmark 又把这些任务变成模型训练、评估和系统改进的共同坐标系。结果是，代码 Agent 的能力提升不只来自更强模型，也来自一个高频、可验证、可复用的工程环境。

系统级工程设计尚未拥有类似循环。数据中心、通信基站、机械装配、能源系统和施工排程等任务同样充满确定性反馈：连接是否兼容、功率是否超限、结构是否满足载荷、布局是否冲突、施工计划是否违反资源约束、全年运营是否满足 PUE 与 SLA。然而，这些反馈通常被锁在 CAD/BIM 文件、PDF 规范、Excel 表格、商业 GUI、供应商手册和人工审查流程中。结果是，工程 AI 的评估常常退化为 toy problem、文档问答、自然语言方案评审，或昂贵且难复现的仿真 wrapper。

本文的核心主张是：**工程 AI 的瓶颈不是单一模型能力，而是表征、约束与知识三类基础设施缺失。**

第一，**表征鸿沟**使工程任务难以变成可计算 benchmark。没有类似代码的工程表征，就难以获得稳定的 diff、提交、检查、修复和复现实验。

第二，**约束鸿沟**使 Agent 容易产生 pseudo-correctness。自然语言方案可能术语完整、结构合理、看似专业，却在引用完整性、接口兼容、功率预算、结构载荷、防雷消防、日照遮挡或施工可建性上遗漏关键约束。工程评估需要的不是“像答案”，而是可执行约束下的闭合设计。

第三，**知识鸿沟**限制了工程 AI 的优化上限。许多规则是保守、静态或滞后的；前沿设备、材料、工艺、气候、电价和供应链知识往往位于供应商手册、项目经验或局部工业软件中，未能像半导体 PDK 一样被形式化并进入跨组织、跨工具的优化循环。当基础材料和单点器件进步放缓，性能提升越来越依赖穿透抽象层级的协同优化，例如设备选型、布局、调度、施工和成本模型的联合设计。

我们以 SD-HWE-Bench 作为可执行实验平台来验证这一主张。SD-HWE-Bench 本身不是本文的全部贡献；它在本文中承担一个更具体的角色：让三类鸿沟能够被隔离、比较和测量。我们提出三组实验：

1. 表征实验：比较 MCP、CUA、OpenSCAD 与 ADL 在任务形式化、提交确定性、反馈延迟、repair 能力和评分覆盖上的差异。
2. 约束实验：比较自然语言规范、目录化规范和可执行 DTS 约束下的 pseudo-correctness、遗漏项分布和 repair 饱和曲线。
3. 知识实验：通过 AIDC 设计-调度联合优化 probe，分析设备选型、布局、气候、电价与调度策略等情境化知识如何改变 PUE/TCO/SLA 的 Pareto frontier。

本文因此不是一篇单纯的 benchmark 规模论文。我们的目标是解释为什么工程 AI 尚未出现软件工程式的快速评估-训练闭环，并给出一种可执行、可扩展的研究框架：从“看起来正确”的自然语言工程方案，走向“可验证正确”的工程闭合，再走向“前沿最优”的知识驱动协同设计。
