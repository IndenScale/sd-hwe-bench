## 1. 核心叙事

**一句话贡献**：SD-HWE-Bench 是第一个系统评估 AI Agent 在声明式硬件工程（Hardware Engineering）任务上设计能力的可执行 benchmark；我们证明，通过 EaC/ADL 把物理工程转化为可计算表示，并配合 ESA 确定性反馈，可以显著降低任务构建成本、提升 Agent 生成合法设计的通过率，从而把为 SWE-bench 优化的训练/评估基础设施对齐到硬件工程领域。

**四幕叙事弧线**：

1. **AI4E 的重要性**：物理工程（芯片、机械、建筑、能源、航空航天等）决定人类基础设施的成本、性能与可持续性，AI for Engineering 具有巨大社会与经济价值。
2. **E4AI 的重要性**：工程是除游戏、数学、代码之外，又一个“确定性奖励稠密”的领域——每个设计决策都可以通过仿真、规则或测试得到快速、明确的反馈，是 RL/RLVR 理想的下一个战场。
3. **差距**：软件工程（SWE）已借助可计算表示（代码）和自动化测试形成成熟闭环，催生了 SWE-bench 等大规模 benchmark；而硬件工程（HWE）表示碎片化、依赖商业 GUI 软件和昂贵仿真，训练/评估基础设施薄弱，导致 HWE 在 AI Lab 的叙事、注意力、资源分配中显著不足，形成系统性欠训练。
4. **填补空白**：本项目用 EaC（Engineering as Code）和 ADL 统一硬件工程的可计算表示，参考 IC/DEA 在语言、格式、协议、工具链上的成功实践，并借鉴 SWE-bench 及其扩展的任务形式，构建跨领域可扩展、低成本高频反馈的 SD-HWE-Bench。

**三幕结构（正文组织）**：

1. **问题**：物理工程 AI 缺的不是模型能力，而是可计算表示层与可高频迭代的评估基础设施。
2. **方法**：ADL + ESA + canonical 工程 commit 提取 = 可自动评分的 benchmark。
3. **证据**：多模型基线 + ablation + 错误模式分析。

## 4. 从汇报策略倒推的工程任务

要完成这篇论文，必须完成以下工程/实验任务：

1. **canonical 工程 authoring**
   - 电信机柜扩容工程（20–30 commits）
   - 数据中心一排机架工程（15–20 commits）
   - 机械键盘外壳装配工程（15–20 commits）

2. **任务提取工具**
   - 实现 `tools/extract_tasks.py`
   - 为每个 canonical 工程编写 `task_manifest.yaml`
   - 验证所有 reference solution 通过 `piki check`

3. **实验**
   - 3–4 个 Actor 基线（Kimi、DeepSeek Pro、GPT-4o、Claude）
   - pass@5 主实验
   - no-repair vs repair ablation（5 tasks × 1 model × 3 passes）
   - full / oracle / collapsed 上下文消融

4. **容器化**
   - 构建 `sd-hwe-bench-piki` image
   - 所有评分走 container

5. **复现包**
   - README + 脚本 + 数据版本
   - 公开到 Zenodo/GitHub（匿名版）

6. **论文写作**
   - 填充各 section markdown
   - 生成图表
   - 翻译成英文版本（sections-en）
