# 8. 错误模式与难度分析

本章对 Agent 失败进行深入诊断——不只报告数字，而是分析**为什么失败、瓶颈在哪里、难度因子是什么**。这是让 SD-HWE-Bench 从"排行榜工具"升格为"科学分析平台"的关键章节。

## 8.1 失败模式分类

我们将 Agent 的失败按根因分为四类：

### 8.1.1 语法/格式错误（L0 失败）

Agent 输出的 YAML 无法解析，或完全未生成 `expected_files`。

**典型表现**：YAML 缩进不一致、引号未闭合、YAML 内嵌非 YAML 内容（如 Agent 将整个对话输出混入文件）。

**根因**：Agent 对 YAML 语法的掌握不足，或在长输出中失去格式一致性。

**修复潜力**：低——此类错误通常无法通过 repair loop 自我修正（可能需要更好的输出解析器或格式约束 prompt）。

### 8.1.2 类型/约束违背（L1/L2 失败）

Agent 的语义不合理——属性值非法、引用不存在的 Part，或端口名不存在。

**典型表现**：

- `power: -100`（负功率）
- `connector_type: unknown_type`（未定义的连接器类型）
- 引用了一个未在 PDL 中定义的 `part_id`

**根因**：Agent 缺乏对 ADL Schema 的深层理解，或在生成时未参考 PDL 中已定义的 Part 列表。

**修复潜力**：中等——DTS 报告可明确指出缺失/非法的字段，Agent 有可能在下一轮修正。

### 8.1.3 设计逻辑遗漏（L3 失败）

Agent 的方案在语法和语义上都合法，但**遗漏了某条设计规则**——功率预算超限、U 位冲突、接口不兼容、散热不足。

**典型表现**：

- 增加了设备但未更新 PDU 容量
- 两台设备共享同一 U 位
- 将 LC 光纤口连接到 RJ45 电口
- 热功率总和超过空调容量

**根因**：Agent 在长上下文、多约束空间中无法保持全局一致性——它"知道"所有规则，但无法在设计过程中**同时**满足它们。

**修复潜力**：高——这是 DTS 反馈最擅长的场景。明确告知"哪条规则在哪个位置被违反"后，Agent 通常能针对性修正。这类失败最能证明确定性反馈的因果价值。

### 8.1.4 几何/空间推理（L4 失败）

Agent 的方案在逻辑上合法，但在三维空间中不可行——碰撞、间距违规、深度越界。

**典型表现**：

- 两个 Part 的包围盒重叠
- Part 超出机柜深度
- 散热间隙不满足最小要求

**根因**：LLM 的空间推理能力有限——缺乏对三维几何的直观理解。文本形式的布局声明（如 `position: {x: 0, y: 0, z: 2}`）对模型来说是一串数字，而非空间关系。

**修复潜力**：低——DTS 可以告知"x 处发生碰撞"，但 Agent 可能不知道如何在不引入新碰撞的情况下重新布局。未来可能需要几何求解器辅助的 Agent 设计（如让 Agent 调用 piki 的自动布局工具）。

### 8.4.5 不可能任务与 Reward Hacking

并非所有任务在物理上都有合法解。实际工程中，需求本身可能自相矛盾——例如要求将 10 台 2U 设备塞入仅剩 15U 的机柜——此时正确的工程行为不是"硬塞"，而是**识别不可能性并合理驳回**。

SD-HWE-Bench 通过两种机制捕捉这一维度：

**不可能任务的标注与用途**：数据集中少量任务被显式标注为 `impossible`，其 scaffold 与需求之间存在不可调和矛盾（如容量不足、Part 库中不存在兼容型号等）。这些任务不参与 pass@k 主排名，而是用于评估 Agent 的**需求分析能力**——Agent 应主动创建 `.sdhwe.info_gap` 或 `.sdhwe.give_up` 并给出合理解释。即便 Agent 通过扭曲方案（如将设备声明为 0U 高度）强行通过 DTS，也会在 L5 交付物检查中被捕获。

**Reward Hacking 风险**：确定性反馈的一个潜在副作用是，Agent 可能学会"满足 DTS 但不满足设计意图"——例如将碰撞检测绕过后重新声明 Part 尺寸、或修改非任务要求的 scaffold 文件以消除约束检查。我们发现当前基线的 reward hacking 风险较低，因为通用 LLM Agent 缺乏对 DTS 内部实现的理解；但 ADL-specific 微调模型或 RLVR 训练的 Agent 可能学会利用 DTS 盲区。因此 SD-HWE-Bench 建议：

1. `expected_files` 和 scaffold 只读约束作为基础防护——Agent 无法通过修改 `models/` 中的 Part 定义绕过检查。
2. 未来版本将引入**对抗性验证任务**：在不可能任务中检测 Agent 是否产出了"形式上通过 DTS 但物理上荒谬"的方案。
3. Rubric critic 可作为第二道防线——通过 LLM-as-Judge 评估方案是否"合理"而非仅"合法"。

## 8.6 SWE-Bench 与 SD-HWE-Bench 的相关性分析

一个重要问题是：SD-HWE-Bench 是否仅仅是一个"换了数据集的 SWE-bench"？如果两者高度相关，则 HWE 本质上是 SWE 的领域特例，打通可计算表示层后现有模型能力可直接迁移；如果弱相关，则 HWE 构成一个独立的能力维度。

我们取同一模型在 SWE-bench 和 SD-HWE-Bench 上的 pass@1，计算 Spearman 秩相关系数并绘制散点图（图 X）。选取的模型包括 minimax-m2、kimi-k2.7、glm-4.7、deepseek-v4-pro、deepseek-v4-flash、gemini-3.1-pro、gemini-2.5-flash 等在两个 benchmark 上均有公开或可获取分数的模型。

**预期观察**（待实验验证）：

- 两者呈**弱正相关**（预期 ρ ≈ 0.3–0.5）。代码生成能力强（SWE-bench 高分）的模型在工程设计中往往也表现更好，但优势幅度远低于 SWE-bench 上的表现——deepseek-v4-pro 的 SWE-bench pass@1 可能达到 40%+，但 SD-HWE-Bench pass@1 大概率在个位数到十几。
- **所有模型在 SD-HWE-Bench 上的分数系统性地低于 SWE-bench**——没有一个模型的 SD-HWE-Bench pass@1 接近其 SWE-bench pass@1。这支持了 EaC 的"可计算表示缺口"假说：模型在软件代码上经过了充分训练和强化学习优化，而在 ADL 格式的工程设计声明上几乎是零训练。
- 如果散点图中出现**离群点**（如某模型 SWE-bench 高分但 SD-HWE-Bench 极低，或反之），则提示该模型的训练数据或 RL 策略存在领域特异性。

**这个分析的核心叙事价值**：证明 HWE 不是 SWE 的子集，"打通可计算表示层"≠"模型自动会做工程设计"。SD-HWE-Bench 评测的是一个当前模型体系下系统性欠训练的能力维度。

## 8.6 关键发现总结

预期关键发现（待实验验证）：

1. **瓶颈在 L3/ASA（静态分析），不在 L4/ADA（动态分析）**：DTS 的 L0–L2 层是基本门槛，ASA 是主要战场。这验证了 EaC 的主张——困难在于"同时满足所有约束"，而非"某些约束很复杂"。

2. **确定性反馈有因果价值**（§7 消融实验）：但如果 repair 提升集中在 L3 而非 L4，则说明 Agent 可以通过 DTS 反馈修正"遗漏"，但不能自我修正"空间推理不足"。

3. **长上下文是双刃剑**：更大的 scaffold 提供更多约束上下文，但也增加了上下文淹没风险。不同模型对此的耐受度不同。

4. **定位能力是瓶颈**：如果 Full Context 下定位 F1 很低，说明未来应在 prompt 设计、检索增强（RAG）或 Agent 工作流中重点改进"找到正确修改位置"的能力。
