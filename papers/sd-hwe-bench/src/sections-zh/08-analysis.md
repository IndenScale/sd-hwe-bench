# 8. 错误模式与难度分析

本章对 Agent 失败进行深入诊断——不只报告数字，而是分析**为什么失败、瓶颈在哪里、难度因子是什么**。这是让 SD-HWE-Bench 从"排行榜工具"升格为"科学分析平台"的关键章节。

## 8.1 核心发现：规范驱动性是区分度来源

实验揭示 SD-HWE-Bench 的核心区分度来自**规范驱动性**（Specification-Driven Design）——任务 requirement 仅描述功能目标，具体字段名、公式和阈值必须从规范文档中获取。Agent 可以像人类工程师那样"翻阅规范文档"（`cat docs/guides/canonical-project-design.md`），但必须在长上下文中保持对数十个字段约定的记忆。

各模型在 easy 任务上均达 100%，差异集中在任务需要跨文件协调引用（端口→光模块→光纤→配合的 4 层引用链）或进行物理公式计算（风载荷、链路预算、散热面积、AIDC 能耗/TCO/吊装）时出现。这证明 SD-HWE-Bench 的设计原则——区分度来自工程复杂性而非语法难度——是有效的。

### 8.1.1 自检循环未触发

当前 no-repair 实验中 self-check 平均轮次 < 0.5——Agent 几乎不主动执行 `piki check` 自检。这意味着 reported pass@1 更接近「单次生成」而非「多轮修复后」的表现。第 7 章的 repair ablation 将系统评估 DTS 反馈是否能通过多轮修复提升 pass@1。

### 8.1.2 方法论意义

Actor 框架的工程质量（文件系统访问、命令行工具集成、自检循环）对结果有决定性影响。SD-HWE-Bench 的实验结果必须明确报告 Actor 配置（CLI 框架 + 模型 + 上下文设置），不可直接跨框架比较 pass@1 数字。

## 8.2 失败模式分类

我们将 Agent 的失败按根因分为五类（基于当前 28 任务 × 2 模型 × 5 passes 数据）：

### 8.2.1 语法/格式错误（L0 失败）

Agent 输出的 YAML 无法解析，或完全未生成 `expected_files` / `expected_deliverables`。

**出现频率**：极低。当前 CLI Native 路径下，L0 失败主要由交付物生成失败引起，而非 YAML 解析错误。

### 8.2.2 类型/约束违背（L1/L2 失败）

Agent 的语义不合理——属性值非法、引用不存在的 Part，或端口名不存在。

**典型表现**：

- `interface_type: 'SFP28'`（schema 要求 `SFP28-module` 或 `SFP28-cage`）
- 引用了一个未在 PDL 中定义的 `part_id`
- 连接端点 `from_port` 未定义

**根因**：Agent 未严格参照 schema 和 PDL 中的已定义 Part 列表。长输出中容易遗忘字段精确值。

**出现频率**：约 10–15% 的失败归因为 L1/L2。

### 8.2.3 设计逻辑遗漏（L3 失败）

Agent 的方案在语法和语义上都合法，但**遗漏了某条设计规则**——功率预算超限、U 位冲突、接口不兼容、数值断言不满足。

**典型表现**：

- PDU 功率预算超限（TELECOM-POWER-001）
- 连接端口类型不兼容（TELECOM-CONN-002）：如光纤连接器配对错误
- 地板载荷数值断言超出阈值（telecom-cross-001）

**根因**：Agent 在长上下文、多约束空间中无法保持全局一致性——它"知道"所有规则，但无法在设计过程中**同时**满足它们。

**出现频率**：约 30–40% 的失败归因为 L3。

### 8.2.4 动态仿真/ADA 失败（L4 失败）

AIDC 与 EPC 任务中，Agent 方案未满足仿真硬约束（温度越限、PUE 超限、TCO 约束、工期/资源超限、应急预案不合法）。

**典型表现**：

- `aidc-60mw-001` 中 TCO 优化后仍不满足 PUE ≤ 1.22；
- `edge-dc-design-001` 中夏季峰值工况下进风温度超过安全阈值；
- `aidc-60mw-003` 中雨季未启用应急预案导致室外吊装延误，超过 deadline。

**根因**：Agent 对全年 8760h 运营或多专业施工网络的耦合约束（冷却器台数、冷冻水设定点、储能调度、变压器负载率、施工资源、天气窗口）缺乏全局优化能力，往往只满足局部指标。

**出现频率**：仅在 AIDC/EPC 任务出现，约占该类任务失败的 50%。

### 8.2.5 几何/空间/可建性推理（L5 失败）

**出现频率**：较低。当前 task 集中 L5 规则主要为 U 位冲突、3D 碰撞、维护空间等几何检查，Agent 在设计过程中通常能通过。跨专业任务（telecom-cross-001）中的部分失败可归因为 L3 数值断言与 L5 几何约束的联合违反。详细设计/EPC 任务新增的吊装/VDC 可建性检查预计将提供新的区分维度。

## 8.4 难度因子分析

### 8.4.1 规范驱动性

SD-HWE-Bench 最具区分度的设计是规范驱动性——Agent 必须**主动查阅**设计规范文档才能完成设计。这在 easy 任务上不构成障碍（字段少、参照简单），但在 hard 任务上（如 rack-stage4 的跨机柜光纤，需要同时遵循 U 位规则、PDU 字段规范、端口/光模块/光纤/连接/配合 5 类规范；或 AIDC 任务需要查阅 cooling-spec 中的 COP 曲线）成为核心瓶颈。

### 8.4.2 跨文件协调

hard 任务平均涉及 10+ expected_files，Agent 需要在多个 YAML 文件之间保持引用一致性（如端口实例 → 光模块实例 → 光纤实例 → 配合实例的引用链）。CLI Actor 可以通过文件系统逐个验证，避免了单次生成中的信息衰减。

### 8.4.3 长程优化与耦合约束

AIDC/EPC 任务引入时间维度上的耦合约束：冷却器台数、储能 SOC、光伏出力、分时电价、变压器效率曲线共同决定全年 PUE 与 TCO；施工网络中的活动 precedence、资源可用量、天气窗口与应急预案共同决定项目工期与成本。Agent 容易在单次局部修改后满足部分指标，但难以通过全局搜索达到 reference 最优解。

### 8.4.4 Easy 任务占比

当前 37 个任务中 easy 占 19%（7/37）。这些任务主要测试 Agent 是否理解规范文档中的字段名和目录结构，不涉及复杂的跨文件协调。数据集已显著提高 medium/hard 比例，但仍有进一步提升空间。

改进方向（不在本文范围内）：增加 failure injection 任务（在 scaffold 中埋入错误让 Agent 检测和修复）、引入更激进的 cross-domain 联合约束、扩展 AIDC 任务规模。

### 8.5.1 规则密度分析

每个任务底层平均执行 25–35 条确定性规则（5 个 L1 schema 规则 + 6 个 L2 引用规则 + 8 个 L3 工程约束 + 1–3 个 L4 动态规则 + 4 个 L5 几何规则）。对比 SWE-bench 每个 task 通常只有 1-3 个 pytest 测试用例——SD-HWE-Bench 的约束密度更高，但当前规则集中在预防性检查（catch errors），缺少压力测试级别的高难度约束。

未来可通过引入更激进的约束（如 multi-rack 全局功率优化、cross-domain 消防通道与机柜布局的 joint constraint）提升规则的有效区分度。
