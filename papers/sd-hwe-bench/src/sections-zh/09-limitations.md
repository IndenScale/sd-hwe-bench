# 9. 局限性与未来工作

本章诚实讨论 SD-HWE-Bench 当前的局限性，并勾勒未来的改进方向与生态愿景。

## 9.1 当前局限

### 9.1.1 领域覆盖有限

当前 canonical 工程仅覆盖电信机柜、数据中心和机械电子集成三个领域。模拟电路、射频系统、光学设备、建筑结构、HVAC、能源系统等大量硬件工程子领域尚未覆盖。每个新领域需要构建新的 canonical 工程、定义该领域的 Part 类型库和 DTS 规则——这是劳动密集型工作，但一旦完成即可持续扩展。

### 9.1.2 任务规模有限

在 canonical 工程完成后，预期任务规模为 30-60 个。这对于基准评测是足够的，但对于训练大规模模型（如 RLVR 需要大量可验证训练样本）可能仍显不足。SWE-bench 拥有 2294 个任务，这得益于 GitHub 上数以百万计的开源仓库。ADL 工程的开源生态尚未建立，因此 SD-HWE-Bench 短期内无法达到同等任务规模。

### 9.1.3 Canonical 工程 vs. 真实企业流程

Canonical 工程由作者构建，尽管模拟真实设计迭代，但与真实企业中的大型工程设计流程（数十名工程师、数百个约束文档、多轮评审）在复杂度和多样性上存在差距。Canonical 工程是"真实场景的抽象简化版"，其代表性需持续评估。

### 9.1.4 DTS 的物理真实性边界

DTS 的 L4 层目前使用解析公式和轻量代理模型（如包围盒碰撞检测），尚未完全替代高精度 CFD/FEA/SPICE 仿真。对于依赖精确物理行为断言的工程设计（如 PCB 信号完整性、结构应力分析），L4 的通过不代表设计在实际制造中一定可行。DTS 更多是"快速筛除明显违规"，而非"保证设计完全正确"。

### 9.1.5 基线方法简单

当前实验仅使用通用 LLM Agent 的零样本/少样本 prompt，未针对 ADL/DTS 进行专门微调、强化学习、或 Agent 工作流优化。这意味着 SD-HWE-Bench 报告的基线是"下限"而非"上限"——未来可通过 ADL-specific 预训练、RLVR fine-tuning、Agent 工具增强（如让 Agent 调用 `piki check` 做 self-check）等方法显著提升。

### 9.1.6 仅支持文本原生交互

SD-HWE-Bench 当前仅支持纯文本的 ADL 输入/输出。真实硬件工程中大量交互依赖可视化——3D CAD 视图、PCB 布局图、仿真热力图。未来版本可考虑引入多模态通道（如 Agent 可查看 piki generate 生成的 3D 预览截图作为上下文）。

### 9.1.7 基准时效性与防污染

随着社区采用 SD-HWE-Bench，存在 benchmark 污染风险——模型训练数据中可能包含任务内容或 gold patch。为此，SD-HWE-Bench 的设计通过两种机制缓解：(1) 任务从 canonical 工程提取，而非公开的 GitHub issue；(2) canonical 工程可持续扩展和更新，生成新的未见过任务。此外，我们计划定期发布新的 canonical 工程和任务作为"held-out"测试集。

## 9.2 未来方向

### 9.2.1 扩展领域与任务规模

**短期目标**：在电信/数据中心/机械三大领域内扩充 canonical 工程和任务，达到 50+ 任务。
**中期目标**：新增 2–3 个领域（如模拟电路、PCB 布局、建筑 MEP），每个领域有独立的 canonical 工程和领域特定 DTS 规则。
**长期目标**：建立社区贡献机制，允许领域专家提交 canonical 工程和 DTS 规则，通过审核后纳入官方基准。

### 9.2.2 自动/半自动任务生成

借鉴 SWE-smith [@yang2025swesmith] 的思路，探索从真实设计文档、开源硬件项目或其他结构化来源自动提取 ADL 任务。结合 LLM 辅助的 Part 类型推断和规则自动生成，缓解人工构建 canonical 工程的扩展瓶颈。

### 9.2.3 ADL-Specific 模型与 RLVR 训练

SD-HWE-Bench 的 DTS 分层断言提供了确定性、低成本、高频的奖励信号——这正是 RLVR 训练的理想环境。未来可探索：

- 在 ADL 代码数据上预训练专用模型，提升基础 YAML/ADL 语法理解。
- 使用 SD-HWE-Bench 的 DTS 反馈作为 RLVR 的奖励信号，训练能自我修复的工程 Agent。
- 结合 repair loop 的迭代过程作为训练数据，学习从 DTS 错误报告到正确 patch 的映射。

### 9.2.4 多模态反馈通道

当前 SD-HWE-Bench 仅提供文本形式的 DTS 诊断（规则 ID + 错误消息）。真实工程设计中，工程师大量依赖视觉反馈判断设计质量——3D 预览、热力图、布线拓扑图等。一个自然的扩展是为 Agent 增加视觉反馈通道：piki 接收项目文件 + 相机位姿参数，返回 OpenSCAD 渲染的 3D 场景截图或热力图，Agent 可以"看到"自己的设计产物并据此迭代。引入视觉反馈后，可对比 text-only vs text+vision 两种模式在空间/几何密集型任务（如 layout-design、mating-design）上的通过率差异，验证多模态对工程 Agent 的因果价值。

### 9.2.5 多模态交互

集成 piki 生成的可视化产物（3D 预览截图、BOM 表、布线图、热力图）作为 Agent 的附加上下文。Agent 既能"看"到自己的设计产物，也能获得视觉化的 DTS 错误标注（如碰撞区域高亮）。这将缩小纯文本交互与真实工程环境之间的差距。

### 9.2.6 与高精度仿真对接

在 DTS 框架中引入可选的高精度仿真后端接口：PCB 信号完整性用 SPICE、结构应力用 FEA、气流/散热用 CFD。高精度仿真在关键任务中作为最终验证，而日常高频迭代仍依赖 DTS 轻量检查——形成"DTS 快速筛除 + 高精度仿真最终确认"的两阶段验证体系。

## 9.3 社区与生态愿景

SD-HWE-Bench 的长期目标不仅是成为一个 benchmark，更是成为**声明式硬件工程 AI 的研究平台与社区生态**：

- **开源**：Canonical 工程、任务提取工具、DTS 运行时（piki）、评测 harness 全部开源到 GitHub/Zenodo，支持学术研究和工业实验。
- **社区贡献**：建立领域特定的 Part 类型库和 DTS 规则库的社区维护机制——任何领域专家都可贡献新的 ADL 定义和检查规则。
- **Leaderboard**：维护公开 leaderboard，按领域和任务类型分层展示各 Agent 的表现，驱动良性竞争和方法演进。
- **Work-in-progress 友好**：承认 SD-HWE-Bench 当前任务规模和数据多样性有限——我们视基准的持续迭代和社区共建为其核心价值，而非一次性发布。

我们相信，通过 Engineering as Code 建立物理工程的可计算表示层，进而为 Agent 提供可高频迭代的确定性反馈环境，是推动 AI 进入物理工程领域的必要基础设施。SD-HWE-Bench 是这一方向的第一步，其价值不仅在于当前的评测结果，更在于它所验证的"可计算表示 + 执行评测"范式在物理工程领域的可行性。
