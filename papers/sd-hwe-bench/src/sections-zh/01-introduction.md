# 1. 引言

## 1.1 AI for Engineering 的重要性

现代社会的基础设施——从数据中心、通信基站、新能源系统到航空航天器——本质上都是由物理工程设计决定的。设计阶段的决策对最终成本、性能、可靠性与碳排放具有决定性影响。因此，提升物理工程设计的自动化水平不仅是学术前沿问题，也是具有巨大社会经济价值的应用目标。

近年来，大语言模型（LLM）在代码生成、数学推理、科学发现等领域展现出强大能力 [@chen2021evaluating; @hendrycks2021measuring]。将这些能力迁移到物理工程领域，有望加速设计迭代、降低试错成本，并最终实现更自主的工程智能体。然而，与软件代码不同，物理工程设计涉及多物理域耦合、几何约束、材料特性、制造工艺等复杂因素，其可计算表示远不如源代码成熟。

## 1.2 Engineering for AI：确定性奖励稠密的下一个战场

从强化学习与可验证生成的角度看，工程领域具有一个极其宝贵的特性：**确定性奖励稠密（dense verifiable reward）**。与开放域对话或创意写作不同，工程中的几乎每个决策都可以被快速验证——电路连接是否符合电气规则、元件布局是否满足几何约束、散热设计是否满足热预算、结构件是否存在干涉、电源分配网络是否满足压降要求。这些验证可在毫秒到秒级完成（规则检查）或分钟级完成（仿真）。因此，工程是继围棋、数学定理证明、代码生成之后，又一个天然适合 RLVR（Reinforcement Learning with Verifiable Rewards）的领域 [@silver2017mastering; @lightman2023letsverify; @deepseek2025r1]。

然而，要释放这一潜力，前提是存在**可计算的设计表示**和**可高频迭代的评估基础设施**。代码与配置文件是软件工程的可计算表示，风格检查、编译器、测试套件是廉价反馈；物理工程也需要类似的"文本化设计表示 + 自动化验证"闭环。

## 1.3 软件工程的成功与硬件工程的滞后

软件工程已经实现了这一闭环。代码本身就是一种可计算、可版本控制、可自动执行的表示；结合 GitHub issue、pull request 与测试套件，研究者构建了 SWE-bench [@jimenez2024swebench]、SWE-bench Multimodal [@yang2025swebm] 等大规模 benchmark。在这些工作的推动下，代码智能体在真实 GitHub issue 上的解决率从不到 5% 快速提升至 40% 以上 [@jimenez2024swebench]。

相比之下，硬件工程（Hardware Engineering, HWE）的自动化进程明显滞后。除集成电路/数字电子自动化（IC/DEA）因其成熟的 HDL 表示和开放工具链（如 Yosys [@wolf2016yosys]、OpenROAD [@ajayi2019openroad]）取得了 AI 落地的进展外，系统级/设备级硬件工程——如电信机柜部署、数据中心基础设施、机械装配——仍严重缺乏可计算表示层和评估基础设施。这类工程中，设计数据分散在 CAD/BIM 模型、PDF 规格书、Excel 参数表和商业软件 GUI 中，缺乏统一的机器可消费格式。

**这导致一个深层后果**：硬件工程在 AI 社区的叙事、注意力和资源分配中系统性不足，进而形成"表示缺失 → 评估缺位 → 训练不足 → 能力弱"的负反馈循环。我们称之为**可计算表示缺口（representation gap）**：硬件工程 AI 的瓶颈不是模型能力不足，而是缺乏"Design as Code"形式的可计算基础。

## 1.4 本文工作：SD-HWE-Bench

为填补这一缺口，我们采用 **Engineering as Code（EaC）** 范式 [@song2025eac]，提出 **SD-HWE-Bench**——首个面向声明式硬件工程的大规模可执行 benchmark。

具体而言，SD-HWE-Bench 包含以下核心设计：

1. **ADL 文本原生表示**：采用 Assembly Definition Language（ADL），以 Part（工程原子）、PDL（部件定义）、PML（配合关系）、PLL（空间布局）三层正交子语言，将电源、热、结构、信号完整性等多物理域设计统一为可计算文本表示。
2. **DTS 分层确定性反馈**：配套引擎，从 L0 语法检查到 L5 几何干涉、L6 预留高精度物理仿真，在毫秒到秒级提供分层反馈，替代昂贵的仿真与人工审查。
3. **Canonical 工程 + Commit 提取**：参考 SWE-bench 的任务生成范式，从精心构建的 canonical ADL 工程的 commit 历史中提取真实设计增量——每个任务是一个 issue→patch→tests 三元组。
4. **RLVR 兼容评估协议**：多模型基线 + DTS repair loop + no-repair vs repair ablation，系统评估确定性反馈对 Agent 生成质量的因果价值。

## 1.5 主要贡献

1. 提出 **SD-HWE-Bench**，首个面向声明式硬件工程的可执行 benchmark，覆盖电信机柜、数据中心、户外基站与 AIDC 四个子领域，包含 37 个真实设计增量任务。
2. 设计 **ADL + DTS** 的可计算工程表示与评测框架，支持从语法到物理行为的多层确定性反馈。
3. 构建多个 canonical ADL 工程，并实现从 commit 历史自动提取任务的 pipeline（`tools/extract_tasks.py`）。
4. 在多个主流 LLM Agent（Kimi Code/DeepSeek）上建立基线，系统分析其能力与失败模式。
5. 通过 no-repair vs repair ablation，证明 DTS 确定性反馈对硬件工程生成质量的因果提升。
6. 开源 canonical 工程、任务提取工具、评测 harness 与复现环境，支持后续研究与 RLVR 训练。

## 1.6 结果预览

实验表明，SD-HWE-Bench 对当前主流 LLM Agent 具有挑战性。在完整上下文、无 repair 设置下，30 个已评测任务的当前最佳 pass@1 为 86.7%（DeepSeek-v4-Pro），最佳平均加权得分为 82.8%（Kimi k2.7）。错误分析显示，模型失败主要集中在 L2 引用完整性、L3/ASA 静态分析（功率预算、接口兼容性）以及 AIDC 任务的 L4/ADA 仿真硬约束，而非高级推理本身——提示当前 Agent 的瓶颈不在于"想不出方案"，而在于"无法在长上下文、多约束空间中稳定满足工程约束"。
