# 可执行约束：核心主张

本 memo 记录“可执行约束鸿沟”章节的核心论证。它不是正文定稿，而是后续改写 `src/sections-zh/05-constraint-experiment.md` 与实验基础设施的依据。

## 第一性问题

工程 Agent 的很多失败不是因为模型完全不知道工程规则，而是因为规则停留在自然语言、规范文档、经验审查或隐式流程中，没有变成可执行、可定位、可归因的反馈系统。

自然语言规范可以表达“应当满足什么”，但不能稳定回答三个问题：

- 本次任务到底涉及哪些约束？
- 提交物是否真实满足这些约束？
- 若不满足，错误发生在哪个对象、字段、关系或约束层？

因此，可执行约束鸿沟不是“有没有规范”的问题，而是规范是否进入了可计算闭环的问题。

## 章节要证明的核心结果

本章应围绕四个主结果组织证据：

1. **语言化约束会系统性漏实现。**

   当约束只以任务描述、自然语言 checklist 或大型规范文档存在时，模型或 Agent 容易生成看似合理、术语专业、结构完整但不满足要求的 pseudo-correct 方案。即使约束写在文档里，Agent 也可能没有提取本次任务涉及的部分，或没有把它落实到提交文件中。

2. **可执行约束把正确性从“模型是否记得”转成“系统是否能判定”。**

   可执行规范的价值不是让模型本身更聪明，而是把工程规则外部化为确定性 critic。Agent 可以失败，但失败不再停留在自然语言猜测；系统能判断哪些约束未满足，并形成 repair 目标。

3. **可执行程度与最终设计质量大致正相关。**

   可执行规范不是二元变量。可以通过控制哪些约束进入提示词、哪些约束进入 repair 反馈，构造从完全自然语言到部分可执行反馈再到完整可执行反馈的连续谱。最终评分仍应使用完整约束集合；被 mute 的约束只是对 Agent 不可见或不反馈，用于观察它会在哪些约束上丢分。预期结果不是每个任务都单调显著，而是总体上可执行反馈覆盖率越高，遗漏约束越少，最终产出越稳定。

4. **诊断的定位和归因信息决定 repair 效率。**

   同样是发现错误，反馈“设计不满足电气约束”和反馈“`instances/pdus/PDU-C.yaml` 的 `phase` 字段缺失或不满足三相均衡规则”对 Agent 的修复效果完全不同。可执行约束的收益不仅来自发现错误，还来自错误能否被定位、归因和消费。

## 与任务规模和复杂性的关系

本章不能把结论简化成“任务越大，可执行约束越有用”。需要区分两个维度：

- **规模增长**：对象数量、文件数量、接口数量增加，但耦合关系不一定加深。
- **复杂性增长**：选型、布置、调度、成本、SLA、施工、热、电等约束互相耦合，局部正确不保证全局正确。

可执行约束的收益不是简单随任务规模线性增长，而是随约束耦合复杂度上升而显著放大。

对于已饱和的低耦合任务，例如简单设备布置、端口连接或机柜扩容，模型可能靠 scaffold 模式、世界知识和任务文本已经能达到很高 pass@1。此时可执行约束主要表现为稳定性保障、防回归、偶发漏项捕获和可审计评分。

对于高耦合系统级任务，例如 AIDC 的设备选型、空间布置、电力容量、冷却能力、运营调度、成本和 SLA 联合优化，模型不能只靠“看起来合理”的模式完成。此时可执行约束才会显著表现为通过率、漏约束率、repair 轮次和最终质量上的结构性差异。

## 章节主张的一句话版本

工程 Agent 的失败不是单纯模型能力不足，而是约束没有被计算化、覆盖化、定位化。自然语言约束导致遗漏；可执行约束提供确定性收敛目标；可执行覆盖率决定最终设计质量；诊断定位和归因决定修复效率。
