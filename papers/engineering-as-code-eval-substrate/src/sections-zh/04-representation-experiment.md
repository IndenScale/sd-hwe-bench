# 4. 实验一：可闭环表征

本实验验证不同工程表征路径能否支持 engineering agents 的严肃自动化评估闭环。实验目标不是证明 ADL 在所有工程场景中优于 CAD、BIM 或参数化设计，而是拆解：在同等工程意图下，哪些表征能力决定任务能否被提交、检查、定位和修复。

## 4.1 第一性问题

如果本文只说“工程缺少可计算表征”，很容易被参数化设计反驳。OpenSCAD、Revit family、Grasshopper、Dynamo、Excel 配置表和企业内部模板都已经能够用变量生成工程工件。因此本实验真正要问的是：

**参数化设计是否已经足以成为 engineering agents 的 evaluation substrate？**

我们的假设是否定的。参数化设计解决了生成问题，但通常没有同时解决生成-验证、本专业-跨专业、设计-建造、当前项目-未来项目、个人知识-组织知识五重闭环。对于低耦合几何或单专业任务，参数化路径可能表现很好；但在 AIDC 这类跨设备、跨布局、跨调度、跨施工的系统级任务上，缺失闭环能力会表现为 pseudo-correctness、repair 困难和不可复现评分。

## 4.2 比较路径

我们将同一工程需求交给五类表征路径完成，比较其闭环能力：

| 路径       | 典型形式                                 | 核心表征                           | Agent 操作       | 预期特点                                  |
| ---------- | ---------------------------------------- | ---------------------------------- | ---------------- | ----------------------------------------- |
| NL-only    | 自然语言方案                             | 文本描述                           | 直接生成说明     | 无 diff、无验证、无局部 repair            |
| CUA/GUI    | CAD/BIM/表格 GUI                         | 视觉与交互状态                     | 点击、输入、导出 | 能接近真实工具，但交互慢、脆弱、难复现    |
| MCP/Tool   | 外部 API、数据库、插件                   | 工具调用参数                       | 查询、调用、汇总 | 增强访问能力，但不定义统一提交对象        |
| Parametric | OpenSCAD、CAD family、Excel、Grasshopper | 参数、族、配置表、脚本             | 修改变量或脚本   | 可计算、可 diff、可复用，但常以生成为中心 |
| ADL/EaC    | 声明式工程对象与关系                     | Part、属性、引用、连接、配合、调度 | 修改工程声明文件 | 支持生成、验证、诊断和 repair 闭环        |

为了避免把“文本化程度”和“闭环能力”混在一起，实验需要加入关键对照：同一参数化模型在“仅生成”和“生成 + DTS 验证”两种条件下比较；同一 ADL 任务在只反馈本专业错误、反馈跨专业冲突、反馈设计-建造冲突三种条件下比较。

## 4.3 闭环能力轴

实验把闭环能力作为独立变量，而不是只比较工具名称：

| 条件               | 保留的能力                        | 目的                          |
| ------------------ | --------------------------------- | ----------------------------- |
| Gen-only           | 只保留生成能力                    | 测试“可计算但不闭环”的上限    |
| Gen + Verify       | 生成后用独立检查器评分            | 测试验证闭环的价值            |
| Gen + Cross-domain | 电气、热、结构、布局联合检查      | 测试跨专业闭环的价值          |
| Gen + Buildable    | 检查吊装、施工通道、排程和资源    | 测试设计-建造闭环的价值       |
| Gen + Transferable | 同一表征迁移到新项目或新设备库    | 测试当前-未来闭环的价值       |
| Gen + Documented   | 新人或 agent 无需作者解释即可修改 | 测试个人-组织闭环的价值       |
| Full closed-loop   | 五重闭环均保留                    | 测试完整 EaC substrate 的上限 |

最终评分始终使用完整约束集合。不同条件只控制表征路径和 agent 可见的闭环能力，不降低工程正确性标准。

## 4.4 任务复杂度分层

表征实验不能只给一个总体平均值。任务需要按“规模”和“耦合复杂度”分层：

| 任务类型               |  规模 | 耦合复杂度 | 预期现象                                              |
| ---------------------- | ----: | ---------: | ----------------------------------------------------- |
| 简单几何/布置          | 低-中 |         低 | 参数化和 ADL 都可能通过，差异主要在可审计性和可交接性 |
| 单专业参数化优化       |    中 |      低-中 | 参数化路径表现好，ADL 优势不必显著                    |
| 跨文件/跨专业约束      |    中 |      中-高 | 参数化路径开始漏跨专业引用和冲突                      |
| 选型-布置-调度联合优化 | 中-高 |         高 | 完整闭环决定稳定收敛能力                              |
| 设计-施工联合排程      | 中-高 |         高 | 设计-建造闭环成为关键变量                             |

低耦合任务不是反例。它们用于证明 EaC 在简单场景中提供稳定性、防回归和可审计价值；真正放大差距的是高耦合系统级任务。

## 4.5 指标与图表

本实验不以 pass rate 为唯一指标，而关注 substrate 能力：

- **任务形式化成本**：从工程需求到可评测任务需要多少人工建模步骤。
- **提交确定性**：同一提交是否能稳定复现评分。
- **反馈延迟**：从提交到结构化错误反馈的时间。
- **错误可定位性**：错误能否定位到对象、字段、引用或规则。
- **repair 可行性**：agent 是否能基于反馈进行局部修复。
- **评分覆盖度**：是否覆盖语法、引用、工程约束、几何、仿真、交付物和可建造性。
- **跨专业冲突检出率**：本专业检查通过但跨专业联合检查失败的比例。
- **可迁移成本**：同一表征应用到新地点、新设备库或新规范所需调整量。
- **交接成本**：新成员或 agent 在没有原作者解释时复现和修改表征所需轮次。

预期图表包括：表征路径闭环能力矩阵、闭环覆盖率-质量曲线、任务复杂度分层结果、跨专业冲突案例、可迁移/交接成本对比。

## 4.6 预期结论

本实验要支撑的结论是：工程 agent 需要的不是“可计算的工程工件”，而是“可闭环的工程表征”。参数化设计解决了生成问题；EaC substrate 的贡献在于把工程状态变成 agent 可提交、critic 可检查、错误可定位、知识可迁移、repair 可迭代的评估对象。
